Chapter 9

Workflow Design Patterns

This chapter covers how to combine orchestration patterns to design complete workflows used in real-world domains, along with a variety of case studies.

What This Chapter Covers
  • The relationship between orchestration patterns and workflows — patterns are the blocks, workflows are the finished assembly
  • The six components that make up a workflow: stage, branch, fan-out, fan-in, gate, loop
  • Five real-world domain workflow cases with flow diagrams
  • The subagent roster — each agent's tier, parallelizability, and write permissions
  • The role boundary matrix — which agent to use for which scenario
  • The PM phase pipeline — the execution workflow from Phase 0 through Phase 6
  • A guide for deciding which patterns to select and combine based on the nature of the work

What Is a Workflow

The five orchestration patterns learned in Chapter 2 §4 — prompt chaining (pipeline), routing, parallelization (fan-out/voting), orchestrator-worker, and evaluator-optimizer (verify-then-apply) — are fundamental building blocks for structuring cooperation between agents within a single unit of work. In practice, however, it is far more common to connect and layer multiple blocks into one complete flow than to complete a task with just a single block.

A workflow refers to "the entire process by which one or more agents cooperate to complete a complex task." If orchestration patterns answer how agents cooperate, workflows answer in what order and structure the whole process proceeds. Workflows typically combine the following three elements.

  • A combination of orchestration patterns — some stages are sequential (pipeline), some are parallel (fan-out), and some branches are conditional (routing).
  • Domain rules — rules required by the specific business domain, such as "code review must separately examine security and performance perspectives."
  • Human approval gates — following the guardrail principles learned in Chapter 3, a step where a person must confirm before an irreversible point.
Orchestration patterns are Lego blocks. A single one works on its own, but assembling several together produces far more complex and useful shapes. A workflow is the result of that assembly.

This chapter assumes you already know the patterns from Chapter 2. If any term is unfamiliar, first read Chapter 2 §4 Orchestration Patterns and §4b Dynamic Team Composition. The next section goes beyond patterns and lays out the six components that make up a workflow.

Workflow Components

Breaking down a complex workflow reveals a few recurring structural elements. Chapter 2's five patterns focused on how agents cooperate with each other, but here we focus on the structural skeleton of the workflow.

StageA unit of work executed sequentially. The most basic building block of a workflow. The output of the previous stage becomes the input of the next.
BranchA point where the path splits based on a condition. The practical application of the routing pattern — for example, sending a request to a different agent or next stage depending on its nature, such as "a quick review for small changes, a full review for large ones."
Fan-outA point where independent subtasks run simultaneously. The structural expression of the fan-out pattern. Execution time converges to the slowest subtask. Example: reviewing security, performance, and readability perspectives at the same time.
Fan-inA stage that gathers multiple results, executed in parallel, into one. A concept not covered in Chapter 2. Fan-in requires rules for deciding which result to prioritize (voting) and how to merge them (aggregate/select). Example: synthesizing the opinions of three reviewers into final feedback.
GateA point that requires verification or human approval before moving to the next stage. The practical application point of Chapter 3's guardrails. Placed before hard-to-reverse actions (file modification, deployment, PR merge).
LoopA repeating structure that returns to a previous stage when rejected at a verification stage. The structural expression of the evaluator-optimizer pattern (verify-then-apply). A maximum retry count must be set to prevent infinite loops.
These six components do not map 1:1 to Chapter 2's five patterns. A single pattern can include multiple components (e.g., the fan-out pattern = fan-out + fan-in), and a single component can appear in multiple patterns (e.g., a gate appears in both verify-then-apply and pipeline).

Combining these elements can express nearly every real-world workflow. The next section walks through case studies of how these elements are arranged in actual domains.

Real-World Workflow Cases

Five domain workflows are organized by case. Each case shows the core pattern combination, stage flow, responsible agents, and the location of the human gate. Cases already connected to variants implemented in ai-workspace-standards are also included.

Code Review Software Development

Core patterns: Routing → Fan-out → Fan-in → Pipeline → Verify-Then-Apply(loop)

① Change Analysis (Routing): Analyzes the scale and type of the changed files to determine the review scope. Simple typo fixes branch to a fast path; major feature changes branch to a full review path.
② Multi-Perspective Review (Fan-out): Specialized reviewer agents simultaneously examine security, performance, and readability perspectives.
③ Opinion Aggregation (Fan-in): A synthesis agent compiles the three reviewers' feedback into a single feedback report.
④ Fix Proposal (Pipeline): Generates code fix proposals based on the feedback.
⑤ Final Approval (Verify-Then-Apply): Re-reviews the modified code. If it passes, goes to the human approval gate; if rejected, loops back to ④.

variant: structurally similar to co-develop's 6-stage governance pipeline. See Chapter 6 §1.

Market Research Report Consulting · Research

Core patterns: Orchestrator-Worker → Fan-out → Pipeline → Verify-Then-Apply

① Orchestration (PM): The PM agent defines the research scope, target market, and deliverable format, then delegates to workers.
② Parallel Research (Fan-out): Three agents simultaneously perform market-size analysis, competitor analysis, and technology-trend research.
③ Aggregation and Drafting (Pipeline): Synthesizes the three research results into a draft report and generates visualizations (charts).
④ Cross-Verification (Verify-Then-Apply): A specialist reviewer checks data sources, logical consistency, and omissions. After the human approval gate, the final report is confirmed.

variant: based on co-consult's 7-stage (Phase 0-6) pipeline structure. You can experience it directly in Chapter 6 §2 exercise (G-1).

Presentation Production Lecture · Presentation

Core patterns: Pipeline(long pipeline) + Fan-out(intermediate parallelism)

① Research: Collects material relevant to the topic and verifies sources.
② Storyline: Plans the overall content flow and slide structure.
③ Design: Selects the visual layout and theme.
④ Image Curation + Diagram Generation (Fan-out): Generates illustrations and diagrams simultaneously to save time.
⑤⑥⑦ Build · Measure · Export (Pipeline): Sequential processing from HTML build → layout measurement → PDF export.
⑧⑨⑩⑩ Final Review (Verify-Then-Apply): Cross-reviews the completed PDF, then the human approval gate.

variant: this is exactly co-deck's 11-stage (Stage 0-11) pipeline. You can experience it directly with a Korean-food-globalization presentation in Chapter 6 §2 exercise (G-2).

Issue Triage Operations · Incident

Core patterns: Routing → Orchestrator-Worker → Escalation

① Classification (Routing): Analyzes the urgency and type of an incoming issue to determine the path. Bugs branch to a fix path, feature requests to a planning path, and security issues to an urgent path.
② Automated Investigation (Orchestrator-Worker): Worker agents investigate the classified issue by breaking it into subtasks — relevant log search, reproduction attempts, impact-scope assessment.
③ Priority Determination: Recommends priority and an assignee based on the investigation results.
④ Escalation or Auto-Resolution: Minor issues that can be resolved automatically (e.g., configuration errors) are fixed immediately; complex issues are escalated to the human approval gate.

There is no variant dedicated to incident management yet, but it can be implemented by combining co-develop's PM role with Chapter 2's failure-handling patterns (retry, rollback, escalation).

PR Creation and Verification Governance

Core patterns: Pipeline → Verify-Then-Apply(loop) → Gate

① Change Collection: Collects commit history and diffs to draft the PR (title, body).
② Rule Verification (Verify-Then-Apply): Automatically checks whether the PR complies with CONSTITUTION.md rules (English title, specific format, etc.). If any violation exists, loops back to ① for correction.
③ Impact Analysis: Analyzes how the change affects other files/features and adds the findings to the PR body.
④ Final Review + Approval: Re-reviews the entire PR, then the human approval gate. Once approved, transitions to a merge-pending state.

A structure where agents automate the PR workflow defined in ai-workspace-standards's CONSTITUTION.md. Covered together with the deployment pipeline in Chapter 5 §2 Core Files and Chapter 8 §1 Deployment and SSOT.

Common Patterns Across the Cases

A few common structures emerge from the five cases.

  • Every workflow has a human gate. No matter how far automation progresses, human approval is required before irreversible decisions such as finalizing deliverables, merging, or deploying. This is the practical application point of the guardrails learned in Chapter 3.
  • A fan-out is always followed by a fan-in. If results split into parallel branches are simply left side by side, the consumer (the agent in the next stage, or a person) must synthesize them manually. An explicit fan-in stage is needed to produce a single deliverable that can move cleanly into the next pipeline stage.
  • Whether a verification stage loops or escalates determines a workflow's complexity. Problems that can be auto-fixed (format violations, typos) are resolved via a loop, while problems that require judgment (design flaws, security vulnerabilities) are handed off to a person via escalation.
  • Simple pipelines are rarely used. None of the five cases are "purely sequential" workflows. At least one point includes routing (branching) or fan-out (parallelism).

The "Parallel Execution" column of this table must be interpreted together with the Tier Adjustment Rules covered in Chapter 8 §4. In contrast, Auditor is "independent QA" and can run independently at any time, regardless of other agents' work.

The core principle of this matrix is "separating design from execution." architect only designs and does not write scripts directly, while automation-engineer writes scripts but does not design the structure. docs-writer does not verify documentation — auditor verifies it, and docs-writer revises the documentation to reflect auditor's feedback.

PM Phase Pipeline — the Governance View of a Workflow

The workflow patterns covered in this chapter (routing, fan-out, pipeline, etc.) represent the technical component viewpoint. In an actual workspace, governance phases are layered on top of these technical patterns. The seven phases managed by the PM, organized from a workflow perspective, are as follows.

Phase 0Project Initialization (PM-exclusive) — assess workspace requirements, dynamically generate agents/skills, update AGENTS.md. Workflow: orchestrator alone
Phase 1-2Planning · Architecture (specialist autonomy) — PM classifies the request, Architect writes an implementation plan + ADR, and research agents are dispatched in parallel. Workflow: routing → fan-out (parallel research)
Phase 3Design Handoff (per variant) — Architect hands off the approved plan to execution agents. Autonomous handoff between agents. Workflow: orchestrator-worker (handoff)
Phase 4Execution (specialist autonomy) — Automation Engineer implements per the approved plan, Docs Writer updates documentation. Workflow: pipeline (sequential execution)
Phase 5Lifecycle Closure (PM-exclusive) — PM updates the governance record of the changed artifacts. Workflow: orchestrator alone (recording)
Phase 6Quality Assurance (workspace autonomy) — Auditor automatically runs qa-gate.ts. Escalates to the PM after a maximum of 2 iterations. Workflow: verify-then-apply (loop)

These seven phases correspond directly to the workflow cases in §3. For example, the overall flow of the "code review" case is a miniature of Phase 1-2 (analysis) → Phase 4 (review execution) → Phase 6 (QA). Keeping these phases in mind when designing a new workflow in practice makes it easier to systematically decide "where to place the human gate" and "which stages are safe for parallel execution."

Pattern Combination Guide

Here is a summary of which patterns produce synergy when used together.

Fan-out + PipelineParallel review, multilingual translation followed by post-processing. Fan-out provides speed, and pipeline guarantees order.
Routing + Fan-outIssue triage, size-based review. Routing classifies, and each path then performs a different parallel task.
Orchestrator-Worker + VerifyComplex analysis, design review. The PM distributes dynamically, then the result is verified.
Pipeline + Verify (loop)Code generation, PR creation. Write sequentially, but revert to a previous stage when the quality bar isn't met.
Routing + OrchestratorModel tiering. Easy tasks are distributed to a lighter model, difficult tasks to a stronger one.
