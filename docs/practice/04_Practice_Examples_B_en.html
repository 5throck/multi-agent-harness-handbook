Chapter 4 §2-B · Multi-Agent Team Practice

Date writtenJuly 14, 2026
The principle behind this entire course — if you learn practice around a single tool, you'll have to relearn everything from scratch when you move to a different tool. That's why every scenario below first writes the subagent's role as a tool-neutral AGENTS.md specification, then "implements" that same specification as a natural-language goal in the Antigravity family (see Chapter 2 §5 AGENTS.md). The corresponding approach in the Claude family is covered in Chapter 4 §2-A. Confirming hands-on that there is one definition but the execution method differs per tool is the core goal of this practice.

Setup: Installing Required Programs

The practice below focuses on Antigravity (Desktop) and the Antigravity CLI (agy). If your local environment doesn't have these tools yet, the environment setup automation tool bundled with this handbook — the setup guide — will automatically install exactly what this chapter's practice needs. Follow the instructions to download and run the script locally.

📦 Things to Check Before Installing

Before running the script, first read the pre-installation checklist. This document lists the following prerequisites.

  • A Google account for using Antigravity (must be a subscription/account-based plan, not an individual API key)
  • A GitHub account that can complete gh auth login
  • At least 5GB of free disk space
  • Administrator (admin) or sudo privileges
If you don't have a GitHub account yet
① Go to github.com/signup in your browser.
② Enter your email address, password, and username — the username will be used as-is in the repository URL (github.com/<username>/...), so choose it carefully.
③ Complete the puzzle/code verification to prevent automated signups.
④ Open the verification email sent to the address you signed up with and complete email verification — skipping this step can later block gh auth login or repository creation.
⑤ Signing up for the Free plan is sufficient — this handbook's practice/PR exercises don't require a paid plan.
Once signup is done, the GitHub CLI (gh) is included and installed together by the setup script above, and you connect your terminal to that account with the gh auth login command below. 
gh auth login
# Follow the prompts: choose GitHub.com → HTTPS → Log in with a browser (recommended)
# When the browser opens, log in with the account you just created and confirm the verification code

Installing on Windows

Since this handbook's path notation is Windows-based, Windows users should download setup-windows.ps1, place it in a dedicated practice folder (e.g., harness-lab/setup/), open PowerShell as an administrator, and run the following command.

How to open PowerShell as an administrator
① Open the Start menu and search for PowerShell.
② Right-click "Windows PowerShell" in the search results.
③ Select "Run as administrator" from the menu that appears.
④ When the User Account Control (UAC) prompt appears, click "Yes" to approve.
If the title bar shows something like Administrator: Windows PowerShell with "Administrator" attached, it's opened correctly. Some steps, such as winget installation, may error out or silently skip without administrator privileges. 
cd ./setup
.\setup-windows.ps1
# Optional flags: -WSL2, -WezTerm, -Docker, -Force (reinstall already-installed tools too)

After the script finishes, fully close and reopen your terminal for the PATH changes to take effect.

Installing on macOS / Linux

On macOS, download and use setup-mac.sh; on Linux, use setup-linux.sh. Both support the optional --wezterm and --docker flags.

# macOS
cd ./setup && bash setup-mac.sh

# Linux
cd ./setup && bash setup-linux.sh

Verifying the Installation

Regardless of operating system, running setup-common.ts with bun prints a checklist table verifying the installation status. If every item shows ✅, you're ready to proceed with the practice below.

cd ./setup && bun setup-common.ts

Among the tools this project installs, the ones directly relevant to this chapter's practice are as follows: Antigravity (Desktop), agy (Antigravity CLI), and also gh (GitHub CLI), which is used in later chapters.

The Antigravity CLI (agy) is a single compiled binary that requires no separate runtime installation — just unzip it and put it on your PATH to run it immediately. Older guidance still floating around, such as "install the Gemini command-line interface (CLI) globally via npm," is no longer valid. Check the latest installation method at antigravity.google/docs/cli-getting-started.
If a problem occurs during installation, logs are left in the %USERPROFILE%\workshop-setup-logs\ folder on Windows. Check these logs first when troubleshooting.
● Beginner Getting Started The first step: giving a goal in Antigravity (Desktop) and observing dynamic generation
G-1 Beginner Creating and Invoking Your First Custom Subagent
A developer trying the harness for the first time creates a new empty practice folder, harness-lab/, first writes the reviewer role as a tool-neutral AGENTS.md specification, and then "implements" it as a natural-language goal in Antigravity (Desktop) to invoke it. Since this folder is dedicated to practice, it won't mix with other project files.
Elements Used
AGENTS.md (Chapter 2) Defining a Subagent Invoking a Subagent
Step-by-Step Execution
  1. ① Create and enter the practice folder

    Create a new harness-lab folder under the current folder and move into it. Run all subsequent commands inside this folder.

    mkdir -p harness-lab
cd harness-lab
  2. ② Write the common role spec — AGENTS.md

    Write the reviewer role as plain text, free of tool-specific expressions, so that it reads with the same meaning whether opened in Claude, Antigravity, or any other tool that supports the AGENTS.md standard. This single file becomes the sole source of truth for "what this role does," and Antigravity below merely implements this source as a natural-language goal. On macOS/Linux/Git Bash, use the command below as-is.

    cat > AGENTS.md << 'EOF'
## Agent: reviewer

Role: Review text files for typos, logical gaps, and areas for improvement.
Input: Path to the text file to review
Output: A list of typos, logical gaps, and improvement points
Permissions: Read-only — does not modify the file directly.
EOF

    On Windows PowerShell, the cat command itself is an alias for Get-Content and can be used as-is for reading files. However, the heredoc syntax for entering multiple lines directly, like << 'EOF', isn't supported, so write the same content as a quoted here-string (@'...'@) and write it to the file with Set-Content.

    @'
## Agent: reviewer

Role: Review text files for typos, logical gaps, and areas for improvement.
Input: Path to the text file to review
Output: A list of typos, logical gaps, and improvement points
Permissions: Read-only — does not modify the file directly.
'@ | Set-Content AGENTS.md
Antigravity (Desktop) — Implementing the same AGENTS.md spec as a natural-language goal

Antigravity has no concept of predefined files like .claude/agents. Instead, keep the AGENTS.md you just wrote in the workspace as-is, and provide the same specification in your goal prompt.

  1. ① Prepare the file to review beforehand (terminal) 
    echo "This project is very good. This is really great." > notes.txt
  2. ② Create a Workspace in Antigravity Agent Manager

    Open Agent Manager and create a new Workspace pointing at the harness-lab/ folder (which already contains AGENTS.md).

  3. ③ Give the goal — paste the sentence below into the Workspace chat and send it 
    Review notes.txt according to the reviewer role spec written in AGENTS.md.
    The agent handling the Workspace reads AGENTS.md, takes the role/input/output/permissions spec inside it as its goal, and decides on its own the tasks required (reading the file, compiling improvement points). Since AGENTS.md is the sole source of truth for "what criteria to review by," the review criteria (typos, logical gaps, improvement points) are determined by this spec.

The Antigravity CLI (agy) also lets you practice the same concept — give only the goal via /goal, with no predefined file.

cd harness-lab && agy
/goal Review the notes.txt file and summarize the improvement points
Since Antigravity has no definition file, you can't restrict permissions in advance with a tools field like in the Claude family. Instead, it's safer to narrow the goal prompt itself clearly, such as "just read and summarize." The Permissions: Read-only spec written in AGENTS.md also serves to communicate the permission boundary to the orchestrator.
● Intermediate Daily Workflow A practical pattern where the main session becomes an orchestrator directing two specialist agents, writer and reviewer, in sequence
D-1 Intermediate writer → reviewer Two-Stage Pipeline
A practitioner who has completed G-1 first specifies a second role, writer, in the same harness-lab/ folder, in AGENTS.md the same way as in G-1, and then implements it in Antigravity (Desktop) as a natural-language goal. The orchestrator is made to keep to the order in a pipeline where writer drafts first and reviewer critiques the result.
Elements Used
AGENTS.md (Chapter 2) Defining a Subagent Invoking a Subagent
Step-by-Step Execution
  1. Add a common role spec — appending the writer entry to AGENTS.md

    Continue writing the writer role in the same AGENTS.md created in G-1. Leave the reviewer entry as-is and only add the content below. On macOS/Linux/Git Bash, use the command below as-is.

    cat >> AGENTS.md << 'EOF'

## Agent: writer

Role: Write a short 3–5 sentence draft on a given topic.
Input: Topic, path to save the file
Output: A draft text file
Permissions: Write allowed — saves only to the specified file.
handoff_to: reviewer (proceeds to review once the draft is complete)
EOF

    On Windows PowerShell, append the here-string with Add-Content (you must use Add-Content, since Set-Content overwrites the file and would erase the reviewer entry).

    @'

## Agent: writer

Role: Write a short 3–5 sentence draft on a given topic.
Input: Topic, path to save the file
Output: A draft text file
Permissions: Write allowed — saves only to the specified file.
handoff_to: reviewer (proceeds to review once the draft is complete)
'@ | Add-Content AGENTS.md
Antigravity (Desktop) — the orchestrator decomposes two of the same AGENTS.md specs into a single goal on its own

Antigravity doesn't split writer/reviewer into separate predefined files in advance. Instead, have it simply reference the two role specs already written in AGENTS.md (reviewer in G-1, writer just now). Continue in the chat of the same Workspace created in G-1.

  1. ① Give the two-stage goal — paste the sentence below into the Workspace chat and send it 
    Following the writer spec in AGENTS.md, first write a draft in draft.txt on the topic 'The advantages of remote work', and once it's done, review that draft following the reviewer spec in AGENTS.md. Proceed through the two stages in order.
    The orchestrator reads the handoff_to: reviewer relationship in AGENTS.md and splits the goal on its own into two subtasks, "write draft → review," starting the review task only after the draft writing is complete. The distinctive point is that the responsibility for sequencing lies inside the orchestrator, not with the user — though "what criteria to review by" still has AGENTS.md as the sole source of truth.

The Antigravity CLI (agy) also lets you give the same goal at once. Put the whole pipeline into a single sentence with /goal and the orchestrator will decompose it on its own; if needed, you can also hand off the review stage to the background with /agent and continue other work.

agy
/goal First write a draft in draft.txt on the topic 'The advantages of remote work',
then once it's done, review that draft and summarize the improvement points
D-2 Intermediate Observing How Team Composition and Workflow Change per Request
This scenario lets you directly see the "dynamic team composition" and "diverse workflow combinations" covered in Chapter 2 §4b. Leave the two role specs writer and reviewer created in D-1 as-is, and vary only the nature of the request to observe exactly what tasks the orchestrator actually performs, how many times, and in what order — the key point being that a subagent definition is never newly created.
Elements Used
Dynamic Team Composition and Diverse Workflows
Antigravity (Desktop) — the same flexibility shows up even from natural-language goals alone

Since Antigravity generates subagents dynamically from the start, this flexibility shows up more directly. Continue in the same Workspace chat used in D-1.

  1. ① A simple goal instruction 
    Just skim draft.txt for typos.
    The orchestrator judges that this request doesn't need a writing role and assembles only a review role on the spot. This is the simplest example showing that even though both roles are defined in AGENTS.md, the actual composition deployed is decided by the orchestrator each time, per request.
  2. ② A compound goal instruction 
    Write a new draft in draft2.txt on the topic 'The four-day workweek' first, and once it's done, review that draft independently and simultaneously from both a grammar perspective and a logic perspective.
    The orchestrator assembled only a single review role on the spot in ①, and in ② assembles a new team of one writing role and two review roles (with different perspectives). Since there are no predefined subagent files at all, the principle "assemble a new team fitted to this task every time" stands out more clearly.
Comparing ① and ② side by side makes clear that "which roles are defined in AGENTS.md" and "what composition is actually deployed for this task" are different layers. The former is fixed in advance in AGENTS.md, but the latter is decided anew by the orchestrator with every request.

● Advanced Power User Parallel operation running writer and reviewer simultaneously against two independent files — using Agent Manager
P-1 Advanced Parallel Execution of Independent Tasks — Agent Manager
An experienced practitioner who has finished through D-1 runs two mutually independent tasks (drafting topic-a.txt and topic-b.txt) in parallel in Antigravity (Desktop) Agent Manager.
Elements Used
Parallel Execution — Scaling to the Team Level Parallel Execution
Antigravity (Desktop) — Agent Manager, one agent per Workspace

Open Antigravity and prepare a Project pointing at harness-lab/ in Agent Manager. This time, instead of cramming two agents into one Workspace, create two separate Workspaces and assign one agent to each.

  1. ① Create Workspace A

    Create a Workspace using New Worktree Mode.

  2. ② Paste the sentence below into the Workspace A chat and send it 
    Write a 3–5 sentence draft in topic-a.txt on the topic 'Working from home'.
  3. ③ Create Workspace B

    Create one more separate Workspace (New Worktree Mode recommended). It's a completely separate instance from Workspace A's agent.

  4. ④ Paste the sentence below into the Workspace B chat and send it 
    Write a 3–5 sentence draft in topic-b.txt on the topic 'The four-day workweek'.
  5. ⑤ Check the two agents' progress in the Inbox

    While both agents work simultaneously in their own Workspaces, check the Inbox to see if any messages need approval or feedback, and respond as needed.

Do not assign the two agents to the same Workspace — sharing the same folder can cause "cognitive overlap," where one agent mistakes another agent's work context for its own. Always separating them via Workspace or worktree is Antigravity's recommended practice.

The Antigravity CLI (agy) can achieve the same effect by opening multiple terminals instead of the desktop's Agent Manager — work on topic-a.txt with agy in one terminal, and topic-b.txt with a separate agy session in another terminal, simultaneously (parallel across sessions). As long as the two sessions touch different files, they won't conflict.
