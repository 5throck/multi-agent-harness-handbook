Chapter 2

Harness Engineering and Multi-Agent Team Concepts

For an AI agent to work reliably on real tasks, "what framework it operates within" matters more than "what you tell it to do." Chapter 1 looked at why AI agents are needed. This chapter examines the perspective of harness engineering and the basic structure of multi-agent teams, where multiple agents collaborate as a team rather than relying on a single agent.

Sources: Claude Code Official Docs — Subagents, ai-workspace-standards repository, Faros AI — Harness Engineering, Anthropic — Building Effective Agents

What This Chapter Covers
  • The progression from Prompt Engineering → Context Engineering → Harness Engineering
  • What harness engineering is, and why prompt engineering alone isn't enough
  • The structural limits that emerge when a single agent handles complex work
  • The three components that make up a multi-agent team: orchestrator, specialist, and handoff contract
  • Subagents, the mechanism for actually implementing specialists: how they're defined, pre-defined vs. dynamically generated, and when to use them
  • Anthropic's five orchestration patterns: prompt chaining, routing, parallelization (sectioning/voting), orchestrator-workers, evaluator-optimizer (verify-then-apply), and retry/rollback/escalation for failures
  • How team composition can change dynamically per task, and how different workflows can be freely implemented per domain
  • The AGENTS.md standard, which lets different tools like Claude Code/App and Antigravity CLI/Desktop share the same harness

What Is an Agent

Before diving into harness engineering in this chapter, let's pin down exactly what an "agent" is. An AI agent is an AI system given a purpose, distinct from a simple chatbot.

A chatbot's job is simply to "answer questions." The user enters a prompt, it generates a response, and once the conversation ends, nothing persists. An agent differs in three ways.

  • Persistence of purpose: An agent has a purpose defined by a "role." Even as a conversation grows long, it acts consistently without losing its role (researcher, reviewer, writer, etc.).
  • Tool use: An agent can do more than generate text — it can read files, search code, and execute commands. Chapter 4 walks through controlling these tools via the tools field.
  • Autonomous behavior: Rather than a single response to a single prompt, an agent can autonomously perform multiple steps to achieve a goal. It judges for itself when to branch on conditions, retry on errors, and verify results.
The "multi-agent" concept introduced in Chapter 1 is a system that divides such agents into multiple roles and has them collaborate. This chapter covers the structure of that collaboration — orchestrator, specialist, handoff.

Prompt → Context → Harness — How We Got Here

Harness engineering didn't appear out of nowhere; it's the result of three stages built up in how we approach AI. Each stage didn't replace the previous one — it layered on top after revealing that the previous stage alone wasn't enough.

A cooking analogy makes the three stages easy to grasp. Prompt engineering is "what words do you give the chef for the order," context engineering is "what ingredients and recipe do you hand the chef," and harness engineering is "building a system where the kitchen itself — inspection procedures, tools, hygiene rules — produces the same quality every time." No matter how well you phrase the order or how good the ingredients are, if the kitchen system itself is sloppy, the results will be inconsistent every time.

Stage 1 — Prompt Engineering (2022–2024)

Early interest was purely about language. Once it became clear that result quality varied greatly depending on how a request was phrased, "how to write a better prompt" was treated as the core skill. AI at this stage was closer to sophisticated autocomplete, and humans had to steer direction at every step.

For example, simply asking "review this code" leaves the model unsure what criteria to use, producing a vague answer, whereas a specific instruction like "review this function's edge-case handling, naming consistency, and performance issues individually" produces a dramatically better result. This know-how about "how to refine what you say" was the whole game at this stage.

Stage 2 — Context Engineering (2024–2025)

As model performance improved, the bottleneck shifted from "expression" to "information supply." Curating what to fill the model's context window with — relevant files, project rules, architectural constraints — became important, and tools like RAG and MCP systematized this process. In September 2025, Anthropic officially defined this as "the technique of curating and maintaining the optimal set of tokens during inference."

No matter how good the prompt, if the model doesn't know our project's coding conventions or existing function structure, it will produce code in the wrong style. Context engineering is the technique of filling the context window only with what the model actually needs — a handful of files related to the request, project rule documents, summaries of past conversations. Too little information leads to guesswork; too much buries what matters in noise.

Stage 3 — Harness Engineering (2026)

This is the dominant paradigm as of 2026. The model itself is closer to raw material intelligence, and it's the harness that turns that intelligence into autonomous behavior you can actually trust. Tool orchestration, verification loops, context/memory management, guardrails, and observability — only once these five layers are in place does an agent reach a level where you can "actually hand it work." The saying "the agent itself isn't the hard part — the harness is" sums up this era.

If the same prompt and same context produce a different quality of result every time, it's unusable in practice. Harness engineering is about designing the system around the model's answer — not the answer itself — asking questions like "is there a way to automatically verify this result," "are dangerous operations (file deletion, executing payments, etc.) blocked without human approval," and "can we trace what went wrong in the logs when something fails."

The three stages shift focus from behavior (what to make the model do) → understanding (what to make the model know) → reliability (how to make the result repeatable and verifiable). The multi-agent teams this handbook covers are the product of the third stage: harness engineering.

What Is Harness Engineering

If prompt engineering focuses on "what words to give the model," harness engineering steps back and designs "what tools, rules, and verification mechanisms the model should work through." The term "harness" here comes from the equestrian tack used to control a horse — it refers to the framework that keeps AI from running off in any direction it pleases, instead moving predictably within defined procedures, tools, and permission boundaries.

In practice, this framework shows up as several concrete mechanisms: the list of tools an agent can call and an approval policy based on each tool's risk level; hooks that automatically run on certain events (right after a file save, at task completion, etc.); skills that standardize and reuse recurring task procedures; and orchestration scripts that combine all of this to run multiple agents in a fixed order and under fixed conditions. Once these mechanisms accumulate, you get a "reproducible AI workflow" whose quality and format don't waver no matter how many times you repeat the same request.

Take a concrete example. When a request like "delete this file" comes in, an AI without a harness might execute the deletion immediately. An AI with a harness behaves differently: deletion is classified as a "dangerous tool" subject to a policy requiring user approval before execution; once the deletion completes, the fact is logged; and there's even a procedure for undoing an accidental wrong-file deletion (e.g., checking the pre-commit state with git). The model's judgment is the same either way — it's the presence or absence of "safety mechanisms" surrounding that judgment that determines the reliability of the outcome.

A good prompt makes it answer well once. A good harness makes it answer well every time.

Limits of a Single Agent

Assigning every role to a single agent is convenient at first, but as work scales, three problems become clear. First, context overload: when design review, code writing, testing, and documentation are all handled in sequence within one session, the details of earlier steps fade as things go on, and the consistency of results drops. Second, role conflict: an agent that reviews its own code misses many flaws it wrote — the same problem you get when the author and the reviewer are the same person. Third, lack of parallelism: even independent sub-tasks (e.g., refactoring several files, reviewing code from multiple perspectives) must be processed sequentially by a single agent, so total time grows linearly.

A company org analogy makes this easy to understand. If you assign one new hire to handle planning, development, testing, and documentation all at once, it feels fast at first with no communication overhead — but as the workload grows, that person's head (context) turns into a mess, they overlook their own code's flaws (role conflict), and they can't work on three unrelated tasks at the same time (lack of parallelism), so they become a bottleneck. The reason organizations split planners, developers, and QA is exactly why AI work is also split across multiple agents.

A multi-agent team structurally solves all three: it separates context by role so each agent focuses on its own role, separates author from verifier to secure an independent perspective, and runs tasks with no mutual dependency concurrently to cut down total time.

Components of a Multi-Agent Team

A multi-agent team is made up of three main elements.

Basic structure: the orchestrator splits work and hands it off to specialists via handoff contracts (as covered in §4b, specialist composition varies dynamically per task)

Orchestrator

The role that breaks down the overall task into steps, calls the specialist agent suited to each step, and pauses progress at points (gates) where user confirmation is needed. Often called the "PM agent," it doesn't write code itself and instead focuses on coordination.

Think of it like a project manager (PM). A PM doesn't write code directly, but assigns "this task to the backend developer, this one to QA," and before important decisions, asks the team lead (the user) "should we proceed with this?" before moving to the next step. An orchestrator behaves the same way. For example, given a request to "add a new feature," the orchestrator itself breaks it into "1) design review → 2) implementation → 3) testing → 4) documentation" and hands each step off to the agent responsible.

Specialist

An agent with a narrow, clearly defined responsibility — design, implementation, testing, security review, etc. Each specialist has clearly defined inputs it receives and outputs it must produce, so the orchestrator can easily judge "what this agent can be trusted with."

For example, the test-runner specialist has only the responsibility of "running tests and reporting failed items." Writing new code or changing the architecture is not its job. By cutting responsibilities this narrowly, the orchestrator can instantly decide "if I need tests, I call test-runner," while the specialist can focus solely on its own role and produce more accurate results.

Handoff Contract

The format of information exchanged when handing work between agents. If handed off in natural language alone, the next agent has to re-guess the context; but with structured input (e.g., JSON) that spells out "this task's purpose, scope, and prior deliverables," the next agent can start work immediately.

Think of a courier handoff. Saying "just take this over there" forces the recipient to re-ask about the address, quantity, and precautions, but handing over a waybill (a document listing tracking number, recipient, quantity, special notes) lets the recipient start delivery right away. The handoff contract is the "waybill" between agents. When a writer explicitly hands off to a reviewer with "review this file against these criteria in this format," the reviewer can start immediately without asking again.

In practice, this is handed off as a structured document like the following.

{
  "task": "Code Review",
  "agent": "reviewer",
  "scope": {
    "target_file": "src/api/handler.ts",
    "change_description": "Added auth middleware — JWT token verification logic"
  },
  "deliverables": {
    "from": "code-writer",
    "files": ["src/api/handler.ts", "src/auth/verify.ts"],
    "commit_hash": "a3f7c21"
  },
  "review_criteria": [
    "Security: is token expiration handled",
    "Error handling: is there a 401 response for invalid tokens",
    "Tests: were relevant tests added"
  ],
  "output_format": {
    "type": "structured_comment",
    "fields": ["pass_fail", "findings", "suggestions"]
  },
  "failure_conditions": [
    "Report immediately if target_file does not exist",
    "Mark as block if security criteria are not met"
  ]
}

In this example, task, scope, and review_criteria define "what to review," and output_format defines "in what format to return results." failure_conditions covers how to handle problems in the review target itself. In this way, a handoff contract explicitly defines purpose, scope, criteria, format, and exception handling so the next agent can start work immediately without needing to ask again.

If any one of the three elements is missing, what you get isn't a team but "several agents each working separately." Coordination (orchestrator) and contracts (handoff) are what make a team actually a team.

What Is a Subagent

The specialist described above is a concept; a subagent is the mechanism that implements that concept inside an actual tool. A subagent is a helper agent with an independent context window separate from the main session — it handles only the narrow task assigned to it, then summarizes the result and returns it to the main session. Because the main session's conversation history doesn't accumulate the subagent's detailed back-and-forth and only retains the final summary, this structurally mitigates the "context overload" problem noted in §2.

Pre-defined vs. Dynamically Generated

How subagents are handled differs by tool. Claude Code/App uses a pre-defined approach. You place a name.md file under .claude/agents/, and specify name (the identifier referenced when invoking it), description (when to use this subagent), tools (the allowed tool list, to minimize privileges), and model (which model to run it with) in the YAML frontmatter. The main session spawns this definition via the Agent (Task) tool, calling subagent_type="name".

Antigravity, by contrast, is closer to dynamic generation. Rather than creating files in advance, you give the orchestrator a goal in natural language, and it composes and handles the necessary role on the spot at that moment. It's less "invoking a subagent called reviewer" and more the agent constructing whatever role fits the task, as needed, each time. Both approaches ultimately achieve the same goal — "handling a narrowly-scoped task in an isolated context" — but pre-definition is stronger in reusability and permission control, while dynamic generation is stronger in flexibility.

When to Use Subagents

Not every task needs to be broken into a subagent. Good candidates are tasks with clear boundaries (clear about what goes in and what comes out) that are either repetitive or parallelizable — code review, running tests, document review, and similar. Conversely, exploratory work that requires continuously exchanging context with the main session ends up costing more in handoff overhead if split off into a subagent.

A subagent is the "implementation" that makes the "role" concept of a specialist executable in an actual tool. In Chapter 4, we hand-build this definition method and calling approach ourselves, centered on Claude App/Antigravity Desktop.

Orchestration Patterns

The five workflow patterns Anthropic laid out in its December 2024 "Building Effective Agents" are the most widely used in practice: prompt chaining (pipeline), routing, parallelization (sectioning/voting), orchestrator-workers, and evaluator-optimizer (verify-then-apply). The five patterns aren't mutually exclusive alternatives — they're a toolbox to pick from or combine, based on "is there a dependency between tasks," "can sub-tasks be predicted in advance," and "is verification needed."

  • Prompt Chaining / Pipeline. Break a task into an ordered series of steps and place a verification gate between each. If you stream multiple items through concurrently, you get a pipeline where items don't wait on each other — item A can be at step 3 while item B is still at step 1, boosting total throughput.
    Example: like a factory assembly line. Product A is going through the painting stage while product B is already at the assembly stage. If you process three documents through "draft → review → proofread," document 2 can start drafting the moment document 1 moves into review.
  • Routing. First classify an incoming request, then route it to the specialist agent or model suited to its nature. Chapter 5's model tiering is a prime example — send easy questions to a cheap model (haiku), and difficult, rare questions to a high-performance model (opus), balancing cost against quality.
    Example: when a customer inquiry comes in, a classifying agent first judges "is this a simple FAQ or a complex technical question," routing the former to a formatted-response agent and the latter to an agent capable of deeper investigation.
  • Parallelization. Splits into two branches. Sectioning (fan-out) hands several independent sub-tasks (e.g., a code review from four perspectives) to multiple agents at once and gathers the results — total time converges to that of the slowest single task. Voting instead has the same task independently repeated across multiple agents, then picks one answer by majority or consensus.
    Sectioning example: "review this code simultaneously from the perspectives of performance, security, readability, and test coverage." Voting example: have three agents independently analyze the same bug's root cause, and if two or more point to the same cause, adopt that conclusion — filtering out one agent's mistake via majority vote.
  • Orchestrator-Workers. The team structure described in §3 is itself this pattern. Rather than pre-deciding how to split the work, a central orchestrator decides dynamically on the fly based on the input, delegating to workers (specialists) and synthesizing the results. It's especially suited to complex tasks whose sub-tasks can't be predicted in advance (code changes spanning many files, research that must synthesize multiple sources). The "dynamic team composition" covered in §4b is another name for this exact pattern.
  • Evaluator-Optimizer / Verify-then-Apply. One agent produces a draft, and only after an independent agent (or the consensus of multiple agents) challenges and verifies that draft does it actually get applied. This is a mechanism that prevents plausible-but-wrong results from slipping through.
    Example: before modifying code, an independent agent re-reviews from a skeptical stance whether "this change is really safe," and only if judged fine does the file actually get modified. It's the same principle as how another person catches errors an author misses when proofreading their own writing. That said, it has a limit — if the evaluator itself can't distinguish good results from bad ones, this loop can spin without making progress.
The five patterns are more often layered together than used in isolation — routing might feed into the first step of a prompt chain, orchestrator-workers might embed an evaluator-optimizer within each worker's unit of work, and parallelization might slot into the middle of another pattern. §4b covers a concrete example of this combination.

Three of the five patterns visualized: pipeline, fan-out, and verify-then-apply. See the main text for the full list including routing and orchestrator-workers. Patterns aren't substitutes for one another — they're a toolbox meant to be combined.

On Failure — Retry, Rollback, Escalation

All three patterns above assumed "when things end normally," but in practice, subagents sometimes produce wrong results or fail outright. Verify-then-apply is a mechanism for "filtering before execution," while what to do once a problem is discovered after execution has already happened is a separate matter. There are three typical responses.

  • Retry. If it's a transient error (network timeout, tool call failure, etc.), retry the same task. However, retrying the same prompt indefinitely just repeats the same mistake, so a cap on retry count is usually set.
  • Rollback. If a wrong result has already been applied to files or a repository, revert to the state before the change. The principle from Chapter 3 — "write steps should stay within a range git can undo" — really pays off here. Work must be split at the commit level for rollback to land on the exact right point.
  • Escalation. If retries keep failing, or if the situation is inherently ambiguous for the orchestrator to judge, hand it to a human. Without a predefined ceiling like "stop automatic retries after 3 failures and report to a human," you end up burning time and cost endlessly retrying a task that keeps failing.
The criterion for choosing among the three ultimately comes down to "is this failure transient, has it already reached an irreversible state, or is the judgment itself ambiguous." Retry addresses the first, rollback the second, escalation the third. The riskier the task, the safer it is to lower the bar toward escalation rather than automatic retry (see also the risk-tier classification in Chapter 3 §2).

Dynamic Team Composition and Diverse Workflows

So far we've covered the orchestrator, specialist, handoff, and orchestration patterns separately, but the real power of harness engineering lies in the fact that these elements are not locked into a single fixed combination. Conventional automation scripts hardcode order and participants into the code — "A then always B, then always B then C" — but a multi-agent team built on a harness has the orchestrator judge afresh, for every request, "who's needed this time and in what order."

Team Composition Is Not Fixed

Even the same team calls a different number and combination of specialists depending on the nature of the request. For a simple request like "just fix the typos," the orchestrator may call only reviewer and finish; for a request like "take a new feature from design through deployment," it mobilizes architect · code-writer · test-runner · security-monitor · docs-writer in sequence (or partly in parallel). Even if the roster itself (which subagents are defined) is fixed in advance, who actually gets deployed for a given task is decided fresh every time. It's like a soccer team whose full 22-player squad is fixed, but whose starting 11 for any given match is chosen fresh each time based on the opponent and tactics.

This flexibility also connects to the two implementation approaches for subagents covered in §3b. Even under the pre-defined approach (Claude Code/App), the orchestrator dynamically picks, every time, "which of the defined subagents to call this time"; under the dynamic-generation approach (Antigravity), it goes a step further — the subagent's very existence is constructed on the spot to fit the moment's needs. Either way, the principle "the task decides team composition" holds.

Diverse Workflows Can Be Built on the Same Harness

What harness engineering fixes in place is only the skeleton — "there is an orchestrator, it calls specialists, and it passes information via handoffs." How many pipeline stages to build on that skeleton, and with how many participants, can be designed completely differently per domain. The variants of ai-workspace-standards, covered in Chapter 5, demonstrate this. Built on the same harness principles, co-develop is a 6-stage governance pipeline involving PM, Architect, Designer, Code Writer, Test Runner, and Security Monitor; co-design is a 5-stage iterative workflow; and co-deck is an 11-stage workflow — each shaped completely differently. The core advantage of harness engineering is that it doesn't force a single "correct workflow" but lets you design the workflow itself to fit whatever number of stages, participants, and verification rigor a domain demands.

This also means the five orchestration patterns from §4 (prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer) aren't mutually exclusive and can be freely combined within a single workflow. For example, a code review workflow might first use routing to classify whether a change is small or large; for large changes, an orchestrator-worker dynamically splits the affected files and fan-out reviews each from four perspectives simultaneously; the results are then processed sequentially in a pipeline for documentation; and finally, evaluator-optimizer (verify-then-apply) decides whether to actually apply the change — mixing all five patterns into one flow.

Dynamic team composition and workflow diversity are two faces of the same principle. The harness fixes only "what must be upheld," leaving "who moves when and how" open for the task and domain to decide. Without this flexibility, every variant would have needed an entirely different automation system built from scratch. If you'd like to verify this hands-on, the D-2 scenario in Chapter 4 §2's practice exercises lets you observe team composition and workflow actually changing with the same subagent team, just by varying the request.

A Cross-Tool Harness — AGENTS.md

The orchestrator, specialist, and handoff described so far are not concepts tied to a specific tool. The problem has been that each tool configures this harness differently. Claude Code reads CLAUDE.md, Cursor reads .cursorrules, GitHub Copilot reads .github/copilot-instructions.md — so teams using multiple tools together had to maintain duplicate copies of the same content across each file.

If you copy the same content (e.g., "always run tests with npm test, and never modify the src/legacy/ folder") once per tool your team uses (CLAUDE.md, .cursorrules, copilot-instructions.md...), every rule change requires finding and editing all of them, and missing even one leaves whoever uses that tool working under different rules. It's the same problem as separately translating and keeping the same contract in multiple languages, then having versions drift out of sync when only one gets amended.

AGENTS.md is an open standard created in August 2025, led by OpenAI with participation from Google, Cursor, Factory, and others, to solve this problem, and it was donated to the Agentic AI Foundation under the Linux Foundation in December 2025. Placing a single AGENTS.md at the repository root lets over 30 tools — including Codex, Claude Code (via import), GitHub Copilot, Cursor, Google Jules, and the Antigravity family — read it at session start and share common rules like build/test commands, code style, and off-limits areas.

This handbook's ongoing emphasis on "support for all four tools" ultimately comes from the same problem awareness. Whether a team uses Claude App or Antigravity Desktop, keeping shared rules at the repository root (AGENTS.md) separate from per-tool detailed configuration (CLAUDE.md, GEMINI.md, etc.) means the harness itself stays stable even when tools are switched or mixed. The ai-workspace-standards repository, examined in Chapter 5, is a real-world implementation of this principle. The root AGENTS.md file serves as the single source of truth (SSOT) for all agents in the workspace, and the .agents/ folder sits alongside .claude/ and .gemini/, holding engine-agnostic commands and skills.

AGENTS.md acts as the shared spec (SSOT), while per-tool files only add how to execute it in each tool — a layered separation that reduces duplication

AGENTS.md tool-neutrally defines "what must be upheld," and per-tool files like CLAUDE.md/GEMINI.md only additionally define "how to execute that in this particular tool." It's a layered separation that reduces duplication.
