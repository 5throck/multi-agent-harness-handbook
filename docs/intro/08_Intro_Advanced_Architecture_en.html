Advanced Architecture

Where §1 covered "what flows where" across L0→L1→L2, this section digs into how that flow is actually implemented — why files are regenerated rather than copied verbatim, the order in which inherited frontmatter is resolved, concrete examples of how boilerplate and tier strategy are applied in practice, and how the two platforms (.claude/.gemini) are kept from structurally drifting apart.

What This Section Covers

  • The actual resolution order of the extends frontmatter field
  • Examples of applying the LOCKED/MERGE/PRESERVE classification to real files
  • The 3-tier strategy — High/Medium/Low definitions, frontmatter examples, Tier Adjustment Rules, and the Tier Ceiling Rule
  • How .claude//.gemini/ platform parity is enforced

Layout Reconstruction — Why L2 Doesn't Copy L1 Verbatim

Let's dig deeper into the agents/pm.md case briefly mentioned in §1. L0's agents/pm.md holds workspace-wide governance and runs over 300 lines. It mixes in sections that only make sense at the workspace root, such as Auto-Mode behavior and lifecycle state fields. If L1's templates/common/agents/pm.md copied this verbatim, every time a new variant was scaffolded it would drag along workspace-only content that has nothing to do with the project.

That's why L1 performs filtered reconstruction instead of copying — it picks out from the L0 original only the sections actually needed for variant scaffolding, strips the workspace-only sections, and links the rest via an extends reference. The same principle repeats on the way down to L2. Variant-specific overrides (for example, co-marketing's marketing-domain-specific PM rules) are layered on top of L1 and regenerated from scratch, rather than patching the L1 file via diff. As a result, even though L0's pm.md exceeds 300 lines, L2's pm.md stays around 50–100 lines — because the approach is "build fresh with only what's needed," not "copy and delete the leftovers."

This design matters because of the failure mode it prevents. If L1→L2 were a simple copy, every time L0 added a single workspace-only section, every L2 project's pm.md would bloat unnecessarily. Layout Reconstruction blocks this at the root by always placing a filter between "what L0 knows" and "what L2 needs to know."

extends Frontmatter — The Order in Which Inheritance Is Resolved

A frontmatter field like extends: ../../../agents/pm.md is a declaration that "this file references that file" — it is not a macro that merges files. The resolution order is as follows.

  1. When a tool reads the file (for example, when Claude Code loads agents/pm.md), it first checks the frontmatter's extends path.
  2. It reads the referenced original (L0 or L1) and inherits the body, but if the current file explicitly rewrites a section, only that section is overridden.
  3. Sections that exist only in the current file and not in the original (e.g., variant-specific rules) are appended as-is.

In other words, extends is resolved in three stages — "inherit + selective override + append" — not by concatenating two files as text. Because of this structure, L1's templates/common/agents/pm.md can contain almost no actual content (mostly just an extends pointer back to L0) and still function as a valid file, and when a shared rule changes in L0, every downstream file using extends automatically reflects the latest content the next time it's read — however, this is "resolved at read time," a different mechanism from the "file sync (SYNC_IF_NEWER)" covered in §1. extends is a live reference resolved the moment a tool opens the file, while SYNC_IF_NEWER is a static file copy that only updates when upgrade-project.ts is run. Both mechanisms look like "inheritance," but you must not confuse the fact that they update at different times.

Boilerplate Strategy in Practice

Let's walk through, one by one, how the LOCKED/MERGE/PRESERVE classification covered in §1 is applied to actual files.

  • LOCKED example — .githooks/pre-commit: this file runs security gates such as secret scanning and commit message validation. If this validation differed per project, the organization's overall security bar would vary project by project — so upgrade-project.ts always overwrites this file with the latest L1 version regardless of local modifications.
  • MERGE example — CLAUDE.md: this file contains a block wrapped in <!-- WORKSPACE-MANAGED -->...<!-- /WORKSPACE-MANAGED --> (e.g., PM gateway rules) alongside project-specific rules outside that block (e.g., "this project always writes commit messages in Korean"). On upgrade, only the inside of the marker is replaced with the latest L1 version, while the project-specific rules outside the marker are left untouched.
  • PRESERVE example — docs/context.md: this file documents what the project is, so L1 only checks that it exists and never touches its contents.

Looking at the three examples side by side clarifies the classification criteria — would a change lead to a security/governance incident (LOCKED)? Must organizational rules and project context coexist in one file (MERGE)? Is it the project's identity itself (PRESERVE)? When adding a new file to this repository's boilerplate, asking these three questions in order will usually make the category obvious right away.

3-Tier Strategy Deep Dive — Applying Cost Optimization at Runtime

The tier assignment introduced in Chapter 5 is statically defined in agents/*.md frontmatter. The tier field of each agent file is translated into a platform-specific model alias and applied at execution time.

Frontmatter Example

---
name: pm                       # pm, code-writer, automation-engineer ...
tier:                          # platform-specific model alias (translated at runtime)
  claude: high                 #   high = claude-opus-4-7
  antigravity: high            #   high = gemini-3.1-pro
---

Because this translation happens automatically behind model: inherit, whoever writes agents/*.md never needs to know a concrete model name — they only need to think about "how heavy is the judgment this role makes."

Tier Definitions and Model Mapping

TierPurposeClaude-family ExampleAntigravity-family Example
HighComplex reasoning, architecture design, planningclaude-opus-4-7gemini-3.1-pro
MediumCode review, testing, PR review, quality gatesclaude-sonnet-4-6gemini-3.5-flash
LowFast, repetitive coding, script maintenanceclaude-haiku-4-5gemini-3.5-flash

This static assignment forms the baseline, but at actual execution time the PM applies Tier Adjustment Rules to dynamically optimize cost. An agent with a high baseline (architect, pm) needs the strongest model because it must make complex design judgments. automation-engineer, on the other hand, writes standardized scripts per ADR-0036, so the low tier is sufficient — the only question that matters is whether "heavy judgment" is required.

Tier Adjustment Rules — Cost Optimization at Execution Time

When the PM builds an execution plan, tier adjustments away from the baseline are allowed. The rules are simple.

  • Downgrades are allowed — for simple tasks, the PM may lower an agent's tier below its baseline. For example, when docs-writer (baseline medium) receives a task like "fix a single typo in README.md," it can be dropped to low to save cost.
  • Upgrades are forbidden — conversely, raising an agent above its baseline tier is always forbidden. Raising automation-engineer (low) to high to design complex architecture is a governance violation.
  • Restore on failure — if a downgraded task fails, the PM must restore the agent's baseline tier and retry.

The core of this rule is the principle that "an agent role's complexity ceiling is fixed, but its floor can be adjusted to the situation." The PM can lower the tier to low to save cost, but the moment a task exceeds low's capability, it immediately reverts to the baseline. The cost impact can be felt directly in the Chapter 4 §2-A P-2 exercise.

Tier Ceiling Rule

AGENTS.md's Phase Determination table (§3.5) specifies the allowed tier ceiling for each deliverable type. For example, "script implementation (with an approved plan already in place)" is Phase 4, and the responsible agent is automation-engineer (low) — assigning high here would be a Tier Ceiling violation. By contrast, "designing new files, defining schemas" is Phase 1-2, handled by architect (high).

The Tier Ceiling Rule is not "never assign more than the tier best suited to this task" — it is "never use more than this agent's defined tier." It is role-based, not task-based.

Platform Parity — Keeping .claude/ and .gemini/ Structurally Identical

This repository supports two platforms simultaneously — the Claude family (.claude/) and the Antigravity/Gemini family (.gemini/). If the two directories drift apart over time (for example, if .claude/agents/reviewer.md exists but .gemini/agents/reviewer.md is missing), users of one tool end up unable to use a given feature. The mechanism that prevents this is the platform parity check in validate-templates.ts.

This check structurally cross-references the two directories to confirm there is no agent, skill, or command that exists on only one side. The promotion pipeline covered in Chapter 11 also includes this check as a mandatory gate before promotion, so a state where something was "built only for Claude and Antigravity was forgotten" can never become an official variant in the first place. Even though the two platforms use different execution mechanisms (see the CLAUDE.md/GEMINI.md MERGE example in §1), the principle that the set of features provided must always be the same is enforced through this structural check.

Execution Plan Templates — Structured Execution Design for Multi-Agent Work

When the PM distributes complex work across multiple agents, it uses the Execution Plan format. This plan spells out "who does what, at what tier, in what order" in a table, guaranteeing consistency of execution.

Standard Execution Plan Format

#TaskAgentTierModelSpec
0Create/update design doc → docs/designs/<spec-id>-design.mdarchitectHigh[model]NEW
1[task description][specialist]High/Medium/Low[model]<spec-id>
N/sync "type(scope): message"pmMedium[model]

Design Gate (Row 0)

Row 0 is mandatory only for L0 (workspace root) and L1 (common templates). Architect must first write the design document, and only after the PM verifies it can implementation (Row 1 onward) proceed. L2 variant projects manage their own workflow, so this gate does not apply to them.

Design Gate Exception Categories

For simple tasks where Row 0 is excessive overhead, only the following 5 exception categories (E1–E5) are recognized. The PM cannot invent exceptions on its own — only the categories defined below may be used.

IDCategoryDescription
E1memory-logWriting a memory/YYYY-MM-DD.md session log
E2changelogCHANGELOG.md update only
E3hotfix-typoTypo fix, single-line change, minor correction
E4pure-readmeREADME.md body text only (no structural/design change)
E5sync-only/sync run only (lifecycle closeout)

When an exception applies, the Agent/Tier/Model columns of Row 0 are shown blank (—).

Worked Example: An Execution Plan Written by the PM

Below is an actual execution plan example for "a platform parity task updating agents/pm.md and scripts."

#TaskAgentTierModelSpec
1Update agents/pm.mddocs-writerMediumsonnet<spec-id>
