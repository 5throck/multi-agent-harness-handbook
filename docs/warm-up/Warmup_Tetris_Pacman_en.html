From Tetris to Pac-Man

Even when the task is the same — "build a game" — the right approach changes as complexity grows. You'll build Tetris with vibe coding, then fail at Pac-Man with a single prompt, retry it with a three-stage plan/build/QA multi-agent approach, and feel the difference firsthand.

Platform: claude.ai or Claude Code · Duration: 40 minutes (Tetris 10 min + Pac-Man 25 min + takeaway 5 min) · Related: Chapter 2 §2 Single-Agent Limits

What this warm-up teaches
  • Tetris (10 min) — completed with a single vibe-coding prompt; edits apply cleanly too
  • Pac-Man (25 min) — experience failure with a single prompt → succeed with a three-stage plan/build/QA multi-agent approach
  • Takeaway (5 min) — the higher the complexity, the more exponentially valuable role separation becomes

Tetris — Complexity manageable with vibe coding (10 min)

In the Chapter 4 exercise, participants are split into 4 groups such as Claude / Antigravity, but in this warm-up everyone uses the same platform (claude.ai or Claude Code). This keeps the focus on the difference in approach rather than differences between harness tools.

How to run it

  1. Instructor demo (1 min): Enter the following prompt into claude.ai (or Claude Code).
Build a Tetris game using HTML Canvas

Within 10-20 seconds, a playable Tetris game appears.

  1. Participants follow along (5 min): Each participant enters the same prompt to build their own Tetris game.
  2. Request edits (4 min): Freely request modifications to the finished Tetris game.
Change the background to a darker color
Make the scoreboard bigger
Add a next-block preview
Key message: Tetris is a relatively simple game that can be completed in roughly 200 lines of code. At this level of complexity, a single vibe-coding prompt — without a harness — is enough to produce a working result, and edit requests are applied without issue.

Pac-Man — Limits created by complexity (25 min)

This time you'll build a game that looks similar but is far more complex. Pac-Man has many more elements — character movement, ghost AI, collision detection, map design, an item system — and reaches roughly 1,000-1,500 lines once complete.

Stage 1: Experience failure (2-3 min)

First, try it the exact same way you did with Tetris.

Build a Pac-Man game using HTML Canvas

When character movement, ghost AI, collision detection, and map design are all packed into a single prompt at once, the code gets tangled and doesn't work as intended. This "failure" is the core experience of this warm-up.

Why does it fail? Giving one agent too many roles at once causes the context overload discussed in Chapter 2 §2. Each feature may be implementable on its own, but coordinating them all simultaneously produces conflicts and omissions.

Stage 2: Retry with multiple agents (about 7-8 min per step)

This time, instead of a single agent, split the work into three roles and proceed sequentially.

Step 1 — Planning agent

Write a spec for Pac-Man's feature list and map structure. This step defines which characters exist, what the map looks like, and what the game rules are.

Write a spec for the feature list and map structure of a Pac-Man game. Include the following:
- How the characters (Pac-Man, 4 ghost types) behave
- Map structure (walls, dots, power-up locations)
- Game rules (score, lives, levels)
Step 2 — Development agent

Based on the spec produced in Step 1, write the actual HTML Pac-Man code. Since the plan is now clear, the development agent can focus purely on implementation without needing to figure out "what" to build.

Find and fix bugs in the finished code. Actually run the game to verify whether characters pass through walls, whether the ghost AI behaves correctly, and whether scoring is calculated correctly.

Takeaway — the higher the complexity, the more exponentially valuable role separation becomes (5 min)

TetrisPac-Man
Code volume~200 lines~1,000-1,500 lines
Core elementsBlock falling, rotation, line clearingCharacter movement, ghost AI, collision detection, map design, items
Single prompt✅ Succeeds❌ Fails
3-stage multi-agentUnnecessary✅ Succeeds

Simple tasks like Tetris are perfectly handled by vibe coding (a single prompt). But as complexity rises, as with Pac-Man, leaving every role to a single agent has its limits.

This is a firsthand experience of the limits of a single agent covered in Chapter 2 §2. Context overload, role conflicts, omissions — all of these problems are resolved once roles are separated.

Key message:

"The higher the complexity, the more exponentially valuable role separation becomes"

The handoff (plan → build → QA) you experienced manually in this warm-up will be experienced more deeply in the upcoming Chapter 4, automated through the subagent features of Claude Code, Claude App, and Antigravity.
