Chapter 8 §2

Lifecycle Management

This section covers the stages that agents, skills, and scripts each pass through from creation to retirement, and how these stages are actually managed within a real workspace.

What this section covers
  • Agent lifecycle: create → verify → operate → improve → deprecate
  • Skill lifecycle: define → test → deploy → version → retire
  • Script lifecycle: write → lint → test → register → deprecate
  • The distinction between state and lifecycle
  • Why lifecycle management is necessary

Why Lifecycle Management Is Necessary

As you operate a multi-agent system, the number of agents, skills, and scripts in the workspace keeps growing. Simply creating new components isn't enough — you must clearly establish what stage each component is in. Doing so lets you avoid the following problems.

  • Unverified components entering production. If an agent is deployed without passing agent:verify, missing frontmatter fields or an incorrect role definition can cause errors in live sessions.
  • Deprecated components still being invoked. If a skill that is no longer valid remains in the registry, an agent may unintentionally select it and fail.
  • Irreproducibility from version mismatches. If a script is updated but some variants still reference an older version, the same input produces different results.

To systematically prevent these problems, ai-workspace-standards defines named stages for agents, skills, and scripts respectively, along with the validation and transition conditions required at each stage. This section explains each of the three lifecycles and how they are actually managed within the workspace.

Agent Lifecycle

Agents are defined as .md files under the agents/ directory. From birth through operation to eventual retirement, an agent passes through the following five stages.

Create — Creating an Agent

The agent lifecycle begins by creating a new .md file in the agents/ directory. This file contains frontmatter (fields such as name, role, model) and a body (the system prompt). At creation time, confirm the following.

  • Whether the filename clearly reflects the agent's role (e.g. pm.md, reviewer.md)
  • Whether the frontmatter includes all of name, role, model, and status fields
  • Whether the body clearly states the agent's operating principles and constraints

Set the initial status field to draft. In this state, the agent is not automatically selected in production sessions.

Verify — Verifying an Agent

Verify that the created agent file conforms to the workspace specification by running bun run agent:verify. This command checks the following.

  • Whether any frontmatter fields are missing or mistyped
  • Whether the agent is correctly registered in AGENTS.md
  • Whether the skills and scripts it references actually exist
  • Whether its role overlaps with another agent's

Once verification passes, change status to active. If it fails, the agent remains in draft; fix the issues based on the error messages and re-verify.

Operate — Operating an Agent

An agent in the active state can be invoked in real sessions. During the operate stage, observe the agent's actual behavior and monitor session logs for the following.

  • Whether the agent behaves consistently with its defined role
  • Whether there is interference or conflict when collaborating with other agents
  • Whether response quality meets the expected bar

Issues discovered during operation are logged and become input for the Improve stage.

Improve — Improving an Agent

Update the agent file based on operational feedback. Typical improvements include the following.

  • Clarifying ambiguous instructions in the system prompt
  • Adding missing constraints or exception-handling rules
  • Updating the model field when a model change is needed

After modifying the file, you must run bun run agent:verify again to re-verify it. This loop back to the Verify stage is the core feedback cycle in the agent lifecycle. Once verification passes, the agent re-enters the Operate stage.

Deprecate — Deprecating an Agent

When an agent is no longer needed, change its status to deprecated. At deprecation time, perform the following.

  • Remove or comment out the agent's entry in AGENTS.md
  • Remove dependencies from any variants or scripts that referenced the agent
  • Preserve the file itself (rather than deleting it) with status: deprecated, so it can be restored if needed

After a set period (e.g. 30 days), once it is confirmed there are no remaining references, move the file to an archive directory or delete it permanently.

bun run agent:verify is the core tool of the Verify stage in the agent lifecycle. This command goes beyond simple syntax checking — it confirms that the agent is consistently connected to other components in the workspace. Including this check in a CI/CD pipeline systematically prevents unverified agents from entering production.

Skill Lifecycle

A skill is a reusable unit of behavior that an agent can invoke during a session. It is defined by a SKILL.md file and placed either in a plugin or in the workspace's .zcode/skills/. A skill's lifecycle consists of the following five stages.

Define — Defining a Skill

A skill is defined by a SKILL.md file. This file contains frontmatter (name, description, triggers, etc.) along with a description of what the skill does. At definition time, clarify the following.

  • name: the skill's unique identifier
  • description: the explanation that determines when an agent should select this skill
  • triggers: the conditions under which the skill is automatically activated
  • Body: the procedure the agent should follow when executing the skill

Initially leave it as status: draft so agents cannot yet automatically select this skill.

Test — Verifying a Skill

Verify that the skill definition is correct by running validate-skills.ts. This script checks the following.

  • Completeness and format consistency of the frontmatter fields
  • Whether trigger conditions ambiguously overlap with other skills
  • Whether the tools it references (e.g. MCP tools) are available in the current environment
  • Whether the skill description is specific enough for the agent to correctly understand

Once verification passes, the skill is ready to move to the Deploy stage.

Deploy — Deploying a Skill

Deploy a verified skill into an environment where agents can actually invoke it. Concretely, deployment means the following.

  • Changing status to active so it's included among the agent's auto-selection candidates
  • Registering it in the workspace's .zcode/skills/ or a plugin registry as needed
  • Declaring the dependency in the frontmatter of any agent that uses this skill

After deployment, perform a manual review to confirm that the skill triggers and executes correctly during an agent session.

Version — Skill Version Management

When a skill in active use needs a change, update it safely through version management.

  • Record the skill file's change history in the SKILL.md header comment or a changelog
  • For breaking changes, keep the existing skill intact and clone it into a new version before updating
  • Always re-verify with validate-skills.ts after making changes

The loop from the Version stage back to the Test stage is the core feedback cycle of the skill lifecycle. Once the updated skill passes verification, it re-enters the Deploy stage.

Retire — Retiring a Skill

Retire a skill once it is no longer used or has been replaced by another skill.

  • Change status to retired to exclude it from agent selection candidates
  • Remove the dependency from the frontmatter of any agent that referenced the skill
  • Preserve the file but explicitly mark it as retired so the original can still be referenced if reused
validate-skills.ts is the core tool of the Test stage in the skill lifecycle. This script checks not only basic formatting but also trigger conflicts between skills and referential integrity. It must be run every time a new skill is added or an existing skill is modified. Like agent:verify for agents, including it in a CI/CD pipeline automatically prevents deployment of unverified skills.

Script Lifecycle

Scripts are managed as .ts files under the scripts/ directory. They are executable code that automates workspace infrastructure — agent-creation verification, template publishing, project scaffolding, and more. A script's lifecycle consists of the following five stages.

Write — Writing a Script

A new script is written as a .ts file in the scripts/ directory. When writing, follow these guidelines.

  • The filename should clearly indicate the script's purpose (e.g. publish-to-template.ts, validate-skills.ts)
  • Record the purpose, usage, and argument descriptions as comments at the top of the script
  • Define types clearly and minimize external dependencies
  • Include error handling that produces a meaningful message on failure

A script is considered to be in the draft state immediately after being written.

Lint — Static Analysis

Run static analysis (lint) on the written script. In a TypeScript-based workspace, use bun run lint or tsc --noEmit to check the following.

  • No type errors
  • No unused variables or imports
  • Consistent code style
  • Warnings for potential runtime errors (e.g. null references)

All issues found during static analysis must be fixed before proceeding to the next stage.

Test — Testing

Run unit and integration tests on a script that has passed static analysis. If the script depends on the file system or network, test it in a mocked environment.

  • Happy path: does it produce the correct output for expected input?
  • Exception paths: does it handle errors appropriately for invalid input, missing files, permission errors, etc.?
  • Integration tests: does it work correctly when integrated with other scripts or agents in the real workspace environment?

Only scripts that pass testing can move to the Register stage.

Register — Registering a Script

Register a script that has passed testing in the scripts section of package.json so it can be run as bun run <name>.

  • Add an appropriate script name and run command to package.json
  • Record usage instructions in the script's documentation (this handbook or a separate README)
  • Document the dependency if other scripts or agents call this script

After registration is complete, do a final check that bun run <name> runs correctly.

Deprecate — Deprecating a Script

Deprecate a script once it is no longer needed or has been replaced by another script.

  • Remove the script's entry from package.json
  • Remove the dependency from any other script that called it, and update it to use the replacement path
  • Either move the file itself to an archive directory such as scripts/_deprecated/, or preserve it with an @deprecated comment marker
In the script lifecycle, human-performed manual review is recommended once between Lint and Test, and again between Test and Register. Automated checks cannot catch every problem. This is especially critical when a script modifies the workspace's file structure (e.g. publish-to-template.ts) — having a human visually confirm the actual outcome is essential.

Shared Principles and Interdependencies Across the Three Lifecycles

The Distinction Between State and Lifecycle

So far we've described three kinds of lifecycles as pipelines. There is one distinction here that should not be confused — lifecycle and state are different concepts.

  • A lifecycle refers to the entire sequence of stages a component passes through. For an agent, the five-stage pipeline itself — Create → Verify → Operate → Improve → Deprecate — is the lifecycle.
  • State is a snapshot of where the component currently is at a given point in time. The status field in YAML frontmatter (draft, active, deprecated, retired) is the mechanism that stores this state value.

To use an analogy, the lifecycle is a "subway route map," and the state is the real-time display showing "which station the train is currently at." Where the train is headed (the route) is fixed, but where it currently is (the station) changes moment to moment.

State at the level of workflow execution — for example, the Phase Pipeline defined by the PM in Chapter 9 (Phase 0: request received → Phase 1: design → … → Phase 6: sync complete) — is a different dimension of state from an individual component's status field. The former is "what stage the work is at," while the latter is "where the component itself is in its lifecycle."

Although the lifecycles of agents, skills, and scripts are each defined independently, they are tightly interconnected in an actual workspace.

The three lifecycles share the following three common principles.

  • Explicit stage marking: every component must clearly indicate its current stage via a status field or file location. Anything in draft is not put into production, and anything deprecated/retired is excluded from new invocation targets.
  • Verification gates: before moving from one stage to the next, a defined verification must be passed. If verification fails, go back to the previous stage, fix the issue, and re-verify.
  • Retain after deprecation: deprecated components are not deleted immediately but preserved for a period of time. This allows restoration if needed and keeps a traceable record of the deprecation decision.
