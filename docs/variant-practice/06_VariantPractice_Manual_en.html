Repository Structure Walkthrough — 5 Places to Check First After Cloning

After cloning the repository into the workspace root with git clone, let's walk through, in order, the 5 core directories you should check first. This order matches the "runtime flow" covered in Chapter 5.

  1. CONSTITUTION.md — The file to read first. It defines the shared standards for the entire workspace (governance, PR workflow, multi-agent principles).
  2. CLAUDE.md / GEMINI.md — Check the file matching the AI coding tool you use. Each defines platform-specific behavior (hooks, slash commands, agent dispatch rules).
  3. AGENTS.md — The roster of agents callable in this workspace. Lets you see at a glance "who does what."
  4. templates/ — All 7 available variants live under this directory. Open each variant folder to examine its agent composition and governance pipeline.
  5. scripts/ — new-project.ts lives here. This script is the core command that clones a variant into an actual project.
This walkthrough is a short guide meant to give you a sense of "what this repository contains" before you get hands-on with agents in the Chapter 4 exercises. During the exercise, open just these 5 places first, and come back to explore the rest of the structure as needed.

1. Local Environment Setup

git clone. Clone into the workspace root

ai-workspace-standards is not designed to be placed inside a project like a typical library — it's designed to be cloned as the root of the workspace itself. For example, if you clone this repository into a location like C:\git on Windows or ~/git on macOS/Linux, every project you create beneath it automatically inherits the shared rules from CONSTITUTION.md.

What happens if you ignore this rule and drop it into a specific project folder as a submodule (e.g., my-project/ai-workspace-standards/)? Scripts like new-project.ts check for themselves whether they were "run from the workspace root," and since this condition is broken, scaffolding is either rejected outright, or path calculations go wrong and the project gets created in the wrong location. Worse, only that one project would inherit the shared rules, while any other project you create later would share none of them at all — undermining the very reason this repository exists: for the entire workspace to use the same standards.

git clone https://github.com/5throck/ai-workspace-standards.git C:\git
cd C:\git

bun. Install the runtime

Every operational script in the workspace (scripts/*.ts) is written in TypeScript and run with bun. There are no separate .sh/.ps1 versions, so you need to install bun before you can run any scripts.

bun --version
git --version

💡 Key takeaway: once both commands print a version number successfully, your local environment is ready.

Session Start Checklist

The session start checklist defined in CONSTITUTION.md follows this order.

  1. Set the git hooks path: git config core.hooksPath .githooks
  2. Read CONSTITUTION.md: confirm the workspace's shared standards
  3. Read the project's docs/context.md: confirm the project's single source of truth
  4. Read AGENTS.md: confirm the agent roster for the project
  5. Check memory/MEMORY.md: confirm context left over from previous sessions
  6. Load the skill(s) specified by docs/context.md

2. Project Scaffolding Commands

bun scripts/list-template-versions.ts. Check available variants

Before creating a new project, first check which variants exist in the workspace and which versions each is tagged with.

bun scripts/list-template-versions.ts

bun scripts/new-project.ts. Create a project from an existing variant

The most frequently used command. Pick a variant and run it together with a project name, and the entire composition under templates/<variant>/ (agent definitions, skills, commands, governance pipeline) is cloned into a new independent repository.

bun scripts/new-project.ts "my-project-name" --variant co-develop

If you want to pin a specific version when scaffolding, also specify --version.

bun scripts/new-project.ts "my-project-name" --variant co-develop --version 1.4.0
OptionDescriptionNotes
--variant Specifies which project template to use (co-develop, co-design, co-work, etc.) If omitted, either an interactive selection or a default is used (depends on the script's behavior)
--version Pins a specific template-vX.Y.Z tag for the given variant If omitted, the latest version is used

Scaffolding Independent git repository Inherits shared standards

A scaffolded project is created as an independent git repository separate from the workspace root. Be careful not to mix edits to workspace-root files and new-project files within the same task/commit.

If you want to create a brand-new variant

If none of the existing variants fit and you need a completely new domain team composition, you can first create an independent Phase A prototype under Projects/<variant-name>/ using the create-variant skill together with the bun scripts/create-l2-scaffold.ts <variant-name> --domain <type> command. This prototype can be freely developed until it is promoted to an official templates/ variant. The detailed steps of this flow and the promotion procedure are covered in Chapter 11 (New Creation D-1) and Chapter 12 (Promotion P-1).
