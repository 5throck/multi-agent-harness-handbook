● Beginner Getting Started See with your own eyes exactly how a fully promoted template differs structurally from a prototype that hasn't yet been promoted
G-1 Beginner Comparing Template Structure vs. Prototype Structure
In Chapter 6 §2, you scaffolded a project from templates/co-work/ and also created a Phase A prototype at Projects/co-marketing/. Now you want to place the two directories side by side and check exactly which files exist in the official template but not in the prototype. This assumes you're continuing inside the practice workspace (ai-workspace-standards-demo).
Elements Used
variant.json README_ko.md docs/adr/
Step-by-Step
  1. Compare the two directory listings side by side

    Compare the top-level file listings of co-work, an official variant, and co-marketing, which is still in the prototype stage.

    diff <(ls templates/co-work/) <(ls Projects/co-marketing/)
  2. Check which items exist only in the official template

    Typically, variant.json, README.md/README_ko.md, and docs/adr/ exist in the official template but are still missing or empty at the Phase A prototype stage. First confirm these files actually exist on the official template (co-work) side.

    ls templates/co-work/variant.json templates/co-work/README_ko.md templates/co-work/docs/adr/

    Next, check the co-marketing prototype's folder listing to confirm the above files don't yet exist there.

    ls Projects/co-marketing/
variant.json holds the metadata (version, skill_manifest, script_manifest) required for a variant to be registered as an official scaffolding target. It's normal for this file to be missing, or to exist only minimally, during the Phase A prototype stage — the difference itself is a signal that "this hasn't been promoted yet."
Doing this comparison ahead of time lets you immediately see what's missing when you fill in the PROMOTION_CHECKLIST.md in the next scenario, D-1.
● Intermediate Daily Workflow Fill out the checklist and manifests needed to make co-marketing eligible for promotion
D-1 Intermediate Filling Out the Promotion Checklist
This is the situation of a practitioner trying to actually close the gaps identified in G-1. To make the co-marketing prototype eligible for official promotion, you need to satisfy the conditions in PROMOTION_CHECKLIST.md and fill in skill_manifest/script_manifest in variant.json.
Elements Used
PROMOTION_CHECKLIST.md skill_manifest script_manifest
Step-by-Step
  1. Review the PROMOTION_CHECKLIST.md items

    The checklist generally includes items like "required verification scripts pass," "platform parity secured," and "agent frontmatter is complete." For co-marketing, start by re-confirming whether researcher.md and writer.md, created in Chapter 6 §2, have all required fields. The two scripts check different things, so run each separately and check the results independently.

    ① Agent frontmatter completeness check — confirm that required fields such as name, tier, and handoff are all filled in.

    bun run agent:verify

    ② Skill manifest check — confirm that the skills referenced by the variant actually exist and are evenly deployed across all three platforms (.claude/.gemini/.agents).

    bun scripts/validate-skills.ts
  2. Define the manifests in variant.json

    Register the skills co-marketing uses (e.g., campaign-research, copywriting) in skill_manifest.variant_specific, and any variant-specific scripts in script_manifest.

    {
  "name": "co-marketing",
  "inherits_common": "1.0.0",
  "skill_manifest": {
    "variant_specific": ["campaign-research", "copywriting"]
  },
  "script_manifest": []
}
skill_manifest.variant_specific is where this variant declares any skills it needs beyond the common set. Any skill not registered here is at risk of being dropped during the reconcile step of the promotion pipeline.
The checklist items must be filled in against the actual PROMOTION_CHECKLIST.md template in the repository — the example here shows the general flow, not the complete list of items.
● Advanced Power User The actual promotion pipeline command and post-promotion verification procedure
P-1 Advanced Running and Verifying the Promotion Pipeline
An experienced practitioner who has finished D-1 and judged the prototype ready for promotion now actually runs l2-to-variant-pipeline.ts to convert co-marketing into an official templates/co-marketing/. From this point on, this is a hard-to-reverse operation that leaves traces in the actual git history.
This scenario is an operation that produces a substantive change on a real git-tracked workspace. Only run it if you genuinely intend to publish the variant; if you only want to observe the flow, do not execute the commands — just read them.
Elements Used
l2-to-variant-pipeline.ts validate-templates.ts
Step-by-Step
  1. Run the promotion pipeline

    --source is the Phase A prototype path, --variant is the final variant name, and --variantType is the domain type.

    bun scripts/l2-to-variant-pipeline.ts --source Projects/co-marketing --variant co-marketing --variantType marketing
  2. Manually confirm platform parity

    Use diff to confirm that the commands/skills directory listings in .claude/ and .gemini/ correspond to each other.

    diff <(ls templates/co-marketing/.claude/commands/) <(ls templates/co-marketing/.gemini/commands/)
diff <(ls templates/co-marketing/.claude/skills/) <(ls templates/co-marketing/.gemini/skills/)
  3. Final gate — validate-templates.ts

    After all steps are complete, run the structural validation one more time to confirm the new variant satisfies the workspace-wide rules.

    bun scripts/validate-templates.ts
The pipeline's reconcile step removes duplication by deleting files in Projects/co-marketing/ that are identical to common files (L0/common). However, skill directories are intentionally excluded from this reconcile step, so after promotion you need to separately confirm the skills still remain in .claude/skills/ and .gemini/skills/.
If _ORIGIN.md has a "Manual Phase B Steps" section, it lists the domain-specific directories the pipeline couldn't move automatically — do a final check to make sure you've manually moved everything on that list. For example, co-marketing's _ORIGIN.md might read like this. 
## Manual Phase B Steps
- [ ] Copy the brand-assets/ directory to templates/co-marketing/brand-assets/
- [ ] Carry over the campaign brief skeleton files under campaign-templates/ as-is
- [ ] Reset the initial channel list in channel-configs.json to an empty array
These items are a different kind of concern than the "de-duplicate common files" work reconcile handles — they're variant-specific assets, so the pipeline won't process them for you. You have to check each checkbox off and move them yourself.
See also: the concept of designing a new variant from scratch is covered in Chapter 11 · The From-Scratch Path, and the hands-on exercise for creating a Phase A prototype is covered in Chapter 6 §2 Exercise.
