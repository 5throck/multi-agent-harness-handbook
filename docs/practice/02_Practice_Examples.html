2.1 · 멀티 에이전트 팀 실습

준비: 필요한 프로그램 설치

아래 실습은 Claude App(Desktop), Claude Code(CLI), Antigravity(Desktop), Antigravity CLI(agy)를 오가며 진행된다. 로컬 환경에 이 도구들이 아직 없다면, 이 핸드북에 함께 담아 둔 환경 설정 자동화 도구인 setup이 이 장의 실습에 정확히 필요한 구성을 자동으로 설치해준다. 원하는 파일을 내려받아 로컬에서 실행하면 된다.

📦 설치 전 확인 사항

스크립트를 실행하기 전에 SETUP_CHECKLIST_ko.md를 먼저 읽어본다. 이 문서에는 다음 전제 조건이 정리되어 있다.

  • 활성 상태의 Claude Pro/Max 구독 그리고 Antigravity 사용을 위한 Google 계정 (개별 API 키가 아니라 구독제/계정 기반 플랜이어야 한다)
  • gh auth login을 진행할 수 있는 GitHub 계정
  • 최소 5GB의 여유 디스크 공간
  • 관리자(admin) 또는 sudo 권한

Windows에서 설치하기

이 핸드북의 경로 표기가 Windows 기준이므로, Windows 사용자는 setup-windows.ps1을 내려받아 실습 전용 폴더(예: ~/harness-lab/setup/)에 두고, PowerShell을 관리자 권한으로 실행한 뒤 아래 명령을 돌린다.

cd ./setup
# PowerShell을 관리자 권한으로 실행한 뒤:
.\setup-windows.ps1
# 선택 옵션: -WSL2, -WezTerm, -Docker, -Force (이미 설치된 도구도 재설치)

스크립트가 끝나면 터미널을 완전히 닫았다가 다시 열어야 PATH 변경 사항이 적용된다.

macOS / Linux에서 설치하기

macOS는 setup-mac.sh, Linux는 setup-linux.sh를 내려받아 사용한다. 둘 다 --wezterm, --docker 선택 플래그를 지원한다.

# macOS
cd ./setup && bash setup-mac.sh

# Linux
cd ./setup && bash setup-linux.sh

설치 검증하기

운영체제와 무관하게 setup-common.ts를 bun으로 실행하면 설치 상태를 점검하는 체크리스트 표가 출력된다. 모든 항목이 ✅로 표시되면 아래 실습을 진행할 준비가 된 것이다.

cd ./setup && bun setup-common.ts

이 프로젝트가 설치하는 도구 중 이번 장의 4도구 실습과 직접 관련된 것은 다음과 같다. Claude App(Desktop), claude(Claude Code CLI), Antigravity(Desktop), agy(Antigravity CLI), 그리고 .claude/agents/*.md 기반 서브에이전트 실행에 필요한 bun, 이후 장에서 쓰이는 gh(GitHub CLI)까지 함께 설치된다.

Antigravity CLI(agy)는 별도 런타임 설치가 필요 없는 단일 컴파일 바이너리다 — 압축 해제 후 PATH에 넣기만 하면 바로 실행된다. 옛 자료에 남아 있는 "Gemini CLI를 npm으로 전역 설치" 같은 안내는 더 이상 유효하지 않다. 최신 설치 방법은 antigravity.google/docs/cli-getting-started에서 확인한다.
설치 중 문제가 생기면 Windows 기준 %USERPROFILE%\workshop-setup-logs\ 폴더에 로그가 남는다. 문제를 해결할 때 이 로그부터 확인한다.
● 초급 Getting Started 커스텀 서브에이전트를 하나 만들고 Claude App에서 호출을 관찰한 뒤, Antigravity(Desktop)에서 목표를 주고 동적 생성을 관찰하는 첫 단계
G-1 초급 첫 커스텀 서브에이전트 만들고 호출하기
하네스를 처음 써보는 개발자가 실습 전용 빈 폴더 ~/harness-lab/를 새로 만들고, 그 안에서 텍스트 파일을 검토해주는 reviewer 서브에이전트를 Claude App(Desktop)에서 정의·호출해본 뒤, Antigravity(Desktop)에서는 파일을 미리 정의하는 대신 목표를 주고 오케스트레이터가 알아서 처리하는 과정을 관찰한다. 이 폴더는 실습 전용이므로 다른 프로젝트 파일과 섞이지 않는다.
사용 요소
서브에이전트 정의하기 서브에이전트 호출하기
단계별 실행
  1. 실습 폴더 생성 및 진입

    홈 디렉터리 아래 harness-lab 폴더를 새로 만들고 그 안으로 이동한다. 이후 모든 명령은 이 폴더 안에서 실행한다.

    mkdir ~/harness-lab
cd ~/harness-lab
Claude App (Desktop)

.claude/agents 디렉터리를 만들고 그 안에 reviewer.md 파일을 저장한다. YAML frontmatter의 name이 나중에 subagent_type으로 참조하는 식별자가 된다.

mkdir -p .claude/agents
cat > .claude/agents/reviewer.md << 'EOF'
---
name: reviewer
description: 텍스트 파일의 오탈자, 논리적 허점, 개선점을 검토한다.
tools: Read, Grep
model: sonnet
---

너는 리뷰어 역할이다. 주어진 텍스트 파일을 읽고 오탈자,
논리적 허점, 개선하면 좋을 부분을 목록으로 정리해서 보고한다.
EOF

검토 대상 파일을 만들고 Claude App을 실행한 뒤, 채팅창에 자연어로 검토를 요청한다.

echo "이 프로젝트는 매우 좋다. 이것은 아주 좋다." > notes.txt
# Claude App을 열고 채팅창에 입력:
# "notes.txt를 reviewer 서브에이전트로 검토해줘"
Claude App은 요청을 해석해 Agent(Task) 툴을 description="notes.txt 검토", subagent_type="reviewer"로 호출한다. 화면에는 서브에이전트가 스폰되는 과정과, 작업을 마친 뒤 메인 세션으로 돌아오는 요약 결과가 순서대로 표시된다.

Claude Code(CLI)에서도 동일하게 — 같은 .claude/agents/reviewer.md 파일을 그대로 인식한다. 터미널에서 claude를 실행하고 같은 문장을 입력하면 똑같이 동작한다. CLI에는 추가로 PostToolUse 등 자동화 훅이 발화한다는 차이가 있다(App에서는 훅이 발화하지 않는다).

Antigravity (Desktop)

Antigravity는 .claude/agents 같은 사전 정의 파일 개념이 없다. Agent Manager에서 ~/harness-lab/을 가리키는 Workspace를 하나 만들고, 파일을 미리 준비하는 대신 목표를 자연어로 곧바로 준다.

echo "이 프로젝트는 매우 좋다. 이것은 아주 좋다." > notes.txt
# Antigravity Agent Manager에서 ~/harness-lab/ Workspace를 만들고
# 에이전트에게 지시:
# "notes.txt 파일을 검토하고 개선점을 요약해줘"
Workspace를 맡은 에이전트가 목표를 받아 스스로 필요한 작업(파일 읽기, 개선점 정리)을 판단해 수행한다. "reviewer라는 이름의 서브에이전트를 호출했다"기보다는, 에이전트가 그 순간 검토에 필요한 역할을 즉석에서 구성해 처리한 것에 가깝다.

Antigravity CLI(agy)에서도 같은 개념으로 실습할 수 있다 — 사전 정의 파일 없이 /goal로 목표만 준다.

cd ~/harness-lab && agy
/goal notes.txt 파일을 검토하고 개선점을 요약해줘
Claude App/Code에서는 tools 필드를 Read, Grep처럼 필요한 최소 권한만 나열하면, 리뷰어가 실수로 파일을 수정하는 사고를 방지할 수 있다. Antigravity는 정의 파일이 없으므로 이런 사전 권한 제한 대신, 목표 프롬프트 자체를 "읽고 요약만 해줘"처럼 명확히 좁히는 것이 안전하다.
● 중급 Daily Workflow 메인 세션이 오케스트레이터가 되어 writer와 reviewer 두 전문 에이전트를 순서대로 지휘하는 실무 패턴
D-1 중급 writer → reviewer 2단계 파이프라인
G-1을 마친 실무자가 같은 ~/harness-lab/ 폴더에서, Claude App에는 두 번째 서브에이전트 writer를 추가하고, Antigravity(Desktop)에는 같은 2단계 목표를 자연어로 준다. writer가 먼저 초안을 쓰고 reviewer가 그 결과를 비평하는 파이프라인을 네 도구 모두에서 순서를 지켜 지휘하게 한다.
사용 요소
서브에이전트 정의하기 서브에이전트 호출하기
Claude App (Desktop) — 메인 세션이 Agent 호출을 순차 디스패치

G-1의 reviewer.md와 같은 폴더에 writer.md를 추가한다. 이 에이전트는 파일을 쓸 수 있어야 하므로 tools에 Write를 포함한다.

cat > .claude/agents/writer.md << 'EOF'
---
name: writer
description: 주어진 주제로 짧은 초안 텍스트 파일을 작성한다.
tools: Write, Read
model: sonnet
---

너는 작가 역할이다. 요청받은 주제로 3~5문장 분량의
간결한 초안을 작성해 지정된 파일에 저장한다.
EOF

Claude App 채팅창에서 writer와 reviewer를 순서대로 쓰라고 자연어로 명시적으로 요청한다. 메인 세션이 두 Agent 호출을 순차적으로(writer 완료 후 reviewer 시작) 디스패치하도록 지시하는 것이 핵심이다.

# 채팅창에 입력:
# "writer 서브에이전트로 draft.txt에 '원격 근무의 장점'
#  주제 초안을 쓰게 하고, 완료되면 reviewer 서브에이전트로
#  draft.txt를 검토해줘. 두 단계를 순서대로 진행해."
메인 세션은 먼저 subagent_type="writer"로 Agent를 호출해 draft.txt를 생성시키고, 그 결과가 돌아온 뒤에야 subagent_type="reviewer" 호출을 보낸다. 두 전문가 사이의 의존 관계를 조율하는 것은 서브에이전트가 아니라 메인 세션의 몫이다.
writer와 reviewer를 병렬(한 메시지에 동시)로 호출하면 reviewer가 아직 존재하지 않는 draft.txt를 검토하려 시도해 실패할 수 있다 — 의존 관계가 있는 작업은 반드시 순차 호출로 요청해야 한다.

Claude Code(CLI)에서도 동일하게 — 같은 writer.md/reviewer.md를 그대로 쓰고, 터미널에서 같은 순차 지시 문장을 입력하면 동일하게 순차 디스패치된다.

Antigravity (Desktop) — 한 목표로 오케스트레이터가 파이프라인을 스스로 분해

Antigravity는 사전에 writer/reviewer 파일을 나눠 정의하지 않는다. Agent Manager에서 같은 Workspace의 에이전트에게 두 단계를 모두 포함한 목표 하나를 자연어로 준다.

# Antigravity Agent Manager에서 에이전트에게 지시:
# "draft.txt에 '원격 근무의 장점' 주제로 3~5문장 초안을
#  먼저 작성하고, 작성이 끝나면 그 초안을 검토해서
#  개선점을 요약해줘. 두 단계를 순서대로 진행해."
오케스트레이터가 목표를 "초안 작성 → 검토"의 두 하위 작업으로 스스로 쪼개고, 초안 작성이 끝난 뒤에야 검토 작업을 시작한다. 순서 조율의 책임이 사용자가 아니라 오케스트레이터 내부에 있다는 점이 Claude 계열과 다르다.

Antigravity CLI(agy)에서도 같은 목표를 한 번에 줄 수 있다. /goal로 전체 파이프라인을 한 문장에 담으면 오케스트레이터가 알아서 분해하고, 필요하면 /agent로 검토 단계를 백그라운드에 맡긴 채 다른 작업을 이어갈 수도 있다.

agy
/goal draft.txt에 '원격 근무의 장점' 주제로 초안을 먼저 쓰고,
완료되면 그 초안을 검토해서 개선점을 요약해줘
● 고급 Power User writer와 reviewer를 독립적인 두 파일에 대해 동시에 돌리는 병렬 운영 — Agent Manager와 Agent Teams 비교
P-1 고급 독립 작업 병렬 실행 — Agent Manager vs Agent Teams
D-1까지 마친 숙련자가 서로 의존하지 않는 두 개의 독립적인 작업(topic-a.txt, topic-b.txt 초안 작성)을 Antigravity(Desktop) Agent Manager와 Claude Code Agent Teams 양쪽에서 각각 병렬로 돌려본다.
사용 요소
병렬 실행 — 팀 단위로 확장하기 병렬 실행
Antigravity (Desktop) — Agent Manager, Workspace별 에이전트 1개

Antigravity를 열고 Agent Manager에서 ~/harness-lab/을 가리키는 Project를 하나 준비한다. 이번에는 하나의 Workspace에 두 에이전트를 몰아넣지 않고, 서로 다른 Workspace 두 개를 만들어 각각 에이전트를 하나씩 배정한다.

  1. Workspace A 생성 — topic-a.txt 작성

    New Worktree Mode로 Workspace를 하나 만들고, "topic-a.txt에 '재택근무' 주제로 3~5문장 초안을 작성해줘"라고 에이전트에게 지시한다.

  2. Workspace B 생성 — topic-b.txt 작성

    별도의 Workspace를 하나 더 만들고(New Worktree Mode 권장), "topic-b.txt에 '4일 근무제' 주제로 3~5문장 초안을 작성해줘"라고 지시한다. Workspace A의 에이전트와는 완전히 분리된 인스턴스다.

  3. Inbox에서 두 에이전트 진행 상황 확인

    두 에이전트가 각자의 Workspace에서 동시에 작업하는 동안, Inbox에서 승인이나 피드백이 필요한 메시지가 오는지 확인하고 필요하면 응답한다.

두 에이전트를 같은 Workspace에 함께 배정하지 않는다 — 같은 폴더를 공유하면 한 에이전트가 다른 에이전트의 작업 맥락을 자기 것으로 착각하는 "인지적 중첩(cognitive overlap)"이 발생할 수 있다. Workspace 또는 worktree로 반드시 분리하는 것이 Antigravity의 권장 관행이다.

Antigravity CLI(agy)에서도 데스크톱의 Agent Manager 대신 터미널을 여러 개 띄워 같은 효과를 낼 수 있다 — 한 터미널에서 agy로 topic-a.txt 작업을, 다른 터미널에서 별도 agy 세션으로 topic-b.txt 작업을 동시에 진행한다(세션 간 병렬). 서로 다른 파일을 건드리는 한 두 세션은 충돌하지 않는다.

Claude App (Desktop) — Agent Teams, teammateMode: in-process

같은 ~/harness-lab/ 폴더에서 실험적 기능인 Agent Teams를 활성화한다. .claude/settings.json이 없다면 새로 만들고, 있다면 아래 두 키를 병합한다. Desktop App에서는 teammateMode가 반드시 in-process여야 한다(tmux는 지원되지 않는다).

cat > .claude/settings.json << 'EOF'
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "in-process"
}
EOF

설정을 적용하려면 Claude App을 재시작한다. writer 두 명을 서로 다른 파일에 대해 동시에 스폰하도록 요청한다.

# Claude App 채팅창에 입력:
# "writer 팀메이트 두 명을 병렬로 스폰해서 하나는
#  topic-a.txt에 '재택근무', 다른 하나는 topic-b.txt에
#  '4일 근무제' 초안을 동시에 쓰게 해줘. 각 파일이 완료되면
#  reviewer 팀메이트가 검토하게 하고, 공유 작업 목록도 보여줘."
Agent Teams가 켜진 상태에서는 여러 Claude 인스턴스가 팀메이트로 병렬 실행되며, 공유 작업 목록에 각 팀메이트의 할 일이 등록되고 상태가 갱신된다. 두 writer 팀메이트는 topic-a.txt/topic-b.txt 사이에 의존성이 없으므로 동시에 진행되고, 팀메이트 간에는 SendMessage로 직접 메시지를 주고받을 수도 있다. 다만 Desktop App에서는 일부 키보드 내비게이션(Shift+Down 등)이 제한된다.
작업 목록이 서로 얽히지 않도록, 병렬로 스폰하는 팀메이트에게는 처음부터 완전히 분리된 파일이나 디렉터리를 배정하는 것이 안전하다.

Claude Code(CLI)에서도 동일하게 — 같은 설정과 같은 요청 문장이 그대로 동작한다. CLI에서만 teammateMode: "tmux"를 추가로 선택할 수 있어, tmux 분할 창으로 각 팀메이트의 화면을 나란히 볼 수 있다는 점이 Desktop App과 다르다.
