Chapter 3

Guardrails and Permission Models

You'll get hands-on practice creating and invoking subagents in Chapter 4. But before that, if you don't first decide "what this agent is allowed to do and what it isn't," you'll run into trouble both during practice and in production. This chapter covers how to actually design the "guardrails" briefly mentioned in Chapter 2.

What This Chapter Covers
  • Why guardrails matter (the more autonomy you grant, the larger the blast radius of a mistake)
  • How to classify risk into three tiers: read, write, and destructive operations
  • How to grant subagents only minimal permissions using the tools field
  • Designing the points (gates) where an orchestrator must always wait for human approval
  • Why worktrees and sandboxes physically isolate parallel work
  • The minimal logging habits needed to trace "what happened" after an incident

Why Guardrails Are Needed

In Chapter 2 we compared a harness to "a harness that keeps a horse from running off in any direction it pleases." Guardrails are the part of that harness that specifically marks "you must never go this way." If a subagent only reads files and summarizes them, a mistake is easy to undo. But once a subagent can delete files, or automatically commit and deploy, a single mistake can lead to consequences that are hard to reverse.

The problem is that "capability" and "risk" grow in the same direction. The more tools and broader permissions you give an agent, the more it can handle — but the larger the blast radius when it makes a wrong call. Guardrails separate these two. They keep "what the agent can do" broad, while narrowing "what it may execute without human confirmation."

The purpose of guardrails isn't distrust of the agent — it's cutting the blast radius of a mistake down to a size a human can handle, in advance.

Classifying Risky Operations into Three Tiers

Controlling every operation with the same intensity would make the system too slow to be practical. In reality, it's more realistic to sort operations into three tiers based on how reversible they are, and handle each differently.

TierExamplesDefault Policy
ReadReading files, searching code, browsing the webGenerally allowed freely (nothing needs to be undone)
WriteCreating/modifying files, local commitsAllowed, but review the results (within the range git can undo)
Destructive/ExternalDeleting files or branches, force pushes, deployments, payments, sending external messagesNever execute without explicit human approval

This three-tier classification matches the actual rules of the workspace documentation this handbook is based on. The "Sensitive File Guard" from the /sync pipeline covered in Chapter 5 — which blocks sensitive files like .pem, .key, and .env right before a commit — and the rule explicitly forbidding force pushes and --no-verify, are both guardrails belonging to the destructive-operation tier.

Principle of Least Privilege — the tools Field

Recall the frontmatter of the reviewer subagent we created in the Chapter 4 §2 exercise. We explicitly listed the tools to allow, like tools: Read, Grep. This is the principle of least privilege applied at the subagent level. Since reviewer never has the Write tool to begin with, no matter how ambiguously the prompt is interpreted, there's simply no way for it to modify a file.

Looking at a real incident scenario makes clear why this matters. If reviewer had been habitually granted Write permission as well, then a slightly ambiguous request like "review this file and improve it" might be interpreted by reviewer as "directly edit," overwriting the original. Narrowing permissions in advance via the tools field prevents this kind of interpretation gap from ever becoming an incident. It blocks the problem structurally, instead of relying on prompt quality.

Permissions should be set based on "what the minimum is that this role needs to function normally," not "what this role could possibly do at maximum." Widening permissions later when needed is easy, but narrowing permissions that were opened too broadly from the start, after an incident has already occurred, is too late.

Human Approval Gates

In Chapter 2 we defined the orchestrator as "the role that halts progress at points (gates) requiring confirmation from the user." The most important question in guardrail design is "exactly where should that gate be placed?" Stop too often, and delegating work to an agent becomes meaningless; stop too rarely, and even destructive operations can slip through silently.

The rule proven in practice is simple. Always stop right before an operation belonging to the "destructive/external impact" tier from §2; post-hoc review is sufficient for tiers below that. For example, the /sync pipeline proceeds automatically through commit, push, and PR creation, but is designed to require separate human approval for actually merging the PR or deploying to production. Explicitly drawing the line — "up to here we trust it to proceed, and from here a human must always look" — is the core of gate design.

Isolation — Worktrees and Sandboxes

In the advanced scenario (P-1) in Chapter 4 §2, we warned that "assigning multiple agents to the same workspace can create cognitive overlap." This is also a form of guardrail. If multiple agents touch the same files and the same git history at once, one agent's mistake can contaminate another agent's working context.

Running each agent in a worktree (an isolated git working directory) or a separate sandbox environment means that even if one agent makes a wrong judgment, its impact doesn't leak outside its own workspace. When running multiple agents in parallel, you must always first decide: "should they share the same space, or be physically separated?"

Audit Logs and Observability

If guardrails are prevention, audit logs are after-the-fact tracing. No guardrail is perfect, so when an incident occurs, you absolutely need a minimal record that lets you retrace "when, which agent, with which tool, did what." If memory/MEMORY.md covered in Chapter 5 serves to record context across sessions, an audit script like audit.ts, run via a PostToolUse hook in CI, is an observability device that mechanically checks every time whether "a change violated the rules."

In the Claude App (Desktop), covered in Chapter 4, hooks like this do not fire. For teams working mainly in the Desktop app, a situation can actually arise where "we thought the audit log was running, but in fact none of it was running at all." When designing guardrails, you must always verify "does this policy actually fire on the tool I'm using?"
Audit Log Structure and Audit Flow
In one sentence: guardrails are the combination of four habits — granting differentiated permissions based on risk level (§2·§3), always requiring human confirmation before an irreversible point (§4), physically isolating parallel work (§5), and keeping records in case everything else fails (§6). Starting in Chapter 4, we apply these principles directly within hands-on practice code.
