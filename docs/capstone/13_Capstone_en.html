Chapter 13 · Capstone

Capstone Exercise — Designing Your Own Workflow

This final exercise brings together everything you've learned so far — the concepts (Chapters 2 and 3), the tools (Chapter 4), the operational sense for running a team (Chapter 7), and the workspace structure (Chapters 5 and 8) — into a project you design and run end to end yourself, even at a tiny scale. There is no single correct answer. This chapter guides you not on "what to build," but on "in what order to make decisions so you don't miss anything."

What You'll Do in This Chapter
  • Pick one task you actually repeat and reframe it as a mini multi-agent-team problem
  • Decide the guardrails (permissions and approval gates) before building any subagents
  • Implement the orchestrator, specialists, and handoff contract as real files
  • Verify the results and reflect on what differed from the plan

Goal of This Exercise

In Chapter 4 §2 you followed a pre-defined scenario (reviewer, writer), and in Chapter 11 you built a variant for a pre-defined domain (co-marketing). This time, you choose both the scenario and the domain yourself. It's fine to keep the scale small — a single five-minute chore you actually repeat every week is enough. What matters is becoming able to explain, in your own words, why you made each decision in this particular order.

Step 1 — Define the Problem and Design the Team

Question 1. What are you repeating?

Think of a task you've been doing repeatedly lately. For example: "every week I read meeting notes and pull out just the action items," "I draft a new blog post and check the tone," or "I write a PR description and make sure the checklist is complete."

Question 2. If you handed this task to a single solo agent, which of the three problems from Chapter 1 §2 (limits of a single agent) would show up?

Is it context overload (multiple stages getting mixed up in one session)? Role conflict (the writer having to review its own output)? Or lack of parallelism (several independent subtasks)? Your answer here determines the team structure from Chapter 1 §3.

Question 3. How many orchestrators and specialists will you split this into?

Sketch out a minimal setup, referring to Chapter 1 §3–§4b. It's fine to have just one specialist — an orchestrator (the main session) plus a single specialist (subagent) already counts as a team. Write down each specialist's input and output in one sentence ("this agent takes X and produces Y").

Step 2 — Set Guardrails First

Following the principle from Chapter 3, decide permissions before actually building the subagent. Fill in the table below.

What is the minimum this specialist needs to be able to do? (e.g., read files only / read and write files / call an external API)
Is there any action that must never run without human confirmation? (Corresponds to the "destructive / externally impactful" tier from Chapter 3 — e.g., actually sending a message, actually committing.)
On failure, should it retry, roll back, or escalate to a human? (Refer to the retry/rollback/escalation criteria from Chapter 2 §4.)

The answers to these three questions become the tools field in the subagent's frontmatter, and the "stop right here" wording you'll include when instructing the orchestrator.

Create a subagent named [role name] at .claude/agents/[role name].md.

If you have two or more specialists, create each one the same way as in the D-1 scenario, then explicitly instruct the orchestration mode at the end — e.g., "run these two stages in sequence" or "run them concurrently." Confirm for yourself which one you chose, and, as you learned in Chapter 1 §4b, that this choice depends on the dependency relationships in the request.

Step 4 — Verify and Reflect

After actually running it once, answer the following questions.

  • Did the result come out in the expected format (the handoff contract)? If not, was the subagent's description perhaps too vague?
  • Were the guardrails you set in Step 2 actually respected? Did any place end up with a tools scope broader than necessary?
  • If this workflow needs to be repeated regularly, is it worth promoting later into a variant or a reusable skill, as covered in Chapter 11?
Once you complete this reflection, you will have gone through, on your own, the full cycle this handbook has emphasized from the start (concepts (Ch. 2) → safeguards (Ch. 3) → tools (Ch. 4) → organization/structure (Ch. 5, 7, 8)). The next time you encounter a similar repetitive task, simply asking these same questions again, in the same order, will let you design a new workflow.

Completion Checklist

Self-Check
☐ Defined one repetitive task as a problem
☐ Can explain which limitation of a single agent makes a team necessary
☐ Decided permissions and approval gates before building any subagent
☐ Actually built a subagent and ran it at least once
☐ Verified the results and wrote down what differed from expectations
Reproducibility Check
☐ Ran the same request 2+ times and confirmed the results were consistent
☐ Confirmed the guardrails activate correctly when given abnormal input (empty file, wrong format, etc.)
☐ Compared before/after snapshots to confirm there were no unexpected file changes

Next Steps After Completion

This handbook ends here, but real-world application starts now. Once you apply the workflow you designed today to actual work, it will naturally lead in one of the following three directions.

Within This Week
☐ Run the subagent you built today on the actual recurring task once more, and this time share the results with a colleague
☐ Ask one person whether this workflow could be useful for another team in your organization
Once This Workflow Is Established
☐ Consider following the promotion process covered in Chapters 10 and 12 to turn it into a formal variant — so your whole team can reuse it
☐ Learning the L2 project upgrade process from Chapter 8 §5 means you won't have to manually apply every future change to the shared standards
If You Get Stuck
☐ Look up the concept again in the Glossary
☐ Check the FAQ for anyone who's hit a similar issue
What matters most isn't "memorizing the right answer," but being able to walk through the sequence this handbook has followed — define the problem → check the limitations → design permissions → implement → verify — on your own, every time you face a new repetitive task.
