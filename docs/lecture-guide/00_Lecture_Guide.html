강의 진행 가이드

이 핸드북 전체(1~7장)를 사용해 강의를 진행할 강사를 위한 자료입니다. 시간은 3시간을 넘어도 무방합니다 — 아래 배분표는 각 절을 충분히 다뤘을 때의 현실적인 소요 시간이며, 억지로 압축하지 않았습니다.

이 문서는 핸드북 본문이 아니라 강사 전용 운영 자료입니다. 학습자에게는 공유하지 않아도 됩니다.

사전 준비물 통합 체크리스트

각 장의 setup 문서(SETUP_CHECKLIST_ko.md)에 흩어져 있던 항목을 강의 시작 전 한 번에 확인할 수 있도록 모았습니다. 강의 시작 최소 하루 전에 참가자에게 이 목록을 공지해 현장에서 설치 문제로 시간을 낭비하지 않도록 합니다.

모든 절을 꼼꼼히 다루고, 실습은 참가자가 직접 타이핑(또는 복붙)할 시간을 포함해 산정했습니다. 휴식 시간은 별도입니다.

장별 강사 노트

1장 — 개념

이 장은 이후 모든 장에서 쓰는 용어(오케스트레이터, 스페셜리스트, 핸드오프, 서브에이전트)의 기준점이므로 속도를 늦춰도 좋습니다. 비유(요리, 신입사원, 택배 송장)를 화이트보드에 옮겨 그리면서 설명하면 이해도가 확 올라갑니다.

참가자 활동 §4의 세 패턴(팬아웃/파이프라인/검증-후-적용) 중 하나를 골라 "우리 팀 업무에 적용한다면?"을 1분씩 옆 사람과 이야기하게 하는 것을 권장합니다.

2.0 / 2.1 — 실습

참가자를 4개 그룹(Claude App / Claude Code / Antigravity Desktop / Antigravity CLI)으로 나눠 각자 다른 도구로 같은 시나리오를 진행하게 하면, 마지막에 결과를 비교하며 3장의 "도구 비교" 내용이 훨씬 잘 와닿습니다.

참가자 활동 G-1, D-1은 전원 실습. P-1(병렬 실행)은 시간이 부족하면 강사 화면 시연으로 대체 가능 — Agent Teams 설정(`.claude/settings.json`)은 그룹당 한 명만 해봐도 개념은 전달됩니다.

3장 — 도구 비교

2.1 실습 직후에 배치돼 있으므로, "방금 여러분이 쓴 도구에서 이게 왜 이렇게 동작했는지" 형태로 복기하면서 진행하면 순수 강의보다 훨씬 몰입도가 높습니다.

4장 — 두 모델 비교 + 칸반

칸반 절은 실제 사내 헬프데스크/지원팀 운영 경험이 있는 참가자가 있다면 그 경험을 먼저 물어보고 시작하는 것이 효과적입니다. "왜 필요한가"를 이론으로 설명하기 전에, "티켓이 밀렸던 경험 있나요?"로 시작하면 공감대가 빨리 형성됩니다.

토론 "우리 조직이라면 어느 모델이 맞을까?"를 5분 소그룹 토론 후 발표.

5장 — ai-workspace-standards 소개

분량이 많은 장입니다(L0→L1→L2, 4대 핵심 파일, 7개 variant, 모델 티어링). 전부 암기시킬 필요는 없고, "왜 이런 계층 구조가 필요한가"라는 동기만 확실히 전달되면 충분합니다 — 세부 스크립트 이름은 6.0/6.1 실습 중에 자연스럽게 다시 나옵니다.

6.0 / 6.1 — variant 실습

new-project.ts 실행 시 실제로 GitHub 클론이 오래 걸릴 수 있으므로, 이 실습을 시작하기 전에 미리 워크스페이스를 클론해 두라고 사전 공지하는 것이 중요합니다(위 사전 준비물 체크리스트 참고).

참가자 활동 G-1(co-consult 스캐폴딩)은 전원 실습. D-1(co-marketing 프로토타입 직접 만들기)은 시간이 빠듯하면 짝 활동(한 명이 타이핑, 한 명이 검증 명령 실행)으로 진행해도 좋습니다.

7.0 / 7.1 — variant 심화

강사 시연 전용 P-1(실제 l2-to-variant-pipeline.ts 실행)은 실제 git 이력에 영향을 주는 비가역적 작업입니다. 참가자 전원이 따라 하게 하지 말고, 강사가 화면 공유로 한 번 실행하는 것을 보여주는 것으로 충분합니다. 참가자가 직접 해보고 싶다면 강의 종료 후 개인 환경(워크숍 전용 클론 폴더)에서 해보도록 안내하세요. 참가자 활동 PROMOTION_CHECKLIST.md 항목 확인과 variant.json manifest 작성까지는 전원 실습 가능합니다.
