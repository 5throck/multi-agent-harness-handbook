# 교재 작성 가이드라인 (Authoring Guidelines)

> 본 문서는 **멀티 에이전트팀 하네스 엔지니어링 핸드북**을 만들며 검토자가 남긴 피드백을
> 원칙으로 정리한 것입니다. **같은 방식으로 다른 과정/교재를 만들 때 일종의 가이드라인**으로
> 삼기 위해 작성되었습니다. 각 원칙은 실제 발생했던 수정 요청에서 비롯되었으며, "왜"와
> "어떻게 적용할지"를 함께 적습니다.

---

## 0. 이 가이드를 쓰는 방법

- 새 챕터나 실습 시나리오를 쓰기 **전**에 §1~§3(대상·구조·접근법)을 먼저 점검한다.
- 강의 자료를 기획할 때는 **§14(강의 소개 필수 포함)**를 먼저 작성하고, 그 학습 목표를 기준으로 본문 구조를 잡는다.
- 초안을 쓴 **뒤**에는 §A의 체크리스트를 한 항목씩 돌려 빠진 게 없는지 확인한다.
- 기술적 정확성(§4), OS 처리(§9), 마크업 디테일(§11), **문체 일관성(§12)**은 리뷰에서 반복적으로
  지적되는 영역이므로 초안 단계부터 미리 적용한다.
- 탈고 직전에는 **§19(오탈자·문맥 점검)**로 전체 문장을 다시 읽는다.
- 강의 자료를 기획할 때는 **§20(강의 진행 가이드 구조)**를 참고하여 강사 운영 자료도 함께 준비한다.

## 1. 대상 독자와 설명의 깊이

**원칙**: "이 분야를 잘 모르는 사람"을 1차 독자로 삼고, 개념 설명은 얕지 않게 쓴다.

- **왜**: 검토 피드백 — *"전반적으로 개념에 대한 설명이 너무 약하다."*
- **어떻게 적용**:
  - 전문 용어를 처음 꺼낼 때 **비유(analogy)**를 한 줄 붙인다(예: 오케스트레이터 = 교향악단 지휘자).
  - "무엇"뿐 아니라 "왜 그렇게 해야 하는가"를 한 문장으로 덧붙인다.
  - 정의 → 비유 → 실습 3단 구조로 한 절을 구성하면 초보자가 따라오기 쉽다.

---

## 2. 실습 단계의 구체성

**원칙**: 실습 단계는 "복사 → 붙여넣기 → 바로 실행"이 되도록 세분한다.

- **왜**: 검토 피드백 — *"실습의 경우도 세부 단계를 매우 구체적으로 나눠서, 복사한 뒤 붙여서
  넣으면 바로 실행될 수 있는 구조로 변경해 줘."*
- **어떻게 적용**:
  - 회색 코드 상자마다 **"복사" 버튼**을 단다(`copyCode(this)` 패턴).
  - 한 단계가 두 가지 이상의 독립 동작을 섞지 않는다(① 파일 만들기 / ② 호출하기는 분리).
  - 터미널 명령은 한 줄 단위로도 의미가 통하도록, 실행 순서를 번호로 명시한다.

---

## 3. 도구 중립적 접근 (Tool-Neutral, AGENTS.md-First)

**원칙**: 특정 도구 하나에 맞춰 실습을 익히지 않는다. **공통 명세를 먼저 정의하고, 각 도구는
그 명세를 "구현"하는 순서**로 쓴다.

- **왜**: 검토 피드백 — *"과정 전체가 다양한 도구를 동시에 지원하는 걸 고려해야 한다.
  AGENTS.md 파일을 기반으로 Claude나 Antigravity가 모두 사용할 수 있도록."*
- **어떻게 적용**:
  - 역할 정의는 도구 종속 표현(`.claude/agents/*.md` YAML, `tools:` 등) 없이
    순수 텍스트 **`AGENTS.md`** 명세로 먼저 적는다(역할/입력/출력/권한/handoff).
  - 그 **같은 명세**를 Claude 계열은 사전 정의 서브에이전트로, Antigravity 계열은 자연어
    목표로 각각 구현하는 절차를 나란히 보여준다.
  - 핵심 구호: **"정의는 하나, 실행 방식만 도구별로 다르다."**
  - 실행이 플랫폼별로 크게 다른 경우(코드·명령·UI 단계), 공통 개념과 도구별 구현을
    **A/B 파일로 분리**한다(§18). 공통 영역은 3-link branch nav로 양쪽을 안내한다.

## 4. 기술적 정확성 (사전 조사 필수)

**원칙**: 외부 공식 자료와 다르면 안 된다. 확신이 없으면 조사부터 한다.

- **왜**: 검토 피드백 — *"오케스트레이션 패턴은 3가지가 아니라 더 있는 것으로 알고 있어.
  관련 내용을 조사 후 보강해 줘."* (실제로 Anthropic 공식은 **5개 패턴**: 프롬프트 체이닝,
  라우팅, 병렬화(sectioning/voting), 오케스트레이터-워커, 평가자-최적화)
- **어떻게 적용**:
  - 숫자/분류/명칭은 공식 문서(Anthropic, Antigravity 등)를 우선 인용한다.
  - "동적 팀 구성이 가능하다", "다양한 워크플로우를 조합할 수 있다" 같은 핵심 역량이
    누락되지 않았는지 점검한다(개념 장 + 실습 장 모두).
  - 문서 간 관계(`CONSTITUTION.md` ↔ `content.md` ↔ `<variant>/content.md`)를 다룰 땐
    SSOT/계층 구조를 정확히 그린다.

## 5. 운영 모델 표현의 일반화

**원칙**: 한 가지 규모를 전제로 단정 짓지 않는다. 범위는 일반화해서 표현한다.

- **왜**: 검토 피드백 — *"회사가 하나의 멀티 에이전트팀을 운영한다"는 표현은 회사 규모가 큰
  경우에 틀린다 — 본부/팀별로 다른 에이전트팀이 운영될 수 있도록 해야 한다.*
- **어떻게 적용**:
  - "전사 하나의 팀"이 아니라 **"조직 단위(본부·팀)별로 팀 운영이 가능하다"**로 쓴다.
  - 비교(서비스 티켓형 vs variant 개발도구형)에서도 한쪽을 정답처럼 단정 짓지 않고
    장·단점을 나란히 제시한다.

---

## 6. 인위적 제약 제거

**원칙**: 학습자에게 불필요한 시간/범위 제약을 문서에 포함하지 않는다.

- **왜**: 검토 피드백 — *"3시간 안에 마쳐야 한다는 제약조건이 포함되어 있어. 관련 내용을
  제거해 줘."*
- **어떻게 적용**:
  - "N시간 안에" 같은 시간 제약은 배제한다(시간 배분표는 참고용일 뿐).
  - 내용 변경 후에는 시간 배분표를 **다시 산정**하여 표와 본문이 어긋나지 않게 한다.

---

## 7. 누락된 전제조건 보완

**원칙**: "당연히 알 것"이라고 건너뛰지 않는다. 실습 직전 단계까지의 모든 준비를 적는다.

- **왜**: 검토 피드백 — *"GitHub 계정을 어떻게 만들어야 하는지가 누락되어 있어."*,
  *"PowerShell을 관리자 권한으로 실행해야 하는데 방법 설명이 없어."*
- **어떻게 적용**:
  - 계정 생성, 구독 플랜, 시스템 요구사항, 관리자 권한 여는 법(Win+X → 터미널(관리자),
    UAC "예")까지 단계별로 적는다.
  - 별도 **사전 설치 체크리스트** 문서로 분리하면 본문이 가벼워진다.

## 8. GitHub Pages 가독성

**원칙**: 정적 사이트로 배포되는 교재는 전체가 HTML로 일관되게 보이게 한다.

- **왜**: 검토 피드백 — *"GitHub Pages로 볼 때 다른 문서는 HTML인데 setup이 md라 가독성이
  떨어진다. 가독성을 높일 방법이 있을까?"*
- **어떻게 적용**:
  - 학습자가 볼 자료(`docs/`)는 `.md`가 아닌 **`.html`**로 제공한다.
  - 좌측 고정 네비게이션, "복사" 버튼, 그리고 **사이트 전체 검색**(`site-search.js`)을 둔다.
  - 단, 작성자용 내부 참고문서(본 파일 등)는 저장소 루트 Markdown으로 둔다.

## 9. OS별 명령어 처리 (가장 자주 지적되는 영역)

**원칙**: **명령어가 OS 특성 때문에 실제로 동작하지 않을 때만 OS별로 분할한다. 그렇지 않으면
단일 예제로만 보여준다.**

- **왜**: 검토 피드백 — *"MacOS/Linux에서는 동작하지만 Windows에서 할 수 없는 경우가
  발생해. 각 OS별 상황을 고려해 줘."* + *"PowerShell의 경우 cat, echo 등 macOS/Linux에서
  주로 쓰는 유틸리티 기능을 제공하고 있어."*
- **어떻게 적용**:
  - **단일 예제로 충분한 경우**(PowerShell이 별칭으로 지원): `cat`, `echo`, `mkdir`, `cp`,
    `ls`, `mv`, `cd`, `>`/`>>` 리다이렉션 등은 OS 분할 없이 한 예제로 쓴다.
  - **OS 분할이 필요한 진짜 비호환 사례**:
    - Bash heredoc 문법 `<< 'EOF'` → PowerShell은 here-string `@'...'@` + `Set-Content`/`Add-Content`
    - 프로세스 치환 `<(...)` → PowerShell은 `Compare-Object`
    - OS별 설치 스크립트(`setup-mac.sh` / `setup-linux.sh` / `setup-windows.ps1`)
    - 경로 표기 관례(`~/git` vs `C:\git`)
  - **표현의 정확성**: "PowerShell은 `cat`을 못 쓴다"가 아니라 **"`cat`은 쓸 수 있지만
    `<< 'EOF'` heredoc 문법이 없다"**로 적는다. 무엇이 안 되는지(명령 vs 문법)를 구분한다.
  - **경로 표기**: `~`(홈 디렉터리)는 macOS/Linux에서는 셸이 자동으로 확장하지만, Windows
    CMD에서는 **지원되지 않는다**. `./` 접두사 역시 CMD에서는 `mkdir` 구문 오류를 유발한다.
    가장 안전한 방법은 접두사 없이 **폴더 이름만** 쓰는 것이다.
    예: `mkdir ~/harness-lab`이 아니라 `mkdir harness-lab`(macOS/Linux, PowerShell, CMD 모두 동작).
  - **mkdir -p 권장**: `mkdir -p`는 대상 폴더가 이미 존재해도 에러 없이 넘어가므로,
    사용자가 실습 단계를 재실행할 때 안전하다. 폴더 생성 명령은 항상 `mkdir -p`로 적는다.

## 10. 시각 자료 (Visual Aids)

**원칙**: 각 섹션에 학습자의 이해를 돕기 위한 이미지나 다이어그램을 충분히 배치한다.

- **왜**: 검토 피드백 — *"문서에 적절한 이미지나 다이어그램이 있으면 이해하는데 많은 도움이
  될 듯 해."*

### 10-1. 섹션별 시각 자료 배치 기준

모든 장의 모든 섹션(§)은 적어도 하나의 시각 요소를 포함해야 한다. 시각 요소의 종류:

- **인라인 SVG 다이어그램** (아키텍처, 흐름도, 파이프라인, 비교 구조)
- **CSS 기반 시각 요소** (`.tree-box`, `.flow-box`, `.compare-grid`, `.stat-grid`, `.variant-list`)
- **HTML 표** (비교표, 분류표, 매트릭스)
- **코드 블록** (실습 섹션 한정 — 코드 자체가 시각적 구조)

| 섹션 유형 | 필수 시각 요소 | 선택 시각 요소 |
|-----------|--------------|--------------|
| 개념 정의 섹션 | 개념을 한눈에 보여주는 SVG 다이어그램 또는 비교 표 | 비유 이미지 |
| 흐름/프로세스 섹션 | SVG 흐름도 또는 단계별 파이프라인 다이어그램 | CSS 단계 표 |
| 비교/선택 섹션 | CSS compare-grid 또는 HTML 비교표 | SVG 비교 다이어그램 |
| 실습 섹션 | flow-box(자동 수행) + prompt-box(사용자 입력) | SVG 파이프라인 |
| 구조/아키텍처 섹션 | SVG 계층 다이어그램 또는 `.tree-box` | 파일 구조 표 |
| 요약/체크리스트 섹션 | 체크리스트 자체가 시각 요소 | 완성도 비교 표 |

예외: 용어집, FAQ, 설치 체크리스트 같은 참조/프로시저 문서는 시각 요소가 필수가 아니다.

### 10-2. SVG 다이어그램 작성 원칙

- **외부 이미지 의존 금지** — 모든 시각 자료는 인라인 SVG 또는 CSS/HTML로 작성한다.
- **다크모드 호환** — CSS 커스텀 프로퍼티(`var(--text)`, `var(--border)` 등)를 활용하거나,
  중립 색상을 사용한다.
- **반응형** — SVG는 `viewBox`를 설정하고 `width="100%"`로 배치하여 창 크기에 맞게 축소된다.
- **일관된 래퍼 스타일** — 모든 SVG 컨테이너는 동일한 래퍼를 사용한다:
  `<div style="background:var(--bg2);border:1px solid var(--border);border-radius:10px;padding:20px;margin:20px 0 28px;">`
- **텍스트 크기** — SVG 내 텍스트는 11~13px(`font-size`)를 사용하고, 레이블은 굵게 처리한다.
- **색상 팔레트** (일관성 유지):

  | 용도 | 색상 |
  |------|------|
  | 메인 텍스트 | `#1f2328` |
  | 보조 텍스트 | `#636c76` |
  | 강조(파랑) | `#0969da` |
  | 성공(초록) | `#1a7f37` |
  | 경고(주황) | `#953800` |
  | 에러(빨강) | `#cf222e` |
  | 보조(보라) | `#6639ba` |
  | 배경 | `#f6f8fa` |
  | 테두리 | `#d0d7de` |

### 10-3. 시각 자료 배치 예시

아래는 실제 적용된 배치 예시다. 새 과정에서는 장 구성에 맞추어 동일한 기준(§10-1)으로 배치표를 작성한다.

| 장 | 섹션 | 최소 시각 요소 | 형태 |
|----|------|--------------|------|
| N장 §0 | 개념 정의 | 개념 비교표·비유 다이어그램 | HTML 표 + SVG |
| N장 §1 | 흐름/프로세스 | 단계별 파이프라인 | SVG |
| N장 §2 | 실습 | flow-box(자동 수행) + prompt-box | CSS + code |
| N장 §3 | 구조/아키텍처 | 계층 다이어그램 또는 tree-box | SVG + CSS |

## 11. 마크업 디테일

**원칙**: 문장 중간의 단일 단어 강조로 인한 불필요한 줄바꿈을 피하고, 중첩 flex 레이아웃에서
글상자가 부모 테두리 밖으로 삐져나가지 않게 한다.

- **왜**: 검토 피드백 — *"불필요하게 줄이 바뀌어져 있어."* (한국어 문장 중간의 한 단어만
  `<strong>`으로 감싸 렌더링 시 단어가 따로 떨어져 보인 사례) / *"글상자가 외부 테두리 밖으로
  나간 뒤 최종 테두리에 의해 잘리고 있어."* (`.flow-box` 등이 `.platform-block` 경계를 벗어나
  `.scenario-card`의 `overflow: hidden`에 잘린 사례)
- **어떻게 적용**:
  - 강조는 **자연스러운 어구 단위**로 묶는다(예: `heredoc <strong>문법</strong>`이 아니라
    의미가 통하는 구절 전체, 또는 강조 제거).
  - 비교 글상자/그리드는 `min-width: 0`을 주어 긴 줄이 박스를 밀어내지 않게 한다.

### 11-1. flex 레이아웃에서 글상자 오버플로우 방지

실습 페이지의 구조는 `.scenario-card` > `.card-body` > `.platform-block` > `ol.step-list` >
`li` > `.step-content` > (`.flow-box` / `.tip-box` / `.warning-box`)로 깊이 중첩된다.
여기서 `.step-list li`는 flex 컨테이너이고 `.step-content`는 `flex: 1` 자식이다.

CSS flex에서 **flex 자식의 기본 `min-width`는 `auto`**이므로, 내부 콘텐츠의 고유 너비보다
줄어들지 않는다. 글상자 안의 긴 `<code>` 인라인 요소(예: `<code>subagent_type="writer"</code>`)
가 `.step-content`를 밀어내어 `.platform-block`의 테두리 밖으로 삐져나가고, 최종적으로
`.scenario-card`의 `overflow: hidden`이 잘라버린다.

또한 `.platform-block`은 자체 padding(20px 양쪽)을 가지므로, 그 안에 있는 글상자는
`.card-body`에 직접 있는 글상자보다 40px 더 좁게 보인다. 이를 보정하기 위해 음수 마진을 준다.

**필수 CSS 수칙** (새 HTML 파일에 `step-list` 구조를 쓸 때 반드시 포함):

| 선택자 | 속성 | 효과 |
|---|---|---|
| `.step-content` | `min-width: 0` | flex 자식이 콘텐츠 고유 너비 이하로 줄어들 수 있게 허용 |
| `.flow-box`, `.tip-box`, `.warning-box` | `overflow-wrap: break-word` | 긴 인라인 요소가 박스 경계에서 줄바꿈 |
| `.platform-block .flow-box` 등 | `margin-left/right: -10px` | padding 누적으로 좁아지는 글상자 너비 보정 |

적용 예:
```css
.step-content { flex: 1; min-width: 0; }
.flow-box {
  /* ... 기존 속성 ... */
  overflow-wrap: break-word;
}
.platform-block .flow-box,
.platform-block .tip-box,
.platform-block .warning-box {
  margin-left: -10px;
  margin-right: -10px;
}
```

### 11-2. flex 컨테이너 안의 고정 요소 줄바꿈 방지

`.level-header`(`display: flex`) 안에 배지, 타이틀, 설명문을 나란히 배치할 때,
기본 `flex-shrink: 1` 때문에 창이 좁아지면 배지·타이틀이 줄바꿈되어 2줄로 보일 수 있다.

**수칙**: flex 컨테이너 안에서 **절대 줄바꿈되면 안 되는 요소**에는 `flex-shrink: 0`을 주고,
**나머지 공간을 채우는 요소**에는 `flex: 1`을 준다.

```css
.level-header .badge    { flex-shrink: 0; }
.level-header .level-title { flex-shrink: 0; }
.level-header .level-desc  { flex: 1; }
```

## 12. 문체 일관성 (Writing Style Consistency)

**원칙**: 모든 장·섹션에서 동일한 문체 기준을 적용하여 독자가 장을 넘나들 때 위화감이 없게 한다.

- **왜**: 장마다 글쓰기 스타일이 다르면 교재로서의 일관성이 떨어진다. 실제로 초안에서 존댓말 체,
  영어 용어 표기, 장 참조 형식, em-dash 사용 빈도가 파일별로 제각각이었다.
- **어떻게 적용**:

### 12-1. 종결 어미

본문·부제·설명문 모두 **해라체(`~다`/`~한다`/`~이다`)**로 통일한다.
하십시오체(`~습니다`/`~해요`)는 UI 버튼 문구 등에서만 허용한다.

```
✗ "이 핸드북은 다르게 구성되어 달라집니다."  (하십시오체)
✓ "이 핸드북은 다르게 구성되어 달라진다."     (해라체)
```

### 12-2. 영어 기술 용어 표기

본문에서 영어 기술 용어를 쓸 때는 **문서(파일) 안에서 처음 등장할 때만** 한글(English) 괄호 풀이를 쓰고,
같은 문서 안에서 그 뒤로 다시 나올 때는 영어 단어만 그대로 쓴다.
코드·파일명·CLI 식별자는 `<code>` 태그로 표기하되, 이와 구분되는 개념적 용어는 괄호 풀이를 쓴다.

```
✗ "hook은 셸 명령 실행 전후에 호출된다."                     (첫 등장인데 영어만 — 풀이 없음)
✗ "<code>hook</code>은 셸 명령 실행 전후에 호출된다."          (code 태그로 대체 — 개념어인데 코드로 표기)
✓ "훅(hook)은 셸 명령 실행 전후에 호출된다. ... 이후 문단에서
   hook이 실패하면 커밋이 중단된다."                          (첫 등장만 병기, 이후 영어 단독)
✗ "... 이후 문단에서 훅(hook)이 실패하면 ..."                 (같은 문서에서 두 번째부터도 계속 병기 — 불필요한 반복)
```

이 규칙은 **문서(파일) 단위**로 적용된다 — 다른 문서에서는 그 문서에서의 첫 등장이 다시 기준이 된다(예: 4장에서 이미 "훅(hook)"을 풀이했어도 8장에서 처음 나오면 다시 풀이한다).

**제목은 예외**다. `<title>`, `<h1>`, `chapter-eyebrow`, nav의 장 제목처럼 **제목 역할을 하는 텍스트**는 괄호 풀이 없이 **영어 단어만** 쓴다(한글 단독 표기도 쓰지 않는다).

```
✗ <h1>신규 베리언트(variant) 만들기</h1>   (제목에 괄호 풀이)
✗ <h1>신규 베리언트 만들기</h1>            (제목에 한글 단독)
✓ <h1>신규 variant 만들기</h1>             (제목은 영어만)
```

### 12-3. 장·절 참조 형식

다른 장이나 절을 참조할 때는 **`N장 §M`** 형식으로 통일한다.

```
✗ "5장에서 자세히 다룬다"
✗ "제5장에서 다룸"
✓ "5장 §3에서 다룬다"
```

### 12-4. em-dash(—) 사용 최소화

em-dash는 부연 설명이 불가피한 곳에만 **최소한으로** 쓴다.
문장을 짧게 쪼개면 em-dash 없이도 의미가 전달되는 경우가 대부분이다.
실습 장(Examples)에서는 특히 사용을 삼간다.

```
✗ "좋은 하네스는 매번 잘 대답하게 만든다 — 그게 핵심 차이다"
✓ "좋은 하네스는 매번 잘 대답하게 만든다. 이것이 핵심 차이다."
```

## 13. 도구 비교의 완결성과 균형

**원칙**: 비교표는 오해를 유발하지 않고, 항목이 빠지지 않으며, 시각적으로 균형 잡힌다.

- **왜**: 검토 피드백 — (a) *"`/goal`, `/agent`, `/agents`로 적으면 다른 도구엔 없거나
  Antigravity 전용인 것처럼 오해의 소지가 있어."*, (b) *"서브에이전트 호출 시 메시지 전달,
  서브에이전트 간 소통 여부가 누락됐어."*, (c) *"비교 글상자 크기 밸런스가 안 맞아."*
- **어떻게 적용**:
  - 한 도구에만 있는 것처럼 쓰지 않는다(공통 역량은 "공통"으로 명시).
  - 비교 차원(호출·병렬·격리·메시징·소통)을 정해두고 **각 도구별로 빠짐없이 채운다**.
  - 컬럼/박스는 동일한 그리드 규칙·여백으로 높이를 맞춘다.

## 14. 강의 소개자료 필수 포함 원칙

**원칙**: 강의 자료(워크숍, 코스, 세미나 등)를 생성할 때 **강의 소개(Course Overview)를 반드시
포함**한다. 소개자료 없이 본문이나 실습만 먼저 만들지 않는다.

- **왜**: 참가자 모집·강의 안내·운영 승인 등 모든 외부 커뮤니케이션의 시작점이 강의 소개다.
  소개가 없으면 "이게 어떤 강의인지"를 설명하는 데 매번 본문을 뒤져야 한다. 또한 강의 목표가
  명확히 정의되지 않은 상태에서 본문을 쓰면 내용의 범위와 깊이가 임의로 흔들린다.
- **어떻게 적용**:
  - 강의 기획 단계(초안 작성 전)에 **강의 소개를 먼저 작성**하고, 그 내용을 기준으로 본문 구조를
    잡는다. 소개에 적힌 학습 목표가 본문의 실제 섹션과 1:1로 매핑되는지 점검한다.
  - 강의 소개에 다음 항목을 **모두** 포함한다(누락 금지):

    | 항목 | 내용 | 비고 |
    |------|------|------|
    | 한 줄 요약 | 이 강의가 무엇인지 2~3문장 | 참가자가 10초 안에 파악 |
    | 학습 목표 | 수료 후 참가자가 할 수 있는 것 (Bloom's Taxonomy 기반) | 설명·설계·적용·실행·비교·판단·구현·종합 등 동사로 시작 |
    | 대상자 | 이 강의가 누구에게 적합한지 | 역할·경험 수준 기준으로 명시 |
    | 사전 요구사항 | 강의 전 준비해야 할 것 | 구독·계정·설치·지식 수준 |
    | 강의 형태 | 강의/실습/토론/시연 비율 | 참가자가 기대를 조정할 수 있음 |
    | 강의 일정 | 시간 단위의 블록별 일정 | 휴식 포함 총 소요 시간 명시 |
    | 다루는 주제 | 블록별 주제와 관련 장/섹션 매핑 | 일정표와 교차 검증에 활용 |
    | 수료 후 얻게 되는 것 | 실무에 즉시 적용 가능한 산출물·역량 | 참가 동기를 부여 |
    | 강사 정보 | 강사 이름·소속·프로필 (빈 칸 허용) | 강사가 직접 채우도록 설계 |

  - 강의 소개 파일은 `docs/lecture-guide/00_Course_Overview.html`(또는 해당 프로젝트 규칙에 맞는 경로)에
    배치하고, 인덱스 페이지(`docs/index.html`)의 강사용 자료 그룹 최상단에 카드로 링크한다.
  - 본문이나 시간 배분표를 수정할 때마다 **강의 소개의 일정·주제·학습 목표가 여전히 일치하는지**
    동기화한다.

## 15. 장 연결성과 목차 보완

**원칙**: 각 장은 다음 장과 매끄럽게 이어지고, 목차 전체를 돌아보며 빈 영역을 제안한다.

- **왜**: 검토 피드백 — *"각 장이 다음 장과 잘 연결되어 있는지도 확인해 줘."*,
  *"목차를 보고 보완해야 할 분야나 추가해야 할 분야를 제안해 줘."*
- **어떻게 적용**:
  - 장 말미에 "다음 장에서는 ~를 다룬다"는 연결 다리를 둔다.
  - 목차 리뷰 시 **보조 자료(용어집·가드레일·FAQ·캡스톤·강의 가이드)**가 빠져있지 않은지
    점검한다.

### 15-1. 마지막 장의 마무리 동선

**원칙**: 과정의 마지막 장(캡스톤 등)은 단순히 "다음 장 →"이 없다고 끝내지 않고,
**수료 후 다음 단계**를 별도 섹션으로 제시한다.

- **왜**: 중간 장은 "다음 장에서는 ~"로 자연스럽게 이어지지만, 마지막 장은 그 장치가 없어
  학습자가 "그래서 이제 뭘 하면 되지?"라는 질문을 안은 채 끝나기 쉽다.
- **어떻게 적용**:
  - 마지막 장에 `수료 후 다음 단계` 섹션(`<h2 id="next-steps">`)을 두고, 체크리스트 형태로
    구체적인 행동을 제시한다 — 예: "실제 업무에 적용해보기", "정식 variant로 승격 검토하기",
    "막히면 용어집/FAQ로 돌아가기".
  - `chapter-nav`의 `next`는 본문 장이 없으므로 용어집·FAQ 같은 참고 자료로 연결하되,
    라벨을 `다음 장 →`이 아니라 `참고 →`처럼 성격이 다름을 표시한다.

## 17. 참고 영상 자료 (Video References)

**원칙**: 섹션의 내용을 보완하는 공식 영상 자료가 있으면 "참고 영상" 섹션에 링크를 제공한다.

- **왜**: 동일한 개념을 시각적으로 설명하는 영상은 학습자의 이해를 크게 높인다.
  특히 오케스트레이션 패턴, 에이전트 동작 흐름, 도구 데모 같은 동적인 내용은
  글과 다이어그램만으로 전달하기 어렵다.

### 17-1. 영상 선정 기준

- **공식 출처 우선**: Anthropic, Google DeepMind, DeepLearning.AI 등 공식 채널을 우선한다.
- **언어**: 한국어 영상을 최우선으로 찾고, 없으면 영어 영상을 링크하되 언어를 명시한다.
- **길이**: 20분 이내의 짧은 영상을 선호한다. 긴 영상은 타임스탬프를 안내한다.
- **시의성**: 발행일이 최근 2년 이내인 것을 선호한다.
- **무료 접근**: YouTube 등 누구나 볼 수 있는 플랫폼에 게시된 것만 링크한다.

### 17-2. 영상 링크 배치 방식

영상은 직접 임베드(`<iframe>`)하지 않고, 섹션 말미 또는 장 말미 "참고 영상" 블록에
링크로 제공한다. 이유: (a) 임베드는 로딩 속도를 저하시키고, (b) 외부 플랫폼 정책 변화로
깨질 수 있으며, (c) 학습자가 자신의 환경에서 보는 것을 선호할 수 있다.

HTML 패턴:
```html
<div class="video-refs">
  <h3 id="videos">참고 영상</h3>
  <ul>
    <li>
      <span class="video-badge en">EN</span>
      <a href="YOUTUBE_URL" target="_blank">영상 제목</a>
      <span class="video-meta">(MM:SS)</span> — 1줄 설명
    </li>
  </ul>
</div>
```

CSS:
```css
.video-refs {
  margin: 28px 0;
  padding: 16px 20px;
  background: #f0f6ff;
  border: 1px solid #d0e8ff;
  border-radius: 8px;
}
.video-refs h3 {
  font-size: 15px;
  font-weight: 700;
  color: var(--accent6);
  margin: 0 0 12px;
}
.video-refs ul { list-style: none; padding: 0; margin: 0; }
.video-refs li { padding: 6px 0; line-height: 1.6; }
.video-badge {
  display: inline-block;
  font-size: 11px;
```

### 17-3. 영상 제공이 권장되는 섹션

## 18. 플랫폼별 분리 (A/B Branching)

**원칙**: 동일 개념을 Claude 계열과 Antigravity 계열이 각각 다른 방식으로 구현하는 섹션은,
공통 영역과 도구별 특성 영역으로 분리한다.

- **왜**: 하나의 파일에 두 계열의 구현을 모두 담으면 (a) 사용자가 자신의 도구와 무관한 내용을
  건너뛰야 하고, (b) 파일이 지나치게 길어지며, (c) 어느 한쪽의 업데이트가 다른 쪽과 어긋날 위험이 있다.
- **어떻게 적용**:

### 18-1. 분리 기준

| 조건 | 분할 여부 |
|------|----------|
| 코드/명령이 플랫폼별로 다름 | 분할 |
| UI 조작 단계가 플랫폼별로 다름 | 분할 |
| 지원 기능 자체가 다름(지원/미지원) | 분할 |
| 개념 설명, 공식 명세 비교표 | 통합 유지 |
| 요약, 참고 링크 | 통합 유지 |

### 18-2. 명명 규칙

| 요소 | A (Claude) | B (Antigravity) |
|------|-----------|----------------|
| 파일명 | `_Examples.html` (접미사 없음) | `_Examples_B.html` |
| 섹션 레이블 | `§N-A` | `§N-B` |
| 색상 | Green `#1a7f37` | Purple `#6639ba` |
| badge class | `badge-claude`, `badge-claudeapp` | `badge-antigravity`, `badge-agy` |

### 18-3. 3-link Branch Navigation (공통 → 도구별)

공통 영역(레퍼런스 파일) 하단에는 `prev | branch(A, B) | next` 3-link 패턴으로
도구별 파일로 분기하는 네비게이션을 배치한다.

HTML 템플릿:
```html
<div class="chapter-nav">
  <a href="이전_장.html" class="prev">
    <div class="dir">← 이전</div><div class="ttl">N장 · 제목</div>
  </a>
  <div class="branch">
    <div class="branch-label">N장 실습</div>
    <a href="Examples.html" style="border-color:#1a7f37;color:#1a7f37;">Claude (A)</a>
    <a href="Examples_B.html" style="border-color:#6639ba;color:#6639ba;">Antigravity (B)</a>
  </div>
  <a href="다음_장.html" class="next">
    <div class="dir">다음 →</div><div class="ttl">N장 · 제목</div>
  </a>
</div>
```

CSS (필요 시 파일에 추가):
```css
.chapter-nav .branch { flex: 0 0 auto; text-align: center; min-width: 120px; }
.chapter-nav .branch a { display: block; font-size: 12px; margin-bottom: 6px; padding: 10px 14px; }
.chapter-nav .branch-label { font-size: 10px; color: var(--text-dim); margin-bottom: 6px; text-transform: uppercase; letter-spacing: .05em; }
```

### 18-4. 2-link Chain Navigation (A↔B 순차)

A/B 파일 상호 간에는 prev/next 링크로 순차 연결한다.

- **A 파일 하단**: `← §N 공통 영역` | `§N-B Antigravity 중심 →`
- **B 파일 하단**: `§N-A Claude 중심 →` | `다음 장 →`

### 18-5. platform-pair / platform-block 규칙

- **`.platform-pair`**: 2-column 비교 그리드 — 공통 레퍼런스 파일에서만 사용.
  4도구를 나란히 비교할 때 쓴다(예: 4장 §1 레퍼런스).
- **`.platform-block`**: 단일 플랫폼 명령 블록 — A/B 분할 파일에서는
  해당 플랫폼의 블록만 표시한다. 반대쪽 플랫폼 블록은 제거한다.
- **4-tool 분류 체계**: Claude App(Desktop), Claude Code(CLI),
  Antigravity(Desktop), Antigravity CLI(agy).

### 18-6. A/B 분할 체크포인트

분할이 완료되면 다음을 확인한다.

- [ ] 공통 영역에서 A/B 양쪽으로 branch navigation이 연결되었는가?
- [ ] A/B 파일 간 상호 링크가 일관되게 되어 있는가?
- [ ] A 파일에는 Claude 계열 내용만, B 파일에는 Antigravity 계열 내용만 있는가?
- [ ] index.html에 A/B 카드가 나란히 배치되었는가?

## 19. 탈고 전 오탈자·문맥 점검 (Proofreading)

**원칙**: AI가 생성한 초안은 반드시 사람이 전체 문장을 다시 읽으며 오탈자, 어색한 문맥, 번호 불일치를
수정한다.

- **왜**: AI가 생성한 초안에는 맞춤법 오류, 문맥에 어울리지 않는 표현, 장/절 번호 불일치가
  남아 있을 수 있다. 교재가 아무리 구조적으로 완비되어도 오탈자가 많으면 신뢰도가 떨어진다.
- **어떻게 적용**:
  - 장/절 파일을 작성한 뒤, **장 전체를 처음부터 끝까지 읽으며** 다음을 확인한다:
    - 오탈자(맞춤법, 띄어쓰기)
    - 문맥에 어울리지 않는 표현(직역투, 어색한 번역투)
    - 장/절 번호 불일치(다른 장을 참조하면서 번호가 틀린 경우)
    - 앞뒤 문장의 논리 단절(인과관계가 거꾸로 된 경우 등)
  - 오탈자 점검은 §A 체크리스트의 **최종 단계**로 수행한다.
  - 강사 운영 자료(강의 진행 가이드, 강의 소개)도 동일하게 적용한다.

## 20. 강의 진행 가이드 (Lecture Guide) 구조

**원칙**: 강의 진행 가이드는 강사가 강의를 운영하는 데 필요한 모든 정보를 한 곳에 모아 제공한다.
본 가이드라인의 강의 소개 원칙(§14)과 함께 강의 자료의 양대 축을 이룬다.

- **왜**: 강사가 여러 장의 내용을 오가며 진행해야 하므로, 장별 강사 노트·시간 배분표·확인 질문이
  한 문서에 모여 있어야 실시간 참조가 가능하다. 또한 강의 소개(참가자용)와 강의 진행 가이드(강사용)가
  내용·순서·시간에 대해 서로 일치해야 한다.
- **어떻게 적용**:

### 20-1. 필수 섹션

강의 진행 가이드에는 다음 섹션을 **모두** 포함한다.

| 섹션 | 내용 | 비고 |
|------|------|------|
| 사전 준비물 체크리스트 | 참가자가 강의 전 준비해야 할 것 | 강의 최소 하루 전 공지용 |
| 시간 배분표 | 일별·장별 구분, 형태, 예상 시간 | 휴식은 별도 산정 |
| 장별 강사 노트 | 각 장의 강의 팁, 참가자 활동, 시연 유의사항 | 시간이 부족할 때 대체 방법 포함 |
| 장별 확인 질문 | 각 장 마무리에 던질 이해도 확인 질문 | 정답 맞히기가 아닌 설명 유도용 |

### 20-2. 강의 소개와의 일치성

강의 진행 가이드는 §14의 강의 소개(Course Overview)와 **다음 항목이 일치**해야 한다.

- 일별 강의 순서(장 배치)
- 장별 예상 시간
- 장별 내용 설명
- 총 소요 시간

한쪽을 수정하면 반드시 다른 쪽도 동기화한다.

### 20-3. 장별 강사 노트 작성 형식

각 장의 강사 노트에는 다음 항목을 포함한다.

- **위치와 역할**: 이 장이 전체 강의의 어느 지점에 있고, 왜 그 순서인지
- **핵심 전달점**: 반드시 전달해야 할 개념(나머지는 시간에 따라 생략 가능)
- **참가자 활동**: 토론·투표·짝 활동 등 참가자가 직접 하는 것
- **시간 부족 시 대체**: 시간이 모자라면 어떤 부분을 시연으로 대체할 수 있는지
- **비가역적 작업 경고**: 환경에 복구 불가능한 영향을 주는 작업 등 참가자 전원이 따라 하면 안 되는 것

### 20-4. 확인 질문 작성 형식

- 장당 **2~3개**의 개방형 질문을 작성한다.
- "정답을 맞히는" 것이 아니라 **"스스로 설명해보게 하는"** 질문으로 구성한다.
- 해당 장의 핵심 개념을 다른 장과 연결하는 질문을 포함하면 통합적 이해에 도움이 된다.
- 예:
  - "이 개념이 우리 팀 업무에 어떻게 적용될까?" (실제 적용)
  - "N장에서 다룬 ~와 이 장의 ~의 차이는 무엇인가?" (장 간 연결)

## 21. 네비게이션과 인덱스 구조

**원칙**: 학습자가 장·섹션 간을 자연스럽게 이동할 수 있도록 모든 HTML 파일에 일관된 네비게이션 구조를 적용하고,
인덱스 페이지에서 전체 구성을 한눈에 파악할 수 있게 한다.

- **왜**: 장이 10개 이상 넘어가면 학습자가 "지금 어디에 있고, 다음이 뭔지"를 파악하기 어렵다.
  사이드바·장 간 이동 링크·인덱스 카드가 없으면 학습 흐름이 끊긴다.
- **어떻게 적용**:

### 21-1. 장 내부 네비게이션

모든 장 HTML 파일에는 다음 네비게이션 요소를 포함한다.

- **사이드바(nav)**: 좌측 고정. 장 내 절(`#id`)의 링크와 "핸드북 홈" 링크를 포함.
- **장 간 이동(chapter-nav)**: 장 하단에 `← 이전 장 | 다음 장 →` 링크를 배치.
- **A/B 분할이 있는 장**: §18의 3-link branch navigation 또는 2-link chain navigation을 적용.

### 21-2. 인덱스 페이지 (index.html)

인덱스는 핸드북 전체의 진입점이다. 다음 구조를 포함한다.

- **강의 제목과 한 줄 설명**: 상단에 배치.
- **일별·유형별 그룹**: 강의 일수별(예: 1일차/2일차)과 형태별(개념/실습)로 장 카드를 그룹화.
- **장 카드**: 각 장의 제목, 1~2줄 설명, 장 번호 태그, 링크를 포함. 개념 장과 실습 장이 시각적으로 구분되게 한다.
- **강사용 자료 그룹**: Course Overview, Lecture Guide 등 강사 전용 자료를 별도 그룹으로 배치.

### 21-3. 일관성 유지

- 모든 장 파일은 동일한 HTML 템플릿(layout, nav, main, article) 구조를 사용한다.
- 사이드바의 "핸드북 홈" 링크는 모든 파일에서 동일한 경로(`../index.html` 또는 `./index.html`)를 가리킨다.
- 장 번호, 태그 색상, 링크 스타일은 장 파일과 인덱스 카드에서 일관되게 적용한다.

### 21-4. 장 번호 변경 시 정합성 검증 (Renumbering Integrity)

**원칙**: 장을 신설·삭제·순서 변경할 때는 **파일 이동만으로 끝내지 않고**, 아래 네 가지를
빠짐없이 갱신·검증한다. 재번호 작업에서 가장 자주 놓치는 지점들이다.

- **왜**: 실제로 장을 재배치하는 과정에서 ①`chapter-nav`의 prev/next가 새 순서를 반영하지
  못해 건너뛰거나 되돌아오지 못하는 링크가 생겼고, ②링크는 새 파일을 가리키는데 라벨 문구는
  옛 장 번호를 그대로 남긴 사례가 반복됐고, ③새로 만든 파일이 `site-search.js`의 `DOCS`
  배열에 추가되지 않아 전체 검색에서 누락됐다.
- **어떻게 적용**:
  1. **prev/next 상호 대칭** — A 문서의 `next`가 B를 가리키면, B 문서의 `prev`는 반드시 A를
     가리켜야 한다. 한쪽만 고치고 반대쪽을 놓치면 "다음"으로는 갈 수 있는데 "이전"으로는
     못 돌아오는(또는 그 반대인) 링크가 생긴다.
  2. **링크 라벨과 대상 일치** — `<a href="...">라벨</a>`의 라벨 문구(장 번호·제목)는 항상
     `href`가 실제로 가리키는 파일의 현재 장 번호·제목과 일치해야 한다. 파일을 옮기면서
     `href`만 바꾸고 라벨 텍스트는 복사해 온 옛 문구를 그대로 두는 실수가 잦다.
  3. **검색 인덱스 동기화** — 새 장 파일을 추가하거나 경로를 바꾸면 `docs/assets/site-search.js`의
     `DOCS` 배열에도 항목을 추가·수정한다. 워밍업처럼 번호가 없는 실습 장도 빠뜨리지 않는다.
  4. **사이드바 "다른 장" 목록** — 재번호로 영향을 받는 모든 장의 사이드바 "다른 장" 목록과
     `index.html`의 카드 순서·태그 번호도 함께 갱신한다.
  - **검증 방법**: 아래 스크립트로 ①깨진 링크, ②prev/next 비대칭을 기계적으로 확인할 수 있다
    (`docs/` 하위 모든 HTML을 대상으로 실행).

    ```
    # (1) 깨진 링크: href가 가리키는 파일이 실제로 존재하는지
    # (2) prev/next 비대칭: A→next→B인데 B→prev→A가 아닌 경우
    # 둘 다 Python 스크립트 한 번으로 확인 가능 — 재번호 PR마다 실행 권장
    ```

    `scripts/validate-nav.ts` 자동 검증 스크립트로 위 네 가지를 CI에서 자동 검증함
    (`bun run scripts/validate-nav.ts`).
