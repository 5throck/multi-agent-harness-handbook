Lecture Guide

This is material for instructors delivering a lecture using the entire handbook (Chapters 1–13 plus the Common Reference).

This document is not part of the handbook body but is operational material for instructors only. It does not need to be shared with learners.
Before finalizing, be sure to re-read every sentence to check for typos, phrasing that doesn't fit the context, and chapter/section number mismatches. AI-generated drafts may still contain errors or awkward phrasing.

Consolidated Prerequisites Checklist

This consolidates items scattered across each chapter's setup document (Pre-Installation Checklist) so they can be checked all at once before the lecture begins. Announce this list to participants at least one day before the lecture starts, so on-site time isn't wasted on installation issues.

☐ An active Claude Pro/Max subscription (a subscription account, not an API key)
☐ A Google account for using Antigravity
☐ A GitHub account that can run gh auth login. Participants without an account should be notified in advance to follow the signup instructions in the Chapter 4 §2-A practice setup section
☐ At least 5GB of free disk space, and administrator/sudo privileges
☐ Claude App (Desktop), Claude Code (CLI), Antigravity (Desktop), and Antigravity CLI (agy) installed. Verify using the OS-specific scripts in the Chapter 4 §2-A practice setup section
☐ Confirm that bun --version and git --version output normally
☐ (For Chapter 6/11 exercises) Clone git clone https://github.com/5throck/ai-workspace-standards.git in advance into the workspace root location (Windows C:\git, macOS/Linux ~/git)
The P-1 scenario in the Chapter 11 new-variant promotion exercise runs a promotion pipeline that affects real git history and file structure. Do not have every participant run it in their own environment — follow the guidance in the "Chapter-by-Chapter Instructor Notes" section below.

Chapter-by-Chapter Instructor Notes

Chapter 1: Work Transformation in the AI Era (Introduction)

This chapter is an introduction that establishes the motivation of "why learn AI" ahead of the technical concepts in Chapter 2. Show the statistics cards (McKinsey 78%, Gartner 80%+, etc.) to convey the acceleration of AI adoption, and use the 3-stage AI usage table (user → power user → designer) to help participants identify where they currently stand.

The harness-engineering-necessity comparison box (without harness vs. with harness) is a preview of a concept covered in detail in Chapter 2. Here, convey only the "feel" of it, and leave the technical details for Chapter 2. The Klarna case (2.3M conversations = 700 agents) is useful for emphasizing that AI redistributes roles rather than replacing people.

Participant activity Having participants raise their hands to vote on "Which of the three AI-usage stages do you feel closest to right now?" builds curiosity that carries into Chapter 2.

Chapter 2: Concepts

Because this chapter establishes the baseline terminology (orchestrator, specialist, handoff, subagent) used throughout every later chapter, it's fine to slow the pace here. Understanding improves significantly if you sketch the analogies (cooking, a new hire, a delivery invoice) on a whiteboard while explaining.

Participant activity It's recommended to have participants pick one of the three patterns in §4 (fan-out/pipeline/verify-then-apply) and spend a minute discussing with a neighbor "How would this apply to our team's work?"

Chapter 3: Guardrails and the Permission Model

There's a reason this must be placed right before the Chapter 4 §2 exercise: it builds the habit of deciding "what to allow and what to block" before actually building a subagent. Showing the 3-tier risk classification table and asking in advance, "Which tier would the reviewer you're about to build fall into?" leads naturally into the Chapter 4 §2 exercise.

Warm-up exercise: From Tetris to Pac-Man

Run this right after Break 1, before the main Chapter 4 exercises. Use claude.ai or Claude Code; unlike Chapter 4, where groups use different tools, everyone uses the same platform here.

Tetris (10 min): The instructor demonstrates for 1 minute first, then participants follow along. Typing "Build a Tetris game with HTML Canvas" produces a working result immediately, from vibe-coding alone. Modification requests (background color, scoreboard, etc.) are also handled without issue. At this point, emphasize that "complexity at this level (~200 lines) is fine even without a harness."

Pac-Man (25 min): First have participants try a single prompt ("Build a Pac-Man game with HTML Canvas") and experience failure for 2–3 minutes — when character movement, ghost AI, collision detection, and map design are all crammed together, the code becomes tangled and doesn't work. Then display the 3-step prompt on screen. Step 1, planning agent ("Write a spec listing Pac-Man's feature list and map structure"); Step 2, development agent ("Write the HTML Pac-Man code based on the spec above"); Step 3, QA agent ("Find and fix bugs in this code"). Roughly 7–8 minutes per step.

Takeaways (5 min): Show the difference in code volume between Tetris (~200 lines) and Pac-Man (~1,000–1,500 lines), and remind participants that they've just directly experienced the single-agent limitation (context overload) from §2 of Chapter 2. Connect to Chapter 4 by saying "You'll go deeper with tool-specific subagents in the Chapter 4 exercise, building on what you just felt in this warm-up."

Chapter 4 §1 / Chapter 4 §2: Exercises

Splitting participants into 4 groups (Claude App / Claude Code / Antigravity Desktop / Antigravity CLI) and having each run the same scenario with a different tool makes the Common Reference "tool comparison" content land much better when results are compared at the end.

Participant activity G-1, D-1, and D-2 (observing dynamic team composition) should be run by everyone. P-1 (parallel execution) and P-2 (model-tier comparison) can be replaced with an instructor screen demo if time is short. Even having just one person per group try the Agent Teams setting (`.claude/settings.json`) or swapping the `model` field is enough to convey the concept.

Chapter 5 (light touch): Introducing ai-workspace-standards

On Day 1, cover only the minimal concepts needed for the Chapter 6 exercise. It's enough to convey what the repository is, the roles of the 4 core files, what a variant is, and the basic flow of creating a project via scaffolding. The L0→L1→L2 layer structure and model tiering are covered in the Day 2 deep dive.

Common Reference: Tool Comparison (Day 2)

Placed on Day 2. Since participants directly experienced the tools during the Day 1 Chapter 4 §1/§2 exercises, revisiting it in the form of "why did the tool you just used behave this way" is far more engaging than a pure lecture. Because the audience is IT professionals, technical details such as teammateMode (in-process vs. tmux) and the internal structure of the Agent Manager can be covered in depth.

Chapter 7: Comparing the Two Models + Kanban (Day 2)

Placed on Day 2. For the kanban section, it's effective to first ask any participants with actual in-house helpdesk/support-team operations experience to share that experience before starting. Rather than explaining "why it's needed" in theory first, opening with "Have you ever experienced a backlog of tickets?" builds rapport quickly. Since the audience is IT professionals, it's good to also mention practical implementation methods such as GitHub Projects/Jira integration.

Discussion Have a 5-minute small-group discussion on "Which model would fit our organization?" followed by presentations.

Chapter 8: The ai-workspace-standards Architecture (Day 2)

Day 2 picks up where the light Day 1 introduction left off, covering in detail the L0→L1→L2 layer structure, the internal structure of the 7 templates/variants, model tiering, the session-start checklist, and even the not-yet-implemented advancement roadmap (§5). L2 project upgrades are covered separately in Chapter 10. There's no need to have participants memorize everything — it's enough that the motivation of "why this layer structure is needed" comes through clearly.

Chapter 10: L2 Project Upgrades (Day 2)

This is the practical-application chapter for the fork model learned in Chapter 8 §1. Start by reminding participants that "changes to the L1 template are not automatically reflected in your project." Understanding improves if you visually show the workspace-management marker structure for the MERGE category among the 5 classifications (LOCKED, MERGE, PRESERVE, SYNC_IF_NEWER). Emphasize the habit of running --dry-run first, and it's recommended that the instructor demonstrate the actual script execution.

Participant activity If time allows, having participants run --dry-run against their own Chapter 6 exercise project (G-1 or G-2) makes it more tangible.

Chapter 6: Using an Existing Variant (Day 1)

This exercise devotes all of Chapter 6 to using an existing variant. G-1 (co-consult Physical AI market research) gets 40 minutes and G-2 (co-deck Korean-cuisine-globalization presentation) gets 40 minutes, giving the two exercises a combined 80 minutes — ample time. This is the core hands-on time of the two-day workshop.

Since running new-project.ts can actually take a long time due to the GitHub clone, it's important to announce in advance that participants should clone the workspace before starting this exercise (see the prerequisites checklist above).

Participant activity In G-1, have participants directly apply the 4-stage design framework (goal, input, output, verification criteria), run the agent team, and track the handoffs. In G-2, have them directly set the presentation topic frame (audience, purpose, time), review and improve the finished slides, and even experience the TTS and auto-slide setup. Guide them to directly compare the structural differences between the two variants using the G-1 and G-2 deliverables. Participant activity During the Day 1 wrap-up (10 min), have everyone share their G-1 report deliverable and G-2 presentation deliverable. Asking "What happens if you feed the co-consult report into co-deck as input?" naturally builds motivation for creating a new variant on Day 2.

Chapter 11: Creating a New Variant

This chapter is placed on Day 2. Building on the experience of directly working with an existing variant on Day 1, participants now build their own variant from scratch. It consists of a 40-minute exercise (D-1) plus a 20-minute lecture.

Participant activity D-1 (building a co-marketing Phase A prototype hands-on) is done by everyone. Since participants already looked at the frontmatter structure of agent definition files during G-1 on Day 1, this time they build the researcher and writer themselves. If time is tight, pair work (one person typing, one running the verification commands) also works well.

Chapter 9: Workflow Design Patterns

After reviewing the 5 orchestration patterns, show real workflow case studies such as code review, market research, and presentations. Work through a "Which pattern fits this task?" decision tree together.

Chapter 12: New-Variant Promotion

Placed in the latter half of Day 2. This is the stage of preparing the co-marketing prototype built in Chapter 11 for promotion.

Instructor demo only Actually running l2-to-variant-pipeline.ts is an irreversible operation that affects git history. Don't have every participant follow along — it's enough for the instructor to demonstrate it once via screen share. If participants want to try it themselves, guide them to do so after the lecture ends, in their own environment (a workshop-only clone folder). Participant activity Everyone can hands-on practice checking the PROMOTION_CHECKLIST.md items and drafting the variant.json manifest.

Chapter 13: Capstone Exercise

This is the core exercise of Day 2. Combining the experience from Day 1 (using an existing variant) and Chapter 11 (creating a new variant), have participants directly design an application of a multi-agent team to their own real-world work.

Make clear up front that there is no single correct answer for this chapter. If participants feel stuck on "what should I build," it's effective for the instructor to demonstrate their own example first (e.g., "organizing weekly meeting-minutes action items") to show the flow of questions. If time is tight, rather than completing all 4 stages, it's fine to have everyone finish at least stages 1–2 (problem definition, guardrail decisions) and leave stages 3–4 as homework.

Participant activity Use the completion checklist to have each participant mark how far they got themselves, and during the capstone presentation time, have 1–2 participants briefly share their own workflow.

Wrap-up

A retrospective on the full two days. Tie together the experience of running an agent team with an existing variant on Day 1 with the experience of building and promoting a new variant on Day 2.

Chapter Comprehension Questions

Ask the questions below at the end of each chapter to check understanding. The goal isn't to get the right answer — it's to have participants explain it in their own words.

📅 Day 1 Comprehension Questions

Chapter 1 comprehension questions
  1. Of the 3 AI-usage stages (user, power user, designer), which do you currently feel closest to?
  2. What is the biggest difference between "using AI without a harness" and "using AI with a harness"?
  3. A single AI vs. a multi-agent team — what are the respective pros and cons in terms of context, verification, and parallelism?
Chapter 2 comprehension questions
  1. Explain the difference between prompt engineering and harness engineering in one sentence.
  2. What are the pros and cons of "predefining" subagents versus "dynamically generating" them?
  3. What determines whether you should use parallel fan-out versus a pipeline?
Chapter 3 comprehension questions
  1. Among the 3 risk tiers (read/write/destructive), where do a reviewer subagent and a file-deletion task each fall?
  2. What criteria determine whether to choose retry, rollback, or escalation?
Warm-up exercise comprehension questions
  1. Tetris was built well with vibe-coding, so why was Pac-Man hard to build with a single prompt?
  2. What's the difference between splitting Pac-Man into plan → develop → QA versus handing everything to one agent? (→ Chapter 2 §2 role conflict)
Chapter 4 §1/§2 comprehension questions
  1. Why might calling the writer and reviewer in parallel fail?
  2. What kind of incident could occur if you didn't know that hooks don't fire in the Claude App?
  3. In D-2, why did the composition of agents deployed differ by request even though it was the same subagent team?
Chapter 5 (light touch) comprehension questions
  1. What role does each of AGENTS.md, CONSTITUTION.md, CLAUDE.md, and GEMINI.md play?
  2. What gets created in a project folder when you scaffold a variant?
Chapter 6 · Using an existing variant comprehension questions
  1. Why is Phase 1.5 (cross-validation) needed in co-consult's 7-stage consulting workflow?
  2. Why must the "verification criteria" in the 4-stage design framework be numerically measurable?
  3. Why are the agent compositions of co-consult (report) and co-deck (presentation) so different?

📅 Day 2 Comprehension Questions

Chapter 8 comprehension questions
  1. Which layer (L0/L2) is the SSOT for CONSTITUTION.md and for context.md, respectively?
  2. Why does L1→L2 end with a "fork"?
  3. What problems arise if there's no model tiering by agent role?
Chapter 10 comprehension questions
  1. What's the difference between the LOCKED and MERGE classifications?
  2. Why should you run --dry-run first?
Common Reference/Chapter 7 comprehension questions
  1. What's the difference between teammateMode's in-process and tmux?
  2. What specific problems arise if the service-ticket model has no kanban? (Explain by choosing one of: visible queue, WIP limits, or prioritization)
Chapter 11 · New-variant creation comprehension questions
  1. Why is the Phase A prototype built in Projects/? Why not build it directly in templates/?
  2. If an agent definition's handoff_to and handoff_from don't line up, how does that show up in agent:verify?
Chapter 9 · Workflow design patterns comprehension questions
  1. Why is the "verify-then-apply" pattern essential in a code-review workflow?
  2. Among the 5 domain case studies, which pattern most closely resembles your own work?
Chapter 12 · New-variant promotion comprehension questions
  1. Between the build-from-scratch path and the promotion path, which one does the phrase "design precedes use" apply to?
  2. What does the reconcile stage handle, versus the manual _ORIGIN.md migration?
Chapter 13 · Capstone comprehension questions
  1. Which of the three limitations from §2 (context overload / role conflict / lack of parallelism) made a team necessary for the recurring task you chose?
  2. Why did you decide on guardrails before building the subagent?
