멀티 에이전트팀 하네스 엔지니어링 워크숍

프롬프트 엔지니어링에서 하네스 엔지니어링으로. Claude Code · Claude App · Antigravity · Antigravity CLI 네 도구를 넘나들며 멀티 에이전트팀을 직접 설계하고 운영하는 실전 워크숍.

이 워크숍은 무엇인가? "프롬프트 엔지니어링만으로는 부족하다"는 문제의식에서 출발해, AI 에이전트에게 좁은 역할을 부여하고 오케스트레이터가 조율하는 멀티 에이전트팀의 설계 방법론을 배우고, Claude Code·Claude App·Antigravity·Antigravity CLI 네 도구에서 실제로 서브에이전트를 만들고 파이프라인을 구축해보는 약 7시간 30분 실전 워크숍이다.

🎯학습 목표

이 워크숍을 수료한 참가자는 다음을 할 수 있다.

LO-1
설명하네스 엔지니어링의 정의를 프롬프트·컨텍스트 엔지니어링과 비교하여 설명하고, 단일 에이전트의 세 가지 구조적 한계(컨텍스트 과부하·역할 충돌·병렬성 부재)를 든다.
LO-2
설계오케스트레이터·스페셜리스트·핸드오프 계약으로 이루어진 멀티 에이전트팀을 구성하고, Anthropic의 다섯 가지 오케스트레이션 패턴(체이닝·라우팅·병렬화·오케스트레이터-워커·평가자-최적화) 중 상황에 맞는 패턴을 선택한다.
LO-3
적용위험 작업 3단계 분류(읽기/쓰기/파괴적)와 최소 권한 원칙, 사람 승인 게이트, 격리, 감사 로그를 포함한 가드레일 체계를 서브에이전트에 설계한다.
LO-4
실행Claude Code·Claude App·Antigravity·Antigravity CLI 네 도구에서 서브에이전트를 정의하고 호출하며, 순차 파이프라인과 병렬 실행을 직접 구현한다.
LO-5
비교네 도구의 서브에이전트 정의 방식, teammateMode, 훅 지원, Agent Manager 차이를 실습 경험에 바탕으로 설명한다.
LO-6
판단기업 내 멀티 에이전트팀 운영의 두 가지 모델(서비스 티켓 모델 vs 개발 도구+variant 모델)을 비교하고, 자신의 조직에 적합한 모델과 그 이유를 설명한다.
LO-7
구현ai-workspace-standards 저장소의 L0→L1→L2 계층 구조와 핵심 파일 4종을 이해하고, 새로운 variant(co-marketing)를 프로토타입으로 만들어 승격 체크리스트를 작성한다.
LO-8
종합본인의 실제 반복 작업을 문제로 재정의하여, 문제 정의→가드레일 설계→서브에이전트 구현→검증·회고의 전체 사이클을 직접 설계하고 실행한다.

👥대상자

다음에 해당하는 분이라면 이 워크숍에서 가장 큰 성과를 얻을 수 있다.

  • AI 코딩 도구를 일상적으로 사용하는 소프트웨어 엔지니어. Claude Code, GitHub Copilot, Cursor 등을 쓰고 있지만 단일 프롬프트로 한계를 느끼는 분
  • AI 에이전트 도입을 고민하는 팀 리더·Tech Lead. 팀 전체에 어떻게 에이전트를 배치하고, 안전하게 운영할지 설계 책임이 있는 분
  • DevOps·플랫폼 엔지니어. 조직 단위 AI 워크스페이스 구축과 variant 관리가 업무에 포함된 분
  • AI 도구 교육을 담당하는 강사·컨설턴트. Claude Code와 Antigravity를 교육 자료로 다루는 분
참고: 이 워크숍은 Claude Code와 Antigravity 계열 도구에 초점을 맞추고 있으며, 네 가지 도구의 설치와 CLI 기본 조작에 익숙한 것을 전제로 한다. CLI 경험이 없는 초보자는 설치 가이드의 사전 준비 단계를 충분히 연습하고 오는 것을 권장한다.

✅워크숍 후 얻게 되는 것

  • 멀티 에이전트팀 설계 역량. 단일 에이전트에서 팀 구조로 전환하는 설계 방법론을 체득하여, 실제 업무에 즉시 적용할 수 있다.
  • 네 도구 실전 경험. Claude Code·Claude App·Antigravity·Antigravity CLI에서 같은 시나리오를 직접 실행해보며 각 도구의 장단점을 체감한다.
  • 가드레일 설계 체크리스트. 위험 작업 분류, 최소 권한, 승인 게이트를 포함한 실무용 가드레일 설계 템플릿을 가지고 간다.
  • variant 프로토타입. ai-workspace-standards 기반으로 co-marketing 프로토타입을 직접 만들어, 조직 내 variant 도입의 구체적인 시작점을 확보한다.
  • 운영 모델 판단 기준. 서비스 티켓 모델과 variant 모델의 장단점을 비교하여, 자신의 조직에 맞는 운영 전략을 세울 수 있다.
  • 나만의 워크플로우. 캡스톤 실습에서 본인의 실제 반복 작업을 멀티 에이전트팀 문제로 재정의하고, 문제 정의부터 검증까지 전체 사이클을 경험한다.
