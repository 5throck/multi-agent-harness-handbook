This page collects the warning and tip boxes scattered across each chapter, ordered by how often they come up during hands-on practice. Look up your symptom first, then follow the "Details" link if you need the original explanation.

Setup & Environment

I ran the Chapter 4 §2 setup script, but some items show ❌.

Regardless of OS, setup-common.ts re-validates the checklist. On Windows, first confirm you ran the script with administrator privileges; on macOS/Linux, check that you didn't forget optional flags like --wezterm/--docker. Installation logs are saved to %USERPROFILE%\workshop-setup-logs\ on Windows.

Details: Chapter 4 §2-A Setup
I'm trying to install the Antigravity CLI globally via npm, but it doesn't work.

As of 2026-06-18, the Gemini CLI service was retired and replaced by the Antigravity CLI (agy). agy is a single compiled binary that requires no separate runtime install — just unzip it and add it to your PATH. Instructions for npm installation are outdated.

Subagent Execution

I had the writer subagent write a file, but the reviewer says "the file doesn't exist."

You most likely invoked writer and reviewer at the same time (in parallel) within a single message. The reviewer attempted its review before the writer had even created the file, causing the failure. For tasks with dependencies, explicitly request sequential execution, e.g. "run these two steps in order."

Details: Chapter 4 §2-A D-1
After saving a file in the Claude App, the audit script doesn't run automatically.

This is expected. Hooks such as PostToolUse/TeammateIdle/TaskCompleted do not fire in the Claude App (Desktop) — they only run automatically in Claude Code (CLI) sessions. Teams using the Desktop App must run whatever the hook would have done manually after the session ends. Missing this can lead to incidents where you assume "the audit ran" when it actually didn't.

Details: Common Reference §1, Chapter 3 §Audit Logging and Observability
I asked the reviewer only to review, but it modified the file anyway.

Check whether the tools field includes Write. For read-only roles like reviewer, grant only the minimum permissions needed from the start — e.g. tools: Read, Grep — so that even if the prompt is interpreted ambiguously, the agent is structurally incapable of making edits.

Details: Chapter 3 §Principle of Least Privilege
The agent keeps repeating the same task and never finishes.

This is a spin state in an evaluator-optimizer loop, where the evaluator keeps judging the output "insufficient" and the optimizer keeps retrying. The fix: set a maximum number of iterations up front, and automatically terminate if the criteria aren't met within that limit. Rather than solving this at the individual pattern level as described in Chapter 2 §Orchestration Patterns, it's easier to prevent by structuring the entire workflow as an Execution Plan, as covered in Chapter 9.

Details: Chapter 2 §Orchestration Patterns, Chapter 9 · Workflow Design Patterns
A subagent calls a tool, fails, and then just stops.

There are two common causes. First, a network timeout — the API call or MCP server connection was briefly interrupted. In Claude Code, check session state with the /cost command; in Antigravity, check agent status in Agent Manager, then retry. Second, an unconnected MCP server — verify whether the local MCP server is running using claude mcp list (Claude Code) or the equivalent server-status command for your platform. For transient errors, see the timeout troubleshooting under Cost & Performance.

Details: FAQ · Cost & Performance, Chapter 3 §Audit Logging and Observability
How do I monitor and audit an agent's destructive actions or file-modification history in an enterprise environment?

The agent's terminal command execution and tool-use history must be collected in real time as system audit logs. To do this, use an audit daemon or shell-hooking technique inside the sandbox environment to forward execution records to a remote log server. Additionally, important destructive operations (e.g., dropping a database, destroying a secret key) must enforce a Human Approval Gate, structurally isolating them so they cannot execute without final sign-off from a system administrator or responsible party.

Parallel & Team Collaboration

In Antigravity Agent Manager, two agents are editing different files, but the results get mixed up.

You most likely assigned both agents to the same workspace (the same folder). Sharing a codebase can cause "cognitive overlap," where one agent mistakes another agent's working context for its own. Always separate agents by workspace or git worktree.

Details: Common Reference §2, Chapter 3 §Isolation
I turned on Agent Teams in the Claude App, but I don't see a tmux option.

This is expected. The Desktop App only supports teammateMode: in-process; tmux split-pane mode is only additionally supported in Claude Code (CLI).

Details: Common Reference §1
How should an agent handle needing final human approval or intermediate input while performing a task?

Use the Human-in-the-Loop (HITL) pattern. As the orchestrator progresses through a task, it pauses at a critical decision point or an operation requiring authorization (e.g., deployment, code merge), preserves state, and requests feedback from the user. The agent must structure the context of the question and the information it needs when relaying it, and if approval is denied or a revision is requested, the agent team should roll back to the prior state, incorporate the feedback, and control the flow to attempt the task again.

Workspace / Variant

I ran new-project.ts, but scaffolding was rejected or paths look wrong.

You likely ran it from inside a project folder rather than the workspace root. ai-workspace-standards must be cloned directly to the workspace root itself (C:\git on Windows, ~/git on macOS/Linux); placing it as a submodule inside another project folder breaks path resolution.

Details: Chapter 6 §1
The "Documented agents" count from bun run agent:verify doesn't match what I expect.

The AGENTS.md table and the actual list of agents/*.md files are out of sync. Check whether workspace-only agents (auditor, lifecycle-manager, etc.) are still lingering in the table, or whether a newly created agent is missing from it.

Details: Chapter 6 D-1
I'm not sure whether it's safe to run the promotion pipeline (l2-to-variant-pipeline.ts).

This command leaves traces in actual git history and is hard to undo. In a lecture or workshop setting, it's sufficient for the instructor to demonstrate it once rather than have every participant run it. See the Lecture Guide for details.

How do I use ai-workspace-standards in an actual production environment (Docker/Kubernetes)?

Currently, ai-workspace-standards only works by integrating directly with local AI coding tools (Claude Code/App, Antigravity, etc.). Supporting per-user agent teams in a Docker or Kubernetes environment in conjunction with a frontend service like Open WebUI is under discussion as a future enhancement, but has not yet been implemented. Can I aggregate memory/MEMORY.md from multiple projects into one place?

Currently, each project maintains an independent memory/MEMORY.md, and a workspace-level feature to consolidate and organize memory is planned as a future enhancement. This is intended to address the difficulty of spotting overall trends across individual memory files as the number of projects grows. Details: Chapter 8 §5 Advanced Roadmap

If a variant template is updated, does it automatically propagate to existing projects?

No. ai-workspace-standards uses a Fork Model. Changes to L1 (the common template) do not automatically propagate to already-scaffolded L2 projects. This is by design to preserve intentional divergence, but it creates the operational burden of manually applying security patches or bug fixes. A consent-based opt-in model and an asynchronous deployment pipeline are being discussed as future enhancements. Details: Chapter 8 §5 Advanced Roadmap

Creating a new variant fails agent:verify validation or validate-templates.

The most common cause is a failure to maintain **symmetry** between agent declarations. For example, if the researcher agent's handoff_to lists writer, then the writer agent's handoff_from must also define researcher as a matching pair. Also check for JSON syntax errors in variant.json or missing required fields (such as the agent list).

Details: Chapter 11 §Creating a New Variant
How do you verify that an agent team you designed and built yourself consistently performs at the intended quality in production?

You need to build an agent test bench to evaluate normal operation. First, define representative input requirements and expected outputs (a gold dataset). Second, run automated regression tests against that dataset every time you modify a prompt or agent rule. Third, for natural-language output that's hard to score quantitatively, set up a high-performing model as an LLM-as-a-Judge, evaluating against a pre-agreed quantitative score.

Cost & Performance

How much does running a multi-agent team cost in tokens?

Cost is determined by the number of agents × each agent's model tier × execution frequency. The model tiering (tier: high/medium/low) described in Chapter 5 is the key cost-control mechanism. For example, running all 10 agents of the co-consult variant on high can consume tens of thousands of tokens in a single session, but lowering the actual implementation roles to medium/low can cut costs by 40-60%. Claude Code's /cost command lets you check cumulative token usage for the current session in real time.

Details: Chapter 5 §Templates and the 7 Variants
Agent responses suddenly become slow or time out.

When the context window fills up, model processing speed drops sharply. Claude Code lets you compress conversation length with the /compact command. Also, when running multiple agents in parallel, API requests can queue up and cause delays — in this case, switch to sequential execution or lower your WIP limit to reduce the number of concurrent runs.

Details: Common Reference §1
Running a multi-agent loop or long workflow costs too many API tokens. How do I optimize it?

First, assign distinct **model tiers** per agent role. Use medium/low models for simple file writes or test execution, and reserve high-tier models for overall flow control or comprehensive design — this alone can cut costs by 40% or more. Second, keep the conversation context stable so prompt caching works — avoid changing the system prompt and maintain a continuous session. Third, set clear hard limits, such as Max Turns, so the conversation between the orchestrator and specialists doesn't spin indefinitely.

Platform-Specific Issues

Running new-project.ts on Windows produces a path-separator error.

ai-workspace-standards scripts use the path module in the Node.js/Bun runtime to handle platform-specific paths, so they're compatible with both Windows and macOS/Linux by default. However, issues can arise if a PowerShell-style path (C:\Users\...) gets mixed in when you run git clone directly inside a Claude Code session instead of Git Bash. In Claude Code, always use /-style paths or Git Bash formatting, and it's recommended to clone the workspace root to a short, shallow location like C:\git.
