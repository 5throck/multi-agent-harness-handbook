As of 2026-06-18, Google discontinued the Gemini CLI service for individual (Free/Pro/Ultra) users, and Antigravity CLI (agy) took its place. This handbook was originally written around the Gemini command-line interface (CLI) before that date, but has since been fully updated to the Antigravity command-line interface (CLI) as of the current date. If you encounter the gemini command or a .gemini/agents/ path in older material, treat it as referring to a since-superseded version.

1. Claude App (Desktop) vs Claude Code (CLI)

This handbook treats Claude App (Desktop) as its default hands-on tool. Because the flow of creating and invoking subagents through the GUI is the same, you can follow the content below just as easily using Claude Code (CLI). Both tools run on the same Claude engine, but since their execution environments differ, some of the mechanisms that make up a multi-agent harness only work in the CLI. The biggest differences are in automation hooks and Agent Teams execution modes.

Claude App (or Claude Code) — .claude/agents/*.md Subagents

Claude App defines custom subagents via .claude/agents/*.md files and invokes them with the Agent/Task tool, exactly as Claude Code does. However, the PostToolUse/TeammateIdle/TaskCompleted hooks defined in .claude/settings.json do not fire in Claude App. They only auto-fire on events like file saves or task completion within a CLI session. Teams using Claude App must manually run whatever the hook would have done (e.g., running an audit script) after the session ends.

Let's look at a concrete accident this difference can cause. Suppose a team has set up a PostToolUse hook in the CLI so that "every time a file is saved, bun scripts/audit.ts runs automatically to catch secret leaks or rule violations." When the same team member makes the identical edit in Claude App (Desktop), the hook never fires at all — yet on screen it looks exactly like the CLI: "the file was saved," with no visible sign the hook didn't run. If that state carries through to a commit, an issue the hook was supposed to catch quietly slips through. The real danger isn't "we forgot to run the audit" — it's "we thought the audit ran, but it didn't."

teammateMode — Claude App Supports Only in-process, Claude Code Also Supports tmux

Agent Teams, an experimental feature, launches multiple Claude instances in parallel. Claude App can only use the in-process mode, which parallelizes within the same process, and some keyboard navigation (e.g., Shift+Down) is restricted. The tmux mode, which uses a terminal split-pane, is additionally supported only in Claude Code (CLI).

FeatureClaude App (Desktop)Claude Code (CLI)
Invoking subagents via Agent/Task tool Supported Supported
Custom subagents via .claude/agents/*.md Supported Supported
PostToolUse / TeammateIdle / TaskCompleted hooks Unsupported (manual execution required) Supported (fires automatically)

To use Agent Teams in the Desktop App, you must explicitly set teammateMode to in-process. Your team's rules should also reflect the fact that hooks don't fire. Be careful not to mistake a silently-skipped automated audit for a "pass."

2. Antigravity (Desktop) vs Antigravity CLI (agy)

On the Google side too, this handbook treats Antigravity (Desktop) as its default hands-on tool. It's a standalone desktop app designed from the ground up to supervise multiple agents across multiple workspaces simultaneously. Antigravity CLI (agy) is the CLI face of the same platform, structured so that a single terminal session receives a goal and breaks it into subtasks on its own. Both tools also share the same configuration folder (antigravity-cli/ under ~/.gemini).

Workspaces · Inbox · Playground — Antigravity (Desktop) Agent Manager

Antigravity's Agent Manager is a top-level interface that supervises multiple workspaces from a single screen. The principle is to assign one agent per workspace, and messages needing approval or feedback collect in the Inbox. The Playground is a space to try things out lightly before committing to a formal workspace. When launching an agent, you choose between Local Mode, which works directly in the active folder, and New Worktree Mode, which works in an isolated git worktree.

While you can assign multiple agents to the same workspace (same folder) at once, there are real reports of different agents confusing and mixing up the codebase's context ("cognitive overlap"). It's safer to separate parallel work by workspace or by git worktree.

Antigravity CLI (agy) Slash Commands — /goal, /agent, /agents

Unlike Claude Code/App, which predefines subagents in files such as .claude/agents/*.md, the Antigravity command-line interface (CLI) works by having the orchestrator agent decide on its own how to split a goal into tasks once given one, creating subagents on the fly and running them in parallel. Each subagent has its own independent context window, preventing the main session's context from getting diluted.

To use an analogy: Claude Code/App works with "a pre-hired roster of full-time team members," while the Antigravity command-line interface (CLI) is closer to "hiring a freelancer on the spot for whatever job comes up." The former makes it easy to control permissions because you always know who your team members are; the latter offers the flexibility of sourcing new talent each time, but you need a way to check after the fact "who exactly got brought in this time." That's what /agents below is for.

One thing worth noting here: the ability to dynamically create subagents is not unique to Antigravity CLI. Claude Code/App's Agent(Task) tool also spawns subagents on the fly from natural-language requests alone, judging what task is needed at the time (Chapter 1 §3b), and Antigravity (Desktop)'s Agent Manager likewise lets you give a goal in natural language without a predefined file (§1). The /goal·/agent·/agents introduced in this section are simply syntax for handling that same dynamic orchestration capability via slash commands on the CLI screen. Claude Code/App does the same thing through chat sentences instead of this syntax, and Antigravity Desktop does the same thing through GUI buttons (Inbox approvals, Playground, etc.) instead of this syntax.

Below are four CLI-specific slash commands you'll frequently use when actually working with agy. Read the table below not as "a list of features unique to Antigravity CLI" but as "the names given to the same capability on the command-line interface (CLI)."

CommandAction
agyStarts a TUI session.
/goal <goal>Runs a goal to completion without intermediate confirmation. The orchestrator splits subtasks on its own.
/agent <task>Delegates a long-running task to a background agent and lets you move on to another prompt without waiting for completion.
/agentsShows a list of subagents created so far, with their running/completed/blocked status.

Here is the actual flow of starting a session and running a goal.

agy

Inside the session (interactive):

/goal Add a summary to every text file in this folder

If, as in CI/automation, you want a single run with no conversation (non-interactive):

agy -p "Draft a commit message"

On the CLI side, too, it's safer to keep sessions separate. If one agy session is refactoring the API layer while another writes tests, they won't conflict as long as they don't touch the same files.

There's one criterion that separates the two rows above. Is the subagent a "one-shot call that only returns a result once delegated" or "a teammate-like entity that stays alive"? Claude Code/App's single Agent(Task) call and Antigravity command-line interface (CLI)'s dynamic subagents are closer to the former. Once spawned, they handle everything alone through to the end and return only a summary, so there's no way to give new instructions midway or have them converse with another subagent. Claude Code/App's Agent Teams, by contrast, are the latter — each teammate can send messages directly to another via SendMessage (already covered in the P-1 exercise in Chapter 2 §1). Antigravity (Desktop) doesn't support agent-to-agent communication at all, but instead lets a human converse individually with each in-progress agent through the Inbox — this is what sets it apart from the other three tools.

4. Which Tool to Choose for This Handbook

This handbook guides you using Claude App and Antigravity (Desktop) as its default hands-on tools, because starting with a GUI has a lower barrier to entry. Chapter 2 likewise shows these two Desktop tools first, then immediately follows with the Claude Code (CLI)·Antigravity CLI (agy) procedures side by side. You can pick just one tool and follow along, or compare and practice with all four together. The following criteria make the choice easier.

  1. If you just want to get a feel for it via GUI first. Start with Claude App or Antigravity (Desktop). This is the handbook's default path.
  2. If you want to supervise several different projects at once. Antigravity (Desktop)'s Agent Manager (isolation per workspace) is designed exactly for this scenario.
  3. If you need hook-based automation (audit/QA gates). Move to Claude Code (CLI). Claude App and the Antigravity family have no corresponding automatic hooks.
  4. If you want an orchestrator to split things up on its own without defining subagents one by one. Antigravity CLI (agy) dynamically creates subagents just from a goal. In exchange, fine-grained control is relatively limited.

💡 Key takeaway: All four tools share the same basic harness-engineering structure: "give subagents a narrow role and let an orchestrator coordinate them." What changes is which execution environment (CLI/GUI, hooks or not, whether subagents are predefined or created on the fly) that structure is laid on top of.

5. AGENTS.md — Tying the Four Tools Together

The hooks, parallel execution, and subagent-definition methods compared so far differ from tool to tool, but the repository-root configuration can still be unified into one. AGENTS.md is an open standard created in August 2025 by OpenAI together with Google, Cursor, and Factory, and donated to the Linux Foundation's Agentic AI Foundation in December 2025. It's supported by more than 30 tools, including Codex, Cursor, GitHub Copilot, and Google Jules, as well as Claude Code (via import) and the Antigravity family.

Placing a single AGENTS.md at the repository root lets all four tools read it at session start and share common rules such as build/test commands, code style, and off-limits areas. Tool-specific behavior (hooks, teammateMode, where subagents are defined, etc.) is still recorded separately, as compared in the table above, in tool-specific files such as CLAUDE.md and GEMINI.md. AGENTS.md handles the common denominator, and tool-specific files handle the differences.

FileRoleTools that read it
AGENTS.md Tool-neutral common rules (build/test commands, code style, off-limits areas) Claude Code (import) · Antigravity family · Codex · Cursor · Copilot, and 30+ others
CLAUDE.md Claude Code/App-specific detailed behavior (hooks, slash commands, agent-dispatch rules) Claude Code, Claude App
GEMINI.md Antigravity CLI/Desktop-specific detailed behavior Antigravity CLI, Antigravity (Desktop)

💡 Key takeaway: The ai-workspace-standards repository covered in Chapter 5 §1 implements exactly this structure. The root AGENTS.md serves as the single source of truth (SSOT) for the agent roster, and the .agents/ folder sits alongside .claude/·.gemini/, holding tool-agnostic commands and skills.
