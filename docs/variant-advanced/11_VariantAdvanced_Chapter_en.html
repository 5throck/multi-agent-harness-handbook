Chapter 11

Creating a New Variant vs. Promoting an Existing Project to a Variant

In Chapter 6 §2, we practiced scaffolding a brand-new variant called co-marketing entirely from scratch. But ai-workspace-standards has a second path that is conceptually quite different: "promoting" a project that is already running well — one that was never intended to become a template — into an official variant later on. This chapter covers how the two paths differ conceptually, and which situations call for which path.

Source: ai-workspace-standards (GitHub)

What This Chapter Covers
  • The fundamental difference between "building from scratch" and "promoting an existing project"
  • The concept of the build-from-scratch path, which flows from Phase A (create-variant) to Phase B (promote-variant)
  • The concept of promoting an already-existing project via the promote-variant skill and l2-to-variant-pipeline.ts
  • Criteria for judging which of the two paths fits your organization's situation

The Difference Between the Two Paths

There is more than one way to add a new variant to the workspace. ai-workspace-standards recognizes two distinct starting points. One is the path of "designing from the very beginning, with the intent that it will become a template, before any project exists yet." The other is the path where "a project that was never meant to be a template gets promoted to one only after enough time has passed that multiple teams have been copying it by hand because it proved itself."

The difference between the two is not merely "different commands." In the former, design comes before use; in the latter, use (and validation) comes before templating. The reason the workspace supports both paths is that the real-world situation of building a new domain's agent team tends to converge on one of these two shapes.

Building from scratch puts design ahead of use; promotion lets use, validated through practice, precede design. — the fundamental ordering difference between the two paths

The Build-From-Scratch Path

This path, which starts from a greenfield state, is divided into two stages. First, using the create-variant skill and bun scripts/create-l2-scaffold.ts <variant-name> --domain <type>, an independent "Phase A prototype" is created under Projects/<variant-name>/. This stage is part of the structure the workspace documentation calls the "L1-L2 Fork Model"; since it doesn't yet belong to the official templates/ directory, you're free to experiment and change direction. The co-marketing prototype built in the Chapter 11 exercise (D-1) is exactly this Phase A stage. This exercise refers back to the Chapter 6 §2 exercise (D-1).

Once the prototype has matured sufficiently, it goes through a "Phase B promotion" via the promote-variant skill to become an official templates/co-<name>/. In other words, even the build-from-scratch path ultimately passes through a promotion stage. What sets it apart from the path in the next section is that the prototype itself was designed from the outset with the premise that it would become a template.

The Path of Promoting an Existing Project

The second path has an entirely different starting point. A team builds a project to solve a specific domain problem, without modeling it on any other template and without reuse in mind from the start. But over time, the project's setup works so well that other teams inside the company start copying its files by hand to build similar things. The purpose of promotion is to turn this kind of "already-validated, but not born as a template" project into an official variant.

This path ultimately uses the same promote-variant skill and the same bun scripts/l2-to-variant-pipeline.ts --source Projects/<name> --variant co-<name> --variantType <type> pipeline, so the mechanism is identical to Phase B of the build-from-scratch path. Before promotion, there is a gate requiring all the conditions in PROMOTION_CHECKLIST.md to be satisfied (whether bun run agent:verify, bun scripts/validate-skills.ts, and bun scripts/audit.ts pass, among others). After the pipeline runs, this is followed by checking the inherits_common version in variant.json, a "reconcile" step that removes parts duplicated with common files, and validation of platform parity between .claude/ and .gemini/. Domain-specific directories that the pipeline cannot handle automatically (such as workflows/ or regulations/) must be moved manually as specified in _ORIGIN.md.

The mechanical procedure is the same for both paths, but "what is being validated" differs. On the promotion path, the fact that it was already being used in production is itself the basis for quality, and the pipeline's role is to organize that into a reusable form.

Let's unpack what "reconcile" and "manual migration via _ORIGIN.md" actually do with a concrete example. Suppose the finance team's Projects/budget-review/ is being promoted. This project mixes two kinds of files: ① common files copied as-is from templates/common/, such as PM definitions or session checklists, and ② assets unique to this project, such as internal accounting-regulation documents inside the regulations/ folder. The reconcile step reverts files matching ① from "copies" back into "references that inherit the common file," so that when templates/common/ is later updated, this variant is updated along with it. On the other hand, the pipeline has no way of knowing about ②, so it is left in the original project's _ORIGIN.md under a "Manual Phase B Steps" list — something like "manually copy the regulations/ folder to the new variant" — and the person handling the promotion moves each item over by hand, following the checklist.

Which Path to Choose, and When

Building From Scratch (Phase A → B)

  • A greenfield domain with no existing project to reference yet
  • Design precedes use. Built from the start with reuse in mind
  • Free to change direction during the Phase A stage
  • Example: introducing an agent team into a completely new area of work for the first time

Promoting an Existing Project (Phase B Only)

  • A project already built and validated in production by a team
  • Use and validation precede design. Already proven in practice
  • Must pass the PROMOTION_CHECKLIST.md gate
  • Example: formalizing an internal standard project that multiple teams have been copying by hand

In the end, the criterion for choosing comes down to one thing. If it's a new domain with no precedent to refer to at all, the design-from-scratch path is the natural choice. Conversely, if there's already a project within the organization that has been validated at least once and that everyone is copying, it's safer and more efficient to respect that validated result and formalize it through the promotion path. Since both paths ultimately converge on the same templates/co-<name>/ shape, this is a question of "which mountain to climb," not "which mountain you end up on."
