Chapter 8 §5

Advanced Roadmap

What §1 through §4 covered so far is the current implementation of ai-workspace-standards. This section covers the opposite — five directions the repository is heading in that have not yet been implemented. They split into three infrastructure-level directions (container integration, memory aggregation, variant deployment automation) and two governance-level directions (boilerplate strategy, model tier strategy).

What This Section Covers
  • The need for service integration in containerized (Docker/Kubernetes) environments
  • Workspace-level aggregation of independent MEMORY.md files
  • Ways to selectively propagate security patches and bug fixes while preserving the benefits of the fork model
  • The limits of the LOCKED/MERGE/PRESERVE three-way classification being a fixed taxonomy, and directions for per-organization customization
  • Directions for adjusting statically assigned model tiers based on real organizational operating data

Container-Based Service Integration

Today, a project scaffolded from any given variant only operates by connecting directly, inside a local workspace, to AI coding tools like Claude Code/App or Antigravity. But in real enterprise environments, projects often need to be deployed as independent services on a per-project basis.

The core of this advancement is that a scaffolded project should be able to run in a Docker or Kubernetes environment, **connected to a frontend service such as Open WebUI so it can be supported on a per-user basis**. In that form, each user invokes their assigned agent team through a browser interface, reviews results, and retains context across sessions.

Memory Aggregation and Curation

Today, each L2 project maintains its own independent memory/MEMORY.md file (covered in Chapter 5 §4). This file records cross-session context for that project, but as the number of projects grows, memory files become scattered and it becomes difficult to see workspace-wide insights.

The second axis of advancement is the need for a capability to **aggregate and curate** memory files that are otherwise scattered across individual projects. For example, it should become possible to run workspace-level trend analyses such as "what error patterns did agent teams across all projects run into most often last week" or "which role combinations were used most frequently."

Evolution of Variant Deployment

Today, variant deployment is centered on local scaffolding via new-project.ts, following a fork model in which template updates are not automatically propagated to L2. This approach has the advantage of preserving intentional differences, but from an operational standpoint it creates a burden: template bug fixes or security patches must be manually applied to each project one by one.

Two directions for advancement are being discussed: (a) a **consent-based receiving model** that lets only security patches or critical bug fixes propagate selectively to L2 while preserving the benefits of the fork model, and (b) an **asynchronous deployment pipeline** in which variants are managed in a central registry and new-project.ts pulls the latest version over the network.

If the three items above are advancement directions at the infrastructure level, the two below are advancement directions at the governance level — covering what limitations the boilerplate strategy and model tier strategy from §1 and §4 currently have, and which directions they could be refined toward.

Advancing the Boilerplate Strategy

In §1 and §4 we covered the LOCKED·MERGE·PRESERVE·SYNC_IF_NEWER·PRUNE five-way classification as the criterion for deciding "what to lock and what to leave open." Right now this classification is hardcoded per file path inside the upgrade-project.ts code — whether a given file is LOCKED or MERGE only changes if the script itself is edited. This works fine when the repository manages few file types, but as organization-specific exceptions accumulate — "we also want to treat this file as LOCKED" or "we want to fully open this MERGE file to PRESERVE for our organization" — it increasingly forces organizations to fork the script itself.

The direction for advancement is to move this classification out of code and into a declarative manifest. For example, one option under discussion is placing a file such as boilerplate-policy.json at the workspace root, letting each organization redefine its own LOCKED/MERGE/PRESERVE boundaries at the file-pattern level. With this in place, upgrade-project.ts would read this manifest instead of relying on a hardcoded classification table, and organizations could adjust the boundaries to fit their own security and governance standards without touching the script.

Advancing the Model Tier Strategy

In §1 and §4 we saw that tier assignment is statically fixed in agents/*.md frontmatter, and that changes at L0 only propagate via upgrade-project.ts's SYNC_IF_NEWER. The limitation of this approach is that tier assignment is tied only to "role," and does not reflect the actual difficulty of the work or the organization's cost budget. For example, the code-writer role is always medium, but in practice the complexity of the codebase a code-writer works with varies widely from project to project. In the current structure, reflecting that difference requires a person to manually edit each project's agents/*.md by hand.

Two directions for advancement are being discussed: (a) operational-data-driven tier recommendations, based on session logs or /cost data, that surface guidance such as "this role has repeatedly triggered retries over the last N sessions → recommend raising it to high," and (b) tier budget cap governance at the workspace or organization level, such as "this month's budget only allows high-tier assignment for N roles." Both directions preserve the current "static assignment + manual override" model while layering observability and organizational policy on top of it.

This handbook is written based on the current implementation state of ai-workspace-standards (the main branch, as of July 2026). The five items mentioned in this section are all planned items on the repository's roadmap, and are not implemented in the current codebase. The actual timing and form of implementation may differ.
