Chapter 8 §1

Deployment and SSOT

This section covers the L0→L1→L2 three-tier hierarchy of ai-workspace-standards, the design rationale behind the fork model, and the per-file SSOT (Single Source of Truth) flow. The source always flows top-down, and there is no automatic synchronization that flows back upward.

What This Section Covers
  • The L0→L1→L2 three-tier principle
  • The fork model and why automatic back-synchronization is not possible
  • The per-file SSOT flow

The L0 → L1 → L2 Hierarchy (3-Tier)

The CONSTITUTION.md, templates/, variants, and new-project.ts covered so far all operate on a single overarching principle. The repository is designed as a three-tier hierarchy — L0 (workspace root) → L1 (common templates) → L2 (variant/project) — and the single source of truth always flows top-down. There is no automatic synchronization flowing back upward. The boilerplate discussed in the previous section corresponds to L1 in this three-tier model. This boilerplate is published from L0 and delivered as a one-time snapshot at the moment an L2 is born.

What Each of the Three Tiers Is Responsible For

L0 (workspace root) is the single source of truth for files such as agents/pm.md, CLAUDE.md, GEMINI.md, and scripts/*.ts. Editing happens here, and only here. L1 (templates/common/) is a snapshot of the common infrastructure published from L0, serving purely as the baseline starting point when a new L2 variant is created. L2 takes two forms. One is templates/co-<name>/, which has solidified into an official template; the other is Projects/<name>/, a living project actually created via new-project.ts. In both cases, once forked off per variant, each evolves independently.

L0 — Workspace root (C:\git\, single source of truth)
  │  L0→L1: publish-to-template.ts (automatic and ongoing, on every dev-sync)
  ▼
L1 — templates/common/ (common template snapshot)
  │  L1→L2: create-l2-scaffold.ts (once, at variant creation time)
  ▼
L2 — templates/co-<name>/ or Projects/co-<name>/
     └─ Evolves independently after forking, no automatic back-sync
     └─ Run l2-to-variant-pipeline.ts explicitly to promote to an official template

Why L1 → L2 Ends in a "Fork"

Early on, an approach was tried where every L1 change was automatically propagated to all existing L2 variants, but three problems kept recurring. Changes in L1 silently overwrote parts a variant had deliberately customized differently; every time a new variant was added, the hardcoded variant list in the propagation script had to be manually updated; and there was no way to distinguish "an intentional difference" from "a difference caused by a missed sync." So the Fork Model is used instead. L1 delivers its content exactly once, at the moment of scaffolding a variant, and from then on L2 evolves completely independently. Reflecting an L2 change back into the official template requires a human to explicitly run l2-to-variant-pipeline.ts rather than any automatic process (Chapter 11 covers this promotion procedure directly). Meanwhile, publish-to-template.ts --check-drift periodically reports the differences between L1 and each L2, but this report is read-only and never automatically overwrites values. The reverse direction — the procedure for actually applying L1's latest changes to an already-created L2 project — is covered in Chapter 10 · Upgrading an L2 Project.

Per-File SSOT — What Flows Where, From Where

File Source (SSOT) Sync Method
CLAUDE.md / GEMINI.mdWorkspace rootManual propagation + validate-templates.ts
agents/*.mdWorkspace root agents/bun run agent:verify
AGENTS.mdWorkspace rootbun run agent:verify
variant.jsontemplates/co-<name>/ itselfThat variant is itself the source

agents/pm.md is the case that best illustrates the L0→L1 relationship. L0's agents/pm.md is the complete source, while L1's templates/common/agents/pm.md is a "pure extension" file that references L0 via the extends: ../../../agents/pm.md frontmatter while stripping out workspace-only sections (e.g., Auto-Mode, lifecycle-state fields). L2's templates/co-*/agents/pm.md inherits from L1 but does not copy the content verbatim — instead it regenerates what it needs from scratch based on variant-specific overrides ("Layout Reconstruction"). That's why even when L0's agents/pm.md exceeds 300 lines, L2's pm.md stays around 50–100 lines.

L0→L1→L2 in one sentence: the source is edited only at the top. The layers below are either snapshots of the source or copies taken at scaffolding time. The flow always moves top-down, never the reverse. A representative mistake that violates this principle is "editing an agent file in a new project (L2) without reflecting the change back into the workspace root (L0)" — since no reverse flow from L2 back to L0 exists in the first place, that edit is valid only within that project and is never reflected anywhere else.

Boilerplate Strategy — What to Lock, and What to Leave Open

Chapter 5 defined boilerplate as "a pre-assembled starting point so you don't have to rewrite the same things for every project." From an architectural standpoint, this definition immediately raises a practical question — how much of what L1 hands down to L2 should projects be free to modify? This repository answers that question differently on a per-file basis. The five categories of upgrade-project.ts covered in §5 (LOCKED, MERGE, PRESERVE, SYNC_IF_NEWER, PRUNE) are essentially this strategy nailed down in code.

  • What gets locked as LOCKED (.githooks/*, .gitattributes, .gitleaks.toml) — files corresponding to the minimum security baseline of the Chapter 3 guardrails. There's no reason for these to differ between projects, and differing would itself lead to a security incident, so no room for customization is given at all.
  • What's half-opened via MERGE (CLAUDE.md/GEMINI.md, .gitignore, agents/pm.md) — only the sections wrapped in WORKSPACE-MANAGED markers continue to be managed by L1, while everything outside the markers is free for the project to write as it sees fit. This is the compromise that satisfies "common rules from above, project context from below" within a single file at the same time.
  • What's fully opened via PRESERVE (README.md, docs/context.md, src/) — L1 only checks that these files exist and does not interfere with their content, because this is the space that holds the project's identity.

When designing a new variant (Chapters 6 and 11), deciding in advance which of these three categories a new file belongs to prevents questions like "why does this file get overwritten every time we upgrade?" from ever arising. The judgment criterion is simple — classify anything that must be identical across the whole organization, such as security and governance, as LOCKED; anything where organizational rules and project context must coexist as MERGE; and anything unique to the project as PRESERVE.

Model Tier Strategy — Operating Tiers at Organizational Scale

Chapter 5 showed how tier: high/medium/low is assigned to individual agent roles. Here we examine, from an architectural standpoint, the flow of changing and maintaining that assignment. The tier value lives in the frontmatter of agents/*.md, and as seen in the SSOT table above, L0 is the source for this file, managed by upgrade-project.ts in §5 under SYNC_IF_NEWER — meaning the tier strategy itself follows the L0→L1→L2 principle exactly.

  • The default assignment is decided once, at L0. Once the principle that PM/Architect-type roles are high, code-writer/docs-writer-type roles are medium, and automation-engineer-type roles are low is finalized in the workspace root's agents/*.md, every variant's L1 template inherits it.
  • When tier criteria change, the version is bumped. An organization-wide re-judgment such as "this role should now be high instead of medium" takes the form of editing L0's agents/*.md and bumping @version (or the frontmatter version:). This is not automatically reflected in already-created L2 projects; each project must run upgrade-project.ts (Chapter 10) to receive the new tier — tier changes follow the fork model without exception, too.
  • Project-specific exceptions are left only at L2. If a specific role in a specific variant needs a different tier (for example, giving a code-writer that handles unusually complex domain logic high instead of medium), that project's agents/*.md is edited directly, locally. However, since this file is subject to SYNC_IF_NEWER, keep in mind that if L1's version is higher at the next upgrade, it may be overwritten along with a ⚠️ CONFLICT warning (see the safeguards in §5).
The boilerplate strategy and the tier strategy are, in the end, two answers to the same question — "enforce at L0/L1 whatever must be identical across the whole organization, and delegate to L2 whatever may differ per project." Both strategies share the common trait of drawing that boundary explicitly, one at the file level (LOCKED/MERGE/PRESERVE) and the other at the field level (tier).
