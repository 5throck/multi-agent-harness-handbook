Chapter 8 §3

AGENTS.md In Depth

This section covers the specification format of AGENTS.md, how it is implemented across tools, authoring principles, and the verification process. It builds on the conceptual introduction to AGENTS.md given in Chapter 5, taking a practitioner's perspective.

Standard spec: agents.md · Hands-on examples: Chapter 4 §2-A / Chapter 4 §2-B

What this section covers
  • The field structure and meaning of an agent entry
  • How the common AGENTS.md specification is each "implemented" differently in the Claude family and the Antigravity family
  • The principle of least privilege and handoff declarations
  • The verification tool (agent:verify) and its connection to the lifecycle

Spec Format

Each agent entry in AGENTS.md is a Markdown section that begins with ## Agent: {name}. Within this section, fields such as role, input, output, permissions, and handoff are described in a free-form manner. The standard specification does not enforce a strict schema; it relies on conventions that tools commonly recognize.

FieldMeaningRequired?Example from Chapter 4 Hands-On
## Agent: {name} Agent identifier. In the Claude family, this must match the subagent_type value to be invokable Required ## Agent: reviewer
Role What this agent does Required "Reviews text files for typos, logical gaps, and improvement points."
Input What input it receives Recommended "Path to the text file to be reviewed"
Output What output it produces Recommended "List of typos, logical gaps, and improvement points"
Permissions Scope of file access. Principle of least privilege — allow only what is needed Recommended "Read-only — does not modify files directly"
handoff_to The other agent to which results are passed after completion. A declarative hint referenced by the orchestrator Optional handoff_to: reviewer

Complete Example

Below is the full picture of an AGENTS.md file containing the two agents authored in the Chapter 4 hands-on exercises (G-1, D-1).

## Agent: reviewer

Role: Reviews text files for typos, logical gaps, and improvement points.
Input: Path to the text file to be reviewed
Output: List of typos, logical gaps, and improvement points
Permissions: Read-only — does not modify files directly.

## Agent: writer

Role: Writes a short 3-5 sentence draft on a given topic.
Input: Topic, path to save the file to
Output: Draft text file
Permissions: Write allowed — saves only to the specified file.
handoff_to: reviewer (proceeds to review after the draft is complete)
AGENTS.md is not an instruction directed at an AI tool but a registry of roles defined by a human. The Claude family reads this registry and "translates" it into tool-specific definition files (.claude/agents/*.md), while the Antigravity family has its orchestration read and interpret the registry directly.

Tool-Specific Implementation Mapping

AGENTS.md is a tool-neutral common specification. However, the way subagents are actually "implemented" differs fundamentally between the Claude family and the Antigravity family.

LayerClaude App / Claude CodeAntigravity Desktop / CLI
Reading AGENTS.md The main session reads AGENTS.md and identifies the agent roster The orchestrator reads AGENTS.md and dynamically creates subagents
Subagent definition .claude/agents/{name}.md files with YAML frontmatter (name, description, tools, model) plus a system prompt No definition file — the orchestrator autonomously determines the role and tool scope at execution time
Permission implementation Only the minimum required tools (Read, Grep, etc.) are listed in the tools field of .claude/agents/*.md The orchestrator interprets the Permissions field of AGENTS.md and restricts tool access on its own
Handoff handoff_to is a reference hint. The main session coordinates the relationship by invoking the Agent tool sequentially The orchestrator reads handoff_to and automatically constructs a sequential pipeline

Authoring Guide

1. Start with a Section

Begin each agent entry with ## Agent: {name}. The name should be in lowercase English, with multiple words joined by - (e.g., code-reviewer, api-tester). This name is used as the subagent_type value in the Claude family.

2. Keep the Role Concise

Describe Role — "what it does" — clearly in one or two sentences. Avoid overlap with other agents, and make sure each agent's area of independent responsibility is clear.

3. Set Permissions with Least Privilege

The Permissions field restricts the scope of file access. Failing to follow this principle can allow a subagent to accidentally overwrite important files. In the Claude family, this permission is translated into the tools field (Read, Grep, Write, etc.) of .claude/agents/*.md.

Since the reviewer cannot write in the first place, mistakes simply cannot happen. — Chapter 4 §2-A, Principle of Least Privilege

4. Declare Order with Handoff

handoff_to declares "where results go once this agent finishes its work." This is a hint, not a command. Actually constructing the sequential pipeline is the responsibility of the main session (the orchestrator).

## Agent: draft-writer
Role: Writes a draft on a given topic.
handoff_to: reviewer (proceeds to review after the draft is complete)

## Agent: reviewer
Role: Reviews the text for typos and logical issues.
handoff_to: editor (proceeds to final editing after review is complete)

## Agent: editor
Role: Finalizes the document by incorporating feedback.
An agent with no handoff_to declared (editor in the example above) is the endpoint of the pipeline. The orchestrator returns that agent's result to the end user.

5. Verification

Once authoring is complete, verify the structure with bun run agent:verify. This script checks the completeness of agent frontmatter, the presence of required sections, and consistency with the AGENTS.md roster.

Verification and Lifecycle

Registering an agent in AGENTS.md corresponds to the Create phase of the lifecycle (see Chapter 8 §2). A registered agent then passes through the following stages.

Lifecycle StageBehavior in AGENTS.mdVerification
Create Adds a ## Agent: {name} section to AGENTS.md bun run agent:verify — structural completeness
Verify Confirms the registration is consistent with other workspace files (CONSTITUTION.md, agents/*.md) bun run agent:verify — integrity check
Operate The orchestrator reads the roster and invokes agents during an actual session Trace agent invocations in session logs
Improve Modifies role definitions, permissions, or handoff relationships Rerun agent:verify after modification
Deprecate Removes or comments out the corresponding agent entry from AGENTS.md agent:verify — reference integrity (ensures no other agent's handoff_to points to a deprecated agent)

Making it a habit to run agent:verify whenever you create or change an agent lets you catch inconsistencies early — such as "listed in the roster but no actual definition file exists" or "the agent pointed to by handoff_to has been deleted."

Automating this verification flow can be integrated as a validate-templates.ts step in the CI/CD pipeline covered in Chapter 8 §1 (Deployment and SSOT).

FAQ: If running bun run agent:verify produces an error such as "the AGENTS.md table and the actual list of agents/*.md files are out of sync," compare the agent names in AGENTS.md against the filenames in the .claude/agents/ folder, then add any missing entries or remove unnecessary files. → FAQ
