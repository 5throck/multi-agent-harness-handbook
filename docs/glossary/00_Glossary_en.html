Core Concepts

Harness Engineering Harness Engineering
The discipline of designing which tools, rules, and verification mechanisms a model works through. It is the third stage in the progression from prompt engineering (what to say) to context engineering (what to make known). Details: Chapter 2 §1
Subagent Subagent
A helper agent that has a context window independent of the main session, handles a narrow task, and returns only a summary. It is the mechanism by which the "specialist" concept is implemented in real tools. Details: Chapter 2 §3b
Handoff Handoff
The act, or the point, at which work is passed between agents. A Handoff Contract structures the format of the information exchanged — by spelling out purpose, scope, and prior-step deliverables, it lets the next agent start immediately without needing to ask follow-up questions. Details: Chapter 2 §3
Dynamic Team Composition Dynamic Team Composition
The orchestrator deciding, for each request, which subagents to actually deploy, how many times, and in what order. Even though the defined team roster is fixed, the actual deployment configuration varies task by task. Details: Chapter 2 §4b · Practice: Chapter 4 D-2
Orchestration Pattern Orchestration Pattern
Anthropic's well-known set of five representative patterns: prompt chaining (sequential step-by-step processing), routing (classifying a request and dispatching it to the appropriate agent/model), parallelization (fan-out to run independent tasks concurrently, or voting to repeat the same task and take a majority), orchestrator-workers (dynamically decomposing subtasks rather than fixing them in advance), and evaluator-optimizer (independently verifying a draft before incorporating it — verify-then-apply). These can be freely combined within a single workflow. Details: Chapter 2 §4
Context Window / Context Overload Context Window / Context Overload
The context window is the limit on the number of tokens (text) an AI model can process at once. Within this limit it must hold the role definition, past conversation, task instructions, and output all together; as content grows, earlier information starts to fade — this is called context overload. Distributing work by role to address this is one of the core motivations behind multi-agent teams. Details: Chapter 1 §Multi-Agent: The Principle of an Expert Team
Frontmatter Frontmatter
A metadata block written in YAML at the top of an agent definition file (agents/*.md). It defines things like the agent's name, role, allowed tools (tools), and model tier (tier) so they can be parsed mechanically. The extends field can be used to inherit from another agent file. Details: Chapter 8 §1 Deployment and the SSOT
Scaffolding Scaffolding
The act of using bun scripts/new-project.ts to clone a variant template's entire configuration (agents, skills, commands, governance pipeline) into an independent git repository. The idea being: "you don't build a project from scratch — you start from a pre-designed skeleton." Details: Chapter 6 §2
Hook Hook
A script that runs automatically before or after an agent calls a particular tool. Example: a PostToolUse hook that runs an automated audit check after a file edit. In Claude Code (CLI), hooks defined under .claude/hooks/ fire as expected, but they do not fire in Claude App (Desktop) — so it's important to be aware of these per-tool behavioral differences. Details: Chapter 3 §Audit Logs and Observability
Kanban / WIP Limit Kanban / WIP Limit
A concurrency-control method that caps the number of tasks in progress (Work In Progress) at any one time. When fanning out to run many agents simultaneously, setting a WIP limit prevents the resource contention and dropped context that arise from too much concurrent execution. Details: Chapter 2 §Orchestration Patterns
Service Ticket Model Service Ticket Model
An operating model that uses AI purely as "a service that handles requests as they come in" — similar to a traditional IT organization's ticketing system. The user must direct every request explicitly, which limits the agent team's autonomy. This is the approach this handbook contrasts with its proposed "development tool + variant model." Details: Chapter 7 · Comparing Enterprise Operating Strategies
Lifecycle / State Lifecycle / State
Lifecycle refers to the entire sequence of stages an agent, skill, or script passes through (e.g., Create → Verify → Operate → Improve → Deprecate). State is a snapshot of which lifecycle stage that component is in at a given moment. The status field in the YAML frontmatter (draft, active, deprecated, retired) stores the state value. Lifecycle can be thought of as a "subway map," and state as the sign showing "which station the current train is at." Details: Chapter 8 §2 · Distinguishing State from Lifecycle

Multi-Agent Team Components

Orchestrator Orchestrator
The role that breaks the overall task into stages, calls on specialists, and pauses at points where user confirmation is needed. Commonly referred to as the "PM agent."
Specialist Specialist
An agent with a narrow, clearly defined responsibility — such as design, implementation, testing, or security review — whose inputs and outputs are clearly specified.
Model Tier Model Tier
The model cost tier (high/medium/low) assigned per agent role. Roles requiring heavy judgment are assigned high; standardized, repetitive work is assigned low — balancing cost and quality. It is specified via the tier field in agents/*.md frontmatter, and the platform translates it into an actual model name via model: inherit. Details: Chapter 5 §3. Note: this is a different concept from the "3-tier" of L0→L1→L2 — don't confuse the two.

ai-workspace-standards Structure

SSOT Single Source of Truth
The principle that a given piece of information has its original copy in exactly one place. Example: CONSTITUTION.md is the SSOT for the entire workspace's rules; context.md is the SSOT for an individual project. Details: Chapter 5 §4
L0 / L1 / L2
The three-tier hierarchy running from the workspace root (L0, the sole original) → the common template snapshot (L1, templates/common/) → the variant/project (L2, templates/co-*/ or Projects/*/). The flow always moves top-down, and the L1→L2 transfer is a "fork" that happens exactly once, at scaffolding time. Details: Chapter 8 §1 Deployment and the SSOT
Fork Model Fork Model
The approach whereby, once L2 branches off from L1, it evolves independently with no automatic reverse synchronization. Promoting it to an official template requires a human to explicitly run l2-to-variant-pipeline.ts.
variant
A complete multi-agent team configuration for a specific domain — living under templates/ — bundling a PM, specialist agents, and a governance pipeline. co-develop, co-design, and co-work are examples. Details: Chapter 5 §3
Phase A / Phase B
The two stages of creating a new variant. Phase A is the stage of experimenting freely as an independent prototype under Projects/; Phase B is the stage of promoting it to an official templates/ variant via promote-variant. Details: Chapter 11 · Creating a New Variant
Boilerplate Boilerplate
A pre-built starting point — PM role definitions, governance pipeline, and command/skill wiring — that would otherwise need to be written from scratch for every project. This is the essence of what ai-workspace-standards distributes.
Container Integration Container Integration
One direction for advancing ai-workspace-standards: the requirement that a scaffolded project be able to integrate with services like Open WebUI in a Docker/Kubernetes environment and be supported on a per-individual basis. Currently only direct integration with local AI coding tools is supported. Details: Chapter 8 §5 Advancement Roadmap
Memory Consolidation Memory Consolidation
One direction for advancing ai-workspace-standards: a feature that gathers and organizes memory/MEMORY.md files scattered across individual projects at the workspace level so that overall trends can be identified. Details: Chapter 8 §5 Advancement Roadmap

Tools & Platforms

AGENTS.md
An open standard file read in common by 30+ AI coding tools. It defines tool-neutral rules such as build/test commands and code style. In this repository it additionally doubles as the "agent roster." Details: Chapter 2 §AGENTS.md, the Harness That Crosses Tools
teammateMode
The parallel execution mode for Agent Teams (an experimental feature). Claude App (Desktop) only supports in-process, while Claude Code (CLI) additionally supports tmux split-pane mode. Details: Common Reference §1
Agent Manager
A higher-level interface in Antigravity (Desktop) for overseeing multiple workspaces from a single screen. The principle is to assign one agent per workspace.
/goal / /agent / /agents
Slash commands in the Antigravity CLI (agy) that invoke dynamic subagent orchestration. /goal runs a goal through to completion, /agent delegates a long-running background task, and /agents lists the status of subagents created so far. This capability isn't unique to agy — Claude Code/App performs the same dynamic orchestration via the Agent (Task) tool and natural-language requests, while Antigravity (Desktop) does so via the Agent Manager GUI. These three commands are simply the names given to that behavior on the CLI screen. Details: Common Reference §2
