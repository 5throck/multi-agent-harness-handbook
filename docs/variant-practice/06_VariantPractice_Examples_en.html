Area 1 · Using an Existing Variant

An exercise in scaffolding an already-built variant into a project and running the agent team on a real topic. The goal is to directly experience how a variant is structured and how the agents collaborate.

● Beginner Getting Started The first step of scaffolding a project from an existing variant and running the agent team on a real topic
G-1 Beginner co-consult — Physical AI Market Research Project
Using co-consult, the variant for strategy consulting, run an actual consulting project analyzing market-entry viability and the competitive landscape for "Physical AI" (AI operating in physical environments such as robotics, autonomous driving, and smart factories). The goal is to experience how the 7-stage consulting workflow — in which the PM (Engagement Leader) dispatches 10 specialist agents (Strategy Analyst, Industry Expert, SME, Change Management Partner, etc.) stage by stage — actually runs on a real project.
Elements Used
git clone new-project.ts list-template-versions.ts
Step-by-Step Execution
  1. Clone a practice repository

    Clone into a separate folder that never overlaps with the workspace you're actually working in.

    git clone https://github.com/5throck/ai-workspace-standards.git ai-workspace-standards-demo
cd ai-workspace-standards-demo
  2. Confirm bun is installed, then list the available variants

    If bun isn't installed, install it first, then check the list of variants and versions the repository provides.

    bun --version
bun scripts/list-template-versions.ts
  3. Scaffold a Physical AI market research project with co-consult

    Create a project with co-consult, the strategy consulting variant. Reflect the practice topic in the project name.

    bun scripts/new-project.ts "physical-ai-market-research" --variant co-consult
  4. Explore the scaffolded project structure

    Review the full structure of the generated project directory. Focus on the agent definition files, the Phase configuration in docs/context.md, and the roster in AGENTS.md to understand how co-consult builds its 7-stage consulting workflow.

    cd physical-ai-market-research
find . -maxdepth 3 -type f | head -40
  5. Define the consulting task — apply the 4-step design framework

    Before starting the agent team, clearly define the goal, inputs, outputs, and validation criteria first. The more concrete these four are, the higher the quality of the results the agent team produces.

    Fill out the framework below, then hand it to the agent team in the PM session.

    Step 1 — Define the goal: "Analyze the market-entry strategy and competitive landscape for Physical AI (robotics, autonomous driving, smart factories) through 2030" Step 2 — Define the inputs: Target industries (robotics, autonomous driving, smart factories), research period (2020–2025 + 2030 forecast), focus keywords (regulation, technology, market size) Step 3 — Define the outputs: A Markdown report structured into 5 sections — Executive Summary / Current-State Analysis / Opportunities & Threats / Strategic Recommendations / References Step 4 — Define validation criteria: Source URLs included for every item, a numbered list of 5+ implications, 2,000+ characters in length
  6. Run the agent team on the Physical AI market research topic

    Start a PM session inside the project and hand the defined consulting task to the PM to run the agent team. Watch which agent is active in each Phase and what output it produces as the process unfolds.

    co-consult's 7-stage workflow runs just like a real consulting project.

    Phase 0 (Kickoff): PM receives the task and defines the project scope and schedule
    Phase 1 (Market Analysis / Organizational Diagnosis): Strategy Analyst estimates the Physical AI market size and maps competitors; Industry Expert provides robotics/autonomous-driving domain technology trends; SME reinforces the practical perspective based on smart-factory deployment cases
    Phase 1.5 (Cross-Validation): Multiple agents cross-review each other's outputs to secure information reliability
    Phase 2 (Proposal Approval Gate): PM reviews the Phase 1 results and judges whether they are client-ready
    Phase 3 (Practical Solution Design): Change Management Partner establishes the adoption roadmap and change-management plan
    Phase 4 (Delivery): Final report (Markdown) is written — Executive Summary / Current-State Analysis / Opportunities & Threats / Strategic Recommendations / References
    Phase 5 (PR/Handoff): Client-facing deliverables are organized
  7. Review the deliverables — trace the agent handoffs

    Once the run completes, review the resulting deliverables. Trace how the market analysis produced by Strategy Analyst in Phase 1 flows through cross-validation in Phase 1.5 into the final report in Phase 4.

    find . -name "*.md" -not -path "*/node_modules/*" | head -20
cat docs/reports/*.md   # Review the generated report
Open physical-ai-market-research/agents/ and look at the agent definition files to see how each agent's phases, handoff_to, and handoff_from fields are structured. In this exercise you can experience how four agent types — the research agent (Strategy Analyst: market/competitor/regulatory research), the analysis agent (Industry Expert: pattern and insight extraction), the writing agent (PM: structuring the report/proposal draft), and the coordination agent (Change Management Partner: overall quality review and integration) — collaborate within a real consulting pipeline.
G-2 Beginner co-deck — Korean Cuisine Globalization Slide Deck
Using co-deck, the variant for producing lecture/presentation material, auto-generate a presentation slide deck on the topic of "Korean Cuisine Globalization Strategy." co-deck is a beta variant that automates an 11-stage pipeline — Research → Source Verifier (reliability check) → Storyline → Design → Image Curation + Diagram Generation (parallel) → HTML Build → Layout Measure → PDF Export. The goal is to compare how a variant with a completely different structure from G-1's co-consult produces a different kind of deliverable.
Elements Used
new-project.ts
Step-by-Step Execution
  1. Scaffold a Korean cuisine globalization deck project with co-deck

    In the same workspace used for G-1, this time create a presentation project with the co-deck variant.

    bun scripts/new-project.ts "korean-cuisine-globalization" --variant co-deck
  2. Explore the scaffolded project structure

    Open it side by side with the co-consult project (physical-ai-market-research/) and compare the structures. Check how the number of agents, Phase (Stage) composition, and specialist skill list differ.

    cd korean-cuisine-globalization
find . -maxdepth 3 -type f | head -40
  3. Set the presentation's framing

    Before running the pipeline, clearly define three things about the direction of the presentation. The clearer the audience and purpose, the higher the quality of the storyline and slides the AI composes.

    Fill out the frame below, then hand it to the agent team in the PM session.

    Audience: Market-entry strategy leads at global food & beverage companies, and officials involved in the Korean government's Korean-food promotion policy Purpose: A persuasive presentation introducing strategies and success stories for globalizing Korean food culture, and proposing market-entry opportunities Time: 15–20 minutes (approx. 12–15 slides)
  4. Run the deck pipeline on the Korean cuisine globalization topic

    Start a PM session inside the project and, together with the defined frame, assign the presentation topic "Korean Cuisine Globalization Strategy — An Analysis of How Korean Food Culture Spreads in Global Markets and Success Stories."

    co-deck's 11-stage pipeline runs as a deck-production workflow.

    Stage 0 (Kickoff / Topic Setting): PM defines the presentation direction based on audience, purpose, and time
    Research: Research on Korean cuisine globalization — global Korean food market size, status by target country, government Korean-food promotion policy
    Source Verifier: Verifies the reliability of statistics and case data — checks source URLs, checks consistency of figures
    Storyline: Builds the presentation storyline — chapters structured as problem statement → current-state analysis → success cases → market-entry strategy → conclusion/recommendations
    Design: Sets the slide design direction — establishes visual consistency through theme/style selection
    Image Curation + Diagram Generation (parallel): Curates Korean-food-related images and auto-generates market-structure and entry-roadmap diagrams
    HTML Build → Layout Measure → PDF Export: Generates the slide HTML, then measures/adjusts the layout, then exports the final PDF
  5. Review and refine the finished deck

    Open the built HTML presentation in a browser to check it. Review slide order, content, and visuals, and request edits where needed.

    Review points:
    · Does the slide order match the storyline (problem statement → current state → cases → strategy → recommendations)?
    · Is the core content of each slide concise, within 3–4 lines?
    · Delete unnecessary slides, add missing content
    · Example edit requests: "Make slide 3's title punchier", "Cut the 4 bullets down to 3"

    Principle: The first draft doesn't have to be the final one — iterative refinement is what makes it polished
  6. TTS and auto-slide setup (optional)

    Try out an additional co-deck feature. Enter the text you want read aloud into the speaker notes (Script), and the browser's TTS engine will automatically narrate it in Korean voice. You can also set up an unattended presentation mode using the Manual/Auto button in the footer.

    TTS setup: Write short, clear sentences in each slide's speaker-notes field
    Auto slides: In the footer, switch the 'Manual' button to 'Auto' (1-second interval by default)
    Use-case scenarios: Online client distribution (sharing the HTML file link), kiosk/exhibition-booth promotion, automatic in-house training playback

    Shortcuts: ←→ move between slides, P (play/stop TTS), A (toggle auto/manual), S (show speaker notes)
Open physical-ai-market-research/agents/ and korean-cuisine-globalization/agents/ side by side and count the files — the difference is stark. co-consult is a 7-stage consulting workflow made up of 10 specialist agents, while co-deck is an 11-stage pipeline made up of specialist agents handling Research, Design, HTML Build, PDF Export, and more — so the agent count and the phase composition in docs/context.md differ substantially. You can directly compare "the real structural differences you couldn't guess from the variant name and badge alone" through the scaffolding output.
It's also possible to feed G-1's co-consult output into G-2's co-deck as input. Handing the completed Markdown report from the Physical AI market research straight to co-deck lets it auto-generate a presentation deck with the same content. This is how the consulting pipeline (co-consult) and the presentation pipeline (co-deck) connect as a full pipeline — from topic input to a presentation-ready HTML deliverable in about 30–45 minutes.
co-deck is still a beta variant — as covered in Ch.5 §3, its structure may change more frequently than an official variant, so it's recommended to try it for learning/experimentation purposes first rather than in production projects.
