2.1 · 멀티 에이전트 팀 실습

빈 실습 폴더에서 시작해 Claude App/Claude Code와 Antigravity(Desktop)/Antigravity CLI, 네 도구에서 같은 시나리오를 따라해보는 실습 가이드 — Desktop 앱을 기준으로 안내하고 CLI 대응을 이어서 소개한다  |  ← 2.0 레퍼런스 문서로 돌아가기

대상 버전/범위Claude Code/App 2026-07 릴리스, Antigravity CLI (agy) 1.1.0+, Antigravity 2.0 — 모두 CLI/앱 기준
이 파일의 목적서브에이전트 정의·호출, 순차 파이프라인, 동적 팀 구성·워크플로우 조합, 병렬 실행을 네 도구에서 각각 손으로 따라해보는 4단계 실습 제공
레퍼런스 문서02_Practice_Manual.html — 각 시나리오에서 링크로 연결됩니다
작성일2026년 7월 14일
이 과정 전체가 따르는 원칙 — 특정 도구 하나에 맞춰 실습을 익히면 다른 도구로 옮길 때 처음부터 다시 배워야 한다. 그래서 아래 모든 시나리오는 먼저 서브에이전트의 역할을 도구 중립적인 AGENTS.md 명세로 적고, 그 같은 명세를 Claude 계열에서는 사전 정의 서브에이전트 파일로, Antigravity 계열에서는 자연어 목표로 각각 "구현"하는 순서를 따른다(1장 §5 AGENTS.md 참고). 정의는 하나, 실행 방식만 도구별로 다르다는 원칙을 손으로 확인하는 것이 이 실습의 핵심 목표다.

준비: 필요한 프로그램 설치

아래 실습은 Claude App(Desktop), Claude Code(CLI), Antigravity(Desktop), Antigravity CLI(agy)를 오가며 진행된다. 로컬 환경에 이 도구들이 아직 없다면, 이 핸드북에 함께 담아 둔 환경 설정 자동화 도구인 setup 가이드가 이 장의 실습에 정확히 필요한 구성을 자동으로 설치해준다. 안내에 따라 스크립트를 내려받아 로컬에서 실행하면 된다.

📦 설치 전 확인 사항

스크립트를 실행하기 전에 사전 설치 체크리스트를 먼저 읽어본다. 이 문서에는 다음 전제 조건이 정리되어 있다.

  • 활성 상태의 Claude Pro/Max 구독 그리고 Antigravity 사용을 위한 Google 계정 (개별 API 키가 아니라 구독제/계정 기반 플랜이어야 한다)
  • gh auth login을 진행할 수 있는 GitHub 계정
  • 최소 5GB의 여유 디스크 공간
  • 관리자(admin) 또는 sudo 권한
GitHub 계정이 아직 없다면
① 브라우저에서 github.com/signup에 접속한다.
② 이메일 주소, 비밀번호, 사용자 이름(username)을 입력한다 — username은 나중에 저장소 URL(github.com/<username>/...)에 그대로 쓰이므로 신중히 정한다.
③ 자동 가입 방지를 위한 퍼즐/코드 인증을 통과한다.
④ 가입 시 입력한 이메일로 온 인증 메일을 열어 이메일 인증을 완료한다 — 이 단계를 건너뛰면 이후 gh auth login이나 저장소 생성이 막힐 수 있다.
⑤ 무료(Free) 플랜으로 가입하면 충분하다 — 이 핸드북의 실습·PR 연습에는 유료 플랜이 필요 없다.
가입이 끝나면 GitHub CLI(gh)가 위 설치 스크립트에 포함돼 함께 설치되고, 아래 gh auth login 명령으로 터미널을 그 계정과 연결한다. 
gh auth login
# 안내에 따라: GitHub.com 선택 → HTTPS 선택 → 브라우저로 로그인(권장) 선택
# 브라우저가 열리면 방금 만든 계정으로 로그인하고 인증 코드를 확인한다

Windows에서 설치하기

이 핸드북의 경로 표기가 Windows 기준이므로, Windows 사용자는 setup-windows.ps1을 내려받아 실습 전용 폴더(예: harness-lab/setup/)에 두고, PowerShell을 관리자 권한으로 실행한 뒤 아래 명령을 돌린다.

PowerShell을 관리자 권한으로 여는 법
① 시작 메뉴를 열고 PowerShell을 검색한다.
② 검색 결과의 "Windows PowerShell"을 마우스 우클릭한다.
③ 나오는 메뉴에서 "관리자 권한으로 실행"을 선택한다.
④ 사용자 계정 컨트롤(UAC) 창이 뜨면 "예"를 눌러 승인한다.
창 제목 표시줄에 관리자: Windows PowerShell처럼 "관리자"가 붙어 있으면 제대로 연 것이다. winget 설치 등 일부 단계는 관리자 권한 없이 실행하면 오류가 나거나 조용히 건너뛰어질 수 있다. 
cd ./setup
.\setup-windows.ps1
# 선택 옵션: -WSL2, -WezTerm, -Docker, -Force (이미 설치된 도구도 재설치)

스크립트가 끝나면 터미널을 완전히 닫았다가 다시 열어야 PATH 변경 사항이 적용된다.

macOS / Linux에서 설치하기

macOS는 setup-mac.sh, Linux는 setup-linux.sh를 내려받아 사용한다. 둘 다 --wezterm, --docker 선택 플래그를 지원한다.

# macOS
cd ./setup && bash setup-mac.sh

# Linux
cd ./setup && bash setup-linux.sh

설치 검증하기

운영체제와 무관하게 setup-common.ts를 bun으로 실행하면 설치 상태를 점검하는 체크리스트 표가 출력된다. 모든 항목이 ✅로 표시되면 아래 실습을 진행할 준비가 된 것이다.

cd ./setup && bun setup-common.ts

이 프로젝트가 설치하는 도구 중 이번 장의 4도구 실습과 직접 관련된 것은 다음과 같다. Claude App(Desktop), claude(Claude Code CLI), Antigravity(Desktop), agy(Antigravity CLI), 그리고 .claude/agents/*.md 기반 서브에이전트 실행에 필요한 bun, 이후 장에서 쓰이는 gh(GitHub CLI)까지 함께 설치된다.

Antigravity CLI(agy)는 별도 런타임 설치가 필요 없는 단일 컴파일 바이너리다 — 압축 해제 후 PATH에 넣기만 하면 바로 실행된다. 옛 자료에 남아 있는 "Gemini CLI를 npm으로 전역 설치" 같은 안내는 더 이상 유효하지 않다. 최신 설치 방법은 antigravity.google/docs/cli-getting-started에서 확인한다.
설치 중 문제가 생기면 Windows 기준 %USERPROFILE%\workshop-setup-logs\ 폴더에 로그가 남는다. 문제를 해결할 때 이 로그부터 확인한다.
● 초급 Getting Started 커스텀 서브에이전트를 하나 만들고 Claude App에서 호출을 관찰한 뒤, Antigravity(Desktop)에서 목표를 주고 동적 생성을 관찰하는 첫 단계
G-1 초급 첫 커스텀 서브에이전트 만들고 호출하기
하네스를 처음 써보는 개발자가 실습 전용 빈 폴더 harness-lab/를 새로 만들고, reviewer 역할을 도구 중립적인 AGENTS.md 명세로 먼저 적어 둔 뒤, 그 같은 명세를 Claude App(Desktop)에서는 사전 정의 서브에이전트로, Antigravity(Desktop)에서는 자연어 목표로 각각 "구현"해 호출해본다. 이 폴더는 실습 전용이므로 다른 프로젝트 파일과 섞이지 않는다.
사용 요소
AGENTS.md (1장) 서브에이전트 정의하기 서브에이전트 호출하기
단계별 실행
  1. ① 실습 폴더 생성 및 진입

    현재 폴더 아래 harness-lab 폴더를 새로 만들고 그 안으로 이동한다. 이후 모든 명령은 이 폴더 안에서 실행한다.

    mkdir harness-lab
cd harness-lab
  2. ② 공통 역할 명세 작성 — AGENTS.md

    Claude나 Antigravity, 그 외 AGENTS.md 표준을 지원하는 어떤 도구로 열어도 같은 의미로 읽히도록, reviewer 역할을 도구 종속적인 표현 없이 순수 텍스트로 적는다. 이 파일 하나가 "무엇을 하는 역할인가"의 유일한 원본이 되고, 아래 두 도구는 각자의 방식으로 이 원본을 구현할 뿐이다. macOS/Linux/Git Bash에서는 아래 명령을 그대로 쓴다.

    cat > AGENTS.md << 'EOF'
## Agent: reviewer

역할: 텍스트 파일의 오탈자, 논리적 허점, 개선점을 검토한다.
입력: 검토 대상 텍스트 파일 경로
출력: 오탈자 · 논리적 허점 · 개선점 목록
권한: 읽기 전용 — 파일을 직접 수정하지 않는다.
EOF

    Windows PowerShell에서는 cat 명령 자체는 Get-Content의 별칭으로 파일 읽기에 그대로 쓸 수 있다. 다만 << 'EOF'처럼 여러 줄을 직접 넣는 heredoc 문법은 지원하지 않으므로, 같은 내용은 따옴표 있는 here-string(@'...'@)으로 만들어 Set-Content로 파일에 쓴다.

    @'
## Agent: reviewer

역할: 텍스트 파일의 오탈자, 논리적 허점, 개선점을 검토한다.
입력: 검토 대상 텍스트 파일 경로
출력: 오탈자 · 논리적 허점 · 개선점 목록
권한: 읽기 전용 — 파일을 직접 수정하지 않는다.
'@ | Set-Content AGENTS.md
Claude App (Desktop) — AGENTS.md 명세를 사전 정의 서브에이전트로 구현

터미널이나 별도 에디터에서 미리 파일을 만들지 않는다. 아래 세 단계를 그대로 따라간다 — 회색 코드 상자는 전부 "복사" 버튼으로 그대로 복사해서 붙여넣으면 되는 내용이다.

  1. ① Claude App에서 프로젝트 열기

    Claude App(Desktop)을 실행하고, 방금 만든 harness-lab/ 폴더를 프로젝트로 연다.

  2. ② 서브에이전트 생성 요청 — 채팅창에 아래 문장을 그대로 붙여넣고 전송 
    AGENTS.md에 적힌 reviewer 역할 명세를 그대로 따라서, .claude/agents/reviewer.md에 reviewer 서브에이전트를 만들어줘. tools는 Read, Grep만 허용하고 model은 sonnet으로 설정해줘.
    Claude App이 AGENTS.md의 "역할/입력/출력/권한" 명세를 읽고, 그 내용을 .claude/agents/reviewer.md의 YAML frontmatter(description, tools)와 시스템 프롬프트로 옮겨 Write 도구로 직접 생성한다. 명세는 그대로고 표현 형식만 도구에 맞게 바뀐 것이다 — 생성이 끝나면 채팅에 표시된 파일 내용을 AGENTS.md와 비교해본다.
  3. ③ 검토 대상 파일 생성 + 서브에이전트 호출 — 같은 채팅창에 이어서 붙여넣고 전송 
    notes.txt 파일을 만들고 "이 프로젝트는 매우 좋다. 이것은 아주 좋다."라고 써넣은 뒤, reviewer 서브에이전트로 notes.txt를 검토해줘.
    Claude App이 Agent(Task) 툴을 description="notes.txt 검토", subagent_type="reviewer"로 호출한다. 화면에는 서브에이전트가 스폰되는 과정과, 작업을 마친 뒤 메인 세션으로 돌아오는 요약 결과가 순서대로 표시된다.

Claude Code(CLI)에서도 동일하게 — 터미널에서 claude를 실행한 뒤 ②③의 문장을 그대로 채팅으로 붙여넣으면, CLI가 똑같이 AGENTS.md 명세를 읽고 .claude/agents/reviewer.md를 직접 생성·호출한다. CLI에는 추가로 PostToolUse 등 자동화 훅이 발화한다는 차이가 있다(App에서는 훅이 발화하지 않는다).

Antigravity (Desktop) — 같은 AGENTS.md 명세를 자연어 목표로 구현

Antigravity는 .claude/agents 같은 사전 정의 파일 개념이 없다. 대신 방금 작성한 AGENTS.md를 워크스페이스에 그대로 둔 채, 같은 명세를 목표 프롬프트에 담아 준다 — 파일 형식만 다를 뿐 "무엇을 검토할지"의 원본은 Claude App 쪽과 동일하다.

  1. ① 검토 대상 파일 미리 준비 (터미널) 
    echo "이 프로젝트는 매우 좋다. 이것은 아주 좋다." > notes.txt
  2. ② Antigravity Agent Manager에서 Workspace 생성

    Agent Manager를 열고 harness-lab/ 폴더(AGENTS.md가 이미 들어있는 폴더)를 가리키는 새 Workspace를 하나 만든다.

  3. ③ 목표 지시 — Workspace 채팅창에 아래 문장을 그대로 붙여넣고 전송 
    AGENTS.md에 적힌 reviewer 역할 명세대로 notes.txt를 검토해줘.
    Workspace를 맡은 에이전트가 AGENTS.md를 읽고, 그 안의 역할/입력/출력/권한 명세를 그대로 목표로 삼아 필요한 작업(파일 읽기, 개선점 정리)을 스스로 판단해 수행한다. Claude App이 만든 .claude/agents/reviewer.md 파일은 Antigravity에서 아무 역할도 하지 않는다 — 대신 두 도구 모두 같은 AGENTS.md 명세를 원본으로 삼았기 때문에, 검토 기준(오탈자·논리적 허점·개선점)은 도구가 달라도 동일하게 유지된다.

Antigravity CLI(agy)에서도 같은 개념으로 실습할 수 있다 — 사전 정의 파일 없이 /goal로 목표만 준다.

cd harness-lab && agy
/goal notes.txt 파일을 검토하고 개선점을 요약해줘
Claude App/Code에서는 tools 필드를 Read, Grep처럼 필요한 최소 권한만 나열하면, 리뷰어가 실수로 파일을 수정하는 사고를 방지할 수 있다. Antigravity는 정의 파일이 없으므로 이런 사전 권한 제한 대신, 목표 프롬프트 자체를 "읽고 요약만 해줘"처럼 명확히 좁히는 것이 안전하다.
● 중급 Daily Workflow 메인 세션이 오케스트레이터가 되어 writer와 reviewer 두 전문 에이전트를 순서대로 지휘하는 실무 패턴
D-1 중급 writer → reviewer 2단계 파이프라인
G-1을 마친 실무자가 같은 harness-lab/ 폴더에서, 두 번째 역할 writer도 G-1과 같은 방식으로 AGENTS.md에 먼저 명세한 뒤, Claude App에는 사전 정의 서브에이전트로, Antigravity(Desktop)에는 자연어 목표로 각각 구현한다. writer가 먼저 초안을 쓰고 reviewer가 그 결과를 비평하는 파이프라인을 네 도구 모두에서 순서를 지켜 지휘하게 한다.
사용 요소
AGENTS.md (1장) 서브에이전트 정의하기 서브에이전트 호출하기

Antigravity (Desktop) — Agent Manager, Workspace별 에이전트 1개

Antigravity를 열고 Agent Manager에서 harness-lab/을 가리키는 Project를 하나 준비한다. 이번에는 하나의 Workspace에 두 에이전트를 몰아넣지 않고, 서로 다른 Workspace 두 개를 만들어 각각 에이전트를 하나씩 배정한다.

  1. ① Workspace A 생성

    New Worktree Mode로 Workspace를 하나 만든다.

  2. ② Workspace A 채팅창에 아래 문장을 그대로 붙여넣고 전송 
    topic-a.txt에 '재택근무' 주제로 3~5문장 초안을 작성해줘.
  3. ③ Workspace B 생성

    별도의 Workspace를 하나 더 만든다(New Worktree Mode 권장). Workspace A의 에이전트와는 완전히 분리된 인스턴스다.

  4. ④ Workspace B 채팅창에 아래 문장을 그대로 붙여넣고 전송 
    topic-b.txt에 '4일 근무제' 주제로 3~5문장 초안을 작성해줘.
  5. ⑤ Inbox에서 두 에이전트 진행 상황 확인

    두 에이전트가 각자의 Workspace에서 동시에 작업하는 동안, Inbox에서 승인이나 피드백이 필요한 메시지가 오는지 확인하고 필요하면 응답한다.

두 에이전트를 같은 Workspace에 함께 배정하지 않는다 — 같은 폴더를 공유하면 한 에이전트가 다른 에이전트의 작업 맥락을 자기 것으로 착각하는 "인지적 중첩(cognitive overlap)"이 발생할 수 있다. Workspace 또는 worktree로 반드시 분리하는 것이 Antigravity의 권장 관행이다.

Antigravity CLI(agy)에서도 데스크톱의 Agent Manager 대신 터미널을 여러 개 띄워 같은 효과를 낼 수 있다 — 한 터미널에서 agy로 topic-a.txt 작업을, 다른 터미널에서 별도 agy 세션으로 topic-b.txt 작업을 동시에 진행한다(세션 간 병렬). 서로 다른 파일을 건드리는 한 두 세션은 충돌하지 않는다.

  3. ③ 팀메이트 병렬 스폰 요청 — 채팅창에 아래 문장을 그대로 붙여넣고 전송

Agent Teams가 켜진 상태에서는 여러 Claude 인스턴스가 팀메이트로 병렬 실행되며, 공유 작업 목록에 각 팀메이트의 할 일이 등록되고 상태가 갱신된다. 두 writer 팀메이트는 topic-a.txt/topic-b.txt 사이에 의존성이 없으므로 동시에 진행되고, 팀메이트 간에는 SendMessage로 직접 메시지를 주고받을 수도 있다. 다만 Desktop App에서는 일부 키보드 내비게이션(Shift+Down 등)이 제한된다.
작업 목록이 서로 얽히지 않도록, 병렬로 스폰하는 팀메이트에게는 처음부터 완전히 분리된 파일이나 디렉터리를 배정하는 것이 안전하다.

Claude Code(CLI)에서도 동일하게 — 같은 설정과 같은 요청 문장이 그대로 동작한다. CLI에서만 teammateMode: "tmux"를 추가로 선택할 수 있어, tmux 분할 창으로 각 팀메이트의 화면을 나란히 볼 수 있다는 점이 Desktop App과 다르다.

G-1의 notes.txt보다 조금 더 복잡한, 논리적 허점이 있는 글을 준비한다. 채팅창에 아래 문장을 그대로 붙여넣는다.

    argument.txt 파일을 만들고 다음 내용을 써줘: "재택근무는 항상 사무실 근무보다 생산적이다. 왜냐하면 재택근무를 하는 사람들이 재택근무가 좋다고 말하기 때문이다. 따라서 모든 회사는 재택근무만 해야 한다."
  2. ② haiku 티어로 검토

    reviewer 서브에이전트 정의(.claude/agents/reviewer.md)의 model 필드를 haiku로 바꾼 뒤 요청한다.

    argument.txt를 reviewer 서브에이전트로 검토해줘.
  3. ③ sonnet 티어로 같은 요청 반복

    model 필드를 sonnet으로 바꾼 뒤, 같은 검토 요청 문장을 다시 붙여넣는다. 순환 논증(circular reasoning) 같은 논리적 허점을 정확히 짚어내는지 haiku 결과와 비교한다.

  4. ④ opus 티어로 같은 요청 반복

    model 필드를 opus로 바꾼 뒤 다시 요청한다. 응답 시간과 지적의 깊이가 ②③과 어떻게 다른지 기록한다.

간단한 오탈자 검토라면 세 모델의 결과 차이가 거의 없지만, 이번처럼 "논증 구조의 허점을 짚어야 하는" 작업에서는 티어가 낮을수록 순환 논증을 순환 논증이라고 정확히 짚지 못하고 표면적인 문장만 지적하는 경향이 뚜렷해진다. 반대로 haiku는 셋 중 가장 빠르게 응답한다 — 5장에서 설명하는 "판단이 무거운 역할일수록 높은 티어, 정형화된 작업일수록 낮은 티어"라는 원칙이 왜 존재하는지, 속도와 깊이의 트레이드오프를 직접 겪어보면 훨씬 분명해진다.
매번 .claude/agents/reviewer.md를 열어 model 줄만 고치고 저장한 뒤 같은 요청을 반복하면 된다 — 정의 파일을 다시 만들 필요는 없다.
