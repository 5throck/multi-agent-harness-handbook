The Principle Behind the Entire Course — Every scenario below follows the same order: first write the subagent's role as a tool-neutral AGENTS.md specification, then "implement" that specification in the Claude family as a predefined subagent file (see Ch.2 §5 AGENTS.md). Confirming by hand that there is one specification but the execution method differs per tool is the core goal of this exercise. For the Antigravity equivalent, see Ch.4 §2-B.

Setup: Installing the Required Programs

The exercises below focus on Claude App (Desktop) and Claude Code (CLI). If your local environment doesn't have these tools yet, the environment setup automation included with this handbook — the setup guide — will automatically install exactly what this chapter's exercises need. Just follow the instructions to download and run the script locally.

📦 Before You Install — Things to Check

Before running the script, read the pre-installation checklist first. It lists the following prerequisites.

  • An active Claude Pro/Max subscription and a Google account for Antigravity access (this must be a subscription/account-based plan, not an individual API key)
  • A GitHub account you can use to run gh auth login
  • At least 5GB of free disk space
  • Administrator or sudo privileges
If you don't already have a GitHub account
① Go to github.com/signup in your browser.
② Enter your email address, password, and username — the username is used as-is in the repository URL later (github.com/<username>/...), so choose it carefully.
③ Complete the puzzle/code verification to prevent automated sign-ups.
④ Open the verification email sent to the address you signed up with and complete email verification — skipping this step can block gh auth login or repository creation later.
⑤ Signing up for the Free plan is enough — this handbook's exercises and PR practice don't require a paid plan.
Once sign-up is complete, the GitHub CLI (gh) is installed together as part of the setup script above, and the gh auth login command below connects your terminal to that account. 
gh auth login
# Follow the prompts: choose GitHub.com → choose HTTPS → choose Login with a web browser (recommended)
# When the browser opens, log in with the account you just created and confirm the authentication code

Installing on Windows

Since this handbook's path notation is Windows-based, Windows users should download setup-windows.ps1, place it in a dedicated practice folder (e.g., harness-lab/setup/), run PowerShell as Administrator, and then run the command below.

How to Open PowerShell as Administrator
① Open the Start menu and search for PowerShell.
② Right-click on "Windows PowerShell" in the search results.
③ Select "Run as administrator" from the menu that appears.
④ When the User Account Control (UAC) prompt appears, click "Yes" to approve.
If the title bar shows something like Administrator: Windows PowerShell, you've opened it correctly. Some steps, like winget installs, may error out or silently skip without administrator privileges. 
cd ./setup
.\setup-windows.ps1
# Optional flags: -WSL2, -WezTerm, -Docker, -Force (reinstall tools even if already installed)

After the script finishes, you must fully close and reopen your terminal for the PATH changes to take effect.

Installing on macOS / Linux

On macOS, download and use setup-mac.sh; on Linux, use setup-linux.sh. Both support the optional --wezterm and --docker flags.

# macOS
cd ./setup && bash setup-mac.sh

# Linux
cd ./setup && bash setup-linux.sh

Verifying the Installation

Regardless of your OS, running setup-common.ts with bun prints a checklist table showing your installation status. If every item shows ✅, you're ready to proceed with the exercises below.

cd ./setup && bun setup-common.ts

Among the tools this project installs, the ones directly relevant to this chapter's four-tool exercises are: Claude App (Desktop), claude (Claude Code CLI), Antigravity (Desktop), agy (Antigravity CLI), bun (needed to run subagents based on .claude/agents/*.md), and gh (GitHub CLI), which is used in later chapters as well.

The Antigravity CLI (agy) is a single compiled binary that requires no separate runtime installation — just unzip it and put it on PATH, and it's ready to run. Older guidance floating around, such as "install the Gemini CLI globally via npm," is no longer valid. Check antigravity.google/docs/cli-getting-started for the current installation instructions.
If you run into problems during installation, logs are saved to %USERPROFILE%\workshop-setup-logs\ on Windows. Check these logs first when troubleshooting.
● Beginner Getting Started The first step: create a custom subagent and observe it being called from Claude App
G-1 Beginner Creating and Calling Your First Custom Subagent
A developer trying the harness for the first time creates a fresh, empty practice folder called harness-lab/, first writes a reviewer role as a tool-neutral AGENTS.md specification, then "implements" it as a predefined subagent in Claude App (Desktop) and calls it. Since this folder is dedicated to practice, it won't get mixed up with other project files.
Elements Used
AGENTS.md (Ch.2) Defining a Subagent Calling a Subagent
Step-by-Step
  1. ① Create and Enter the Practice Folder

    Create a new folder named harness-lab under your current folder and move into it. Every command from here on runs inside this folder.

    mkdir -p harness-lab
cd harness-lab
  2. ② Write the Common Role Specification — AGENTS.md

    Write the reviewer role as plain text, free of any tool-specific wording, so it reads with the same meaning whether opened in Claude, Antigravity, or any other tool that supports the AGENTS.md standard. This single file becomes the sole source of truth for "what this role does," and the two tools below merely implement that source, each in its own way. On macOS/Linux/Git Bash, run the command as-is.

    cat > AGENTS.md << 'EOF'
## Agent: reviewer

Role: Review text files for typos, logical gaps, and areas for improvement.
Input: Path to the text file to review
Output: A list of typos, logical gaps, and improvement suggestions
Permissions: Read-only — does not modify the file directly.
EOF

    On Windows PowerShell, the cat command itself is an alias for Get-Content and can be used as-is for reading files. However, it doesn't support the heredoc syntax (<< 'EOF') for entering multiple lines directly, so build the same content as a quoted here-string (@'...'@) and write it to the file with Set-Content.

    @'
## Agent: reviewer

Role: Review text files for typos, logical gaps, and areas for improvement.
Input: Path to the text file to review
Output: A list of typos, logical gaps, and improvement suggestions
Permissions: Read-only — does not modify the file directly.
'@ | Set-Content AGENTS.md
Claude App (Desktop) — Implementing the AGENTS.md Spec as a Predefined Subagent

Don't create the file ahead of time in a terminal or separate editor. Just follow the three steps below — every gray code box is meant to be copied as-is with the "Copy" button and pasted in.

  1. ① Open the Project in Claude App

    Launch Claude App (Desktop) and open the harness-lab/ folder you just created as a project.

  2. ② Request Subagent Creation — Paste the Text Below into the Chat Box and Send 
    Following the reviewer role spec written in AGENTS.md exactly, create a reviewer subagent at .claude/agents/reviewer.md. Allow only the Read and Grep tools, and set the model to sonnet.
    Claude App reads the "role/input/output/permissions" spec in AGENTS.md and directly creates .claude/agents/reviewer.md using the Write tool, transferring the content into the YAML frontmatter (description, tools) and system prompt — the spec stays the same, only the format has been adapted to the tool. Once creation finishes, compare the file contents shown in the chat against AGENTS.md.
  3. ③ Create a Target File and Call the Subagent — Continue Pasting in the Same Chat and Send 
    Create a file called notes.txt containing the text "This project is very good. This is very good." Then use the reviewer subagent to review notes.txt.
    Claude App calls the Agent (Task) tool with description="Review notes.txt" and subagent_type="reviewer". The screen shows, in order, the subagent being spawned and, once its work is done, a summary result returned to the main session.

The same applies in Claude Code (CLI) — running claude in a terminal and pasting the same sentences from steps ②③ into the chat causes the command-line interface (CLI) to read the AGENTS.md spec the same way and directly create/call .claude/agents/reviewer.md. The one difference in the CLI is that automation hooks such as PostToolUse fire additionally (hooks don't fire in the App).

Verifying the Result — Check whether harness-lab/.claude/agents/reviewer.md was actually created, and whether its description, tools, and model fields match what you requested. The final response should show a list divided into typos, logical gaps, and improvement suggestions — if the reviewer responds with something like "I fixed it," the tools restriction was applied incorrectly.
A Common Failure — If the subagent is never called and the main session reviews notes.txt directly instead, it usually means step ②'s sentence was missing the phrase "with the reviewer subagent," or step ③ was requested before creation of .claude/agents/reviewer.md finished. Confirm that file creation finished in step ②'s response before sending step ③.
● Intermediate Daily Workflow A practical pattern where the main session acts as an orchestrator directing two specialist agents — writer and reviewer — in sequence
D-1 Intermediate A Two-Stage writer → reviewer Pipeline
A practitioner who has finished G-1 writes a second role, writer, in the same harness-lab/ folder, following the same approach as G-1 by first specifying it in AGENTS.md, then implementing it as a predefined subagent in Claude App. They then have it direct a pipeline in which writer drafts first and reviewer critiques the result, in strict order.
Elements Used
AGENTS.md (Ch.2) Defining a Subagent Calling a Subagent
Step-by-Step
  1. Add to the Common Role Specification — Append a writer Entry to AGENTS.md

    Append the writer role to the same AGENTS.md created in G-1. Leave the reviewer entry as-is and add only the content below. On macOS/Linux/Git Bash, run the command as-is.

    cat >> AGENTS.md << 'EOF'

## Agent: writer

Role: Write a short 3-5 sentence draft on a given topic.
Input: Topic, path to save the file to
Output: Draft text file
Permissions: Write access — saves only to the specified file.
handoff_to: reviewer (review follows once the draft is done)
EOF

    On Windows PowerShell, append the here-string with Add-Content (you must use Add-Content rather than Set-Content, which overwrites the file and would erase the reviewer entry).

    @'

## Agent: writer

Role: Write a short 3-5 sentence draft on a given topic.
Input: Topic, path to save the file to
Output: Draft text file
Permissions: Write access — saves only to the specified file.
handoff_to: reviewer (review follows once the draft is done)
'@ | Add-Content AGENTS.md
Claude App (Desktop) — The Main Session Dispatches Agent Calls Sequentially

As in G-1, don't prepare any files in advance. Continue in the same harness-lab/ chat you opened for G-1.

  1. ① Request writer Subagent Creation — Paste the Text Below into the Chat Box and Send 
    Following the writer role spec written in AGENTS.md exactly, create a writer subagent at .claude/agents/writer.md. Allow the Write and Read tools, and set the model to sonnet.
    Claude App reads the writer spec in AGENTS.md and directly creates .claude/agents/writer.md using the Write tool. It sits alongside the reviewer.md already created in G-1, inside the same .claude/agents/ folder — the handoff_to: reviewer entry in AGENTS.md actually plays out in the next step.
  2. ② Request the writer → reviewer Sequential Pipeline — Continue Pasting in the Same Chat and Send 
    Have the writer subagent write a draft.txt draft on the topic 'The Advantages of Remote Work,' and once that's done, have the reviewer subagent review draft.txt. Proceed through the two stages in order.
    The main session first calls Agent with subagent_type="writer" to generate draft.txt, and only after that result comes back does it send the subagent_type="reviewer" call. Coordinating the dependency between the two specialists is the main session's job, not the subagents'.
Calling writer and reviewer in parallel (both in one message) can make reviewer try to review a draft.txt that doesn't exist yet and fail — tasks with a dependency must always be requested as sequential calls (the key is explicitly stating "proceed in order," as in step ② above).
Verifying the Result — The chat should show two subagent spawns in this order: writer call → complete → reviewer call → complete. If it only shows up once, the orchestrator collapsed the pipeline into a single step, so double-check whether your request sentence explicitly said "in order." After completion, also open harness-lab/draft.txt to confirm the draft was actually saved.

The same applies in Claude Code (CLI) — using the same writer.md/reviewer.md as-is and pasting the sentences from steps ①② into a terminal produces the same sequential dispatch.

D-2 Intermediate Observing How Team Composition and Workflow Change with the Request
This scenario lets you see, firsthand, the "dynamic team composition" and "combination of different workflows" covered in Ch.2 §4b. Leave the two subagents created in D-1, writer and reviewer, exactly as they are, and simply vary the nature of the request to observe who the orchestrator actually calls, how many times, and in what order — the key point is that you never define a new subagent again.
A Dynamic Routing Branch Diagram — Same Team, Different Requests
Elements Used
Dynamic Team Composition and Diverse Workflows
Claude App (Desktop) — Same Team, Three Different Requests

Continue in the same chat where you finished D-1. Pay close attention each time to which subagent gets called, how many times, as shown on screen.

  1. ① A Simple Request — A Case That Only Needs reviewer 
    Just skim draft.txt for typos.
    The orchestrator decides writer isn't needed for this request and calls only reviewer. Even with the same team (writer+reviewer) defined, this is the simplest example showing that the orchestrator decides who's actually deployed for each request as it goes.
  2. ② A Complex Request — Combining writer and reviewer with Two Perspectives 
    Write a new draft in draft2.txt on the topic 'The Four-Day Workweek.' Once the draft is done, have it independently reviewed from a grammar perspective and a logic perspective simultaneously.
    This time, after one writer call (the first stage of the pipeline), reviewer is called twice at once (fan-out), each with different instructions — "grammar perspective" and "logic perspective." You can see the pipeline (writer → reviewer) and fan-out (reviewer ×2 running concurrently) orchestration patterns combined within a single request — the same two subagents produced a workflow with a completely different shape than in ①.
Comparing ① and ② side by side makes it clear that the "team roster" (which subagents you've defined) and the "actual composition deployed for this task" (who gets called how many times, in what order) are two different layers. The former is fixed in advance in .claude/agents/, but the latter is decided fresh with every request.

The same applies in Claude Code (CLI) — using the same writer.md/reviewer.md as-is and pasting the sentences from steps ①② into a terminal produces the same behavior.

A Common Failure — If reviewer only gets called once in ② even though you said "review simultaneously," it's likely your request sentence didn't clearly distinguish the two perspectives, like "grammar perspective and logic perspective." Try rephrasing with the perspectives split out more concretely.

● Advanced Power User Running writer and reviewer concurrently against two independent files — parallel operation with Claude Code Agent Teams
P-1 Advanced Running Independent Tasks in Parallel — Agent Manager vs. Agent Teams
A practitioner who has finished through D-1 tries running two independent tasks with no dependency between them — drafting topic-a.txt and topic-b.txt — in parallel using Claude Code Agent Teams.
Elements Used
Parallel Execution — Scaling Up to a Team Parallel Execution
Claude App (Desktop) — Agent Teams, teammateMode: in-process

Enable the experimental Agent Teams feature in the same harness-lab/ folder. In the Desktop App, teammateMode must be in-process (tmux is not supported).

  1. ① Write the Config File (Terminal) — Create a New .claude/settings.json If You Don't Have One, or Merge These Two Keys If You Do

    On macOS/Linux/Git Bash, run the command as-is.

    This command overwrites the entire .claude/settings.json file. If the file already contains other settings, all existing content will be lost, so back it up beforehand or merge the content below manually if you need to keep anything. 
    cat > .claude/settings.json << 'EOF'
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "in-process"
}
EOF

    On Windows PowerShell, create the same file with a here-string.

    @'
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "in-process"
}
'@ | Set-Content .claude/settings.json
  2. ② Restart Claude App

    Fully quit and reopen Claude App for the settings to take effect.

  3. ③ Request Parallel Teammate Spawning — Paste the Text Below into the Chat Box and Send 
    Spawn two writer teammates in parallel — have one write a draft in topic-a.txt on 'remote work' and the other write a draft in topic-b.txt on 'the four-day workweek' at the same time. Once each file is done, have a reviewer teammate review it, and show me the shared task list too.
With Agent Teams enabled, multiple Claude instances run in parallel as teammates, with each teammate's tasks registered and its status updated in a shared task list. Since the two writer teammates have no dependency between topic-a.txt and topic-b.txt, they proceed at the same time, and teammates can also exchange direct messages via SendMessage. Note, however, that some keyboard navigation (such as Shift+Down) is limited in the Desktop App.
To keep task lists from getting tangled, it's safer to assign completely separate files or directories to teammates you spawn in parallel from the start.
A Common Failure — If you can't find the Agent Teams feature or teammates never spawn, it's usually one of two things: ① whether .claude/settings.json was saved relative to the harness-lab/ folder (saving it elsewhere means it won't be recognized), or ② whether Claude App was fully restarted in step ②. A JSON syntax error (like a missing comma) causing the settings file to be silently ignored is also common, so open it in an editor and double-check the syntax.
Before/After Permission Settings — The Principle of Least Privilege

The same applies in Claude Code (CLI) — the same settings and the same request sentence work identically. The one difference from the Desktop App is that the CLI additionally lets you choose teammateMode: "tmux", letting you view each teammate's screen side-by-side in tmux split panes.

P-2 Advanced Building Cost Intuition by Comparing Model Tiers
This scenario lets you experience firsthand, before hearing the numbers explained, what real difference the model tiers (high/medium/low) covered in detail in Chapter 8 actually make. Call the same reviewer subagent three times, changing only the model, and compare response speed and result quality with your own eyes.
Elements Used
Model Tiers (Ch.5)
Model Tier Comparison Chart
Claude App (Desktop) — Same File, Review Requests with Three Different Models

Prepare a piece of writing that's a bit more complex than G-1's notes.txt and contains a logical flaw. Paste the text below into the chat box as-is.

  1. ① Prepare the Target File 
    Create a file called argument.txt and write the following into it: "Remote work is always more productive than office work. This is because people who work remotely say remote work is good. Therefore, every company should switch entirely to remote work."
  2. ② Review with the haiku Tier

    Paste the text below into the chat box as-is. Claude App changes the model field in reviewer.md to haiku, then performs the review.

    Change the model field in .claude/agents/reviewer.md to haiku. Then use the reviewer subagent to review argument.txt.
  3. ③ Repeat the Same Request with the sonnet Tier

    The same way, change model to sonnet and review again. Compare against the haiku result to see whether it precisely identifies logical flaws such as circular reasoning.

    Change the model field in .claude/agents/reviewer.md to sonnet. Then use the reviewer subagent to review argument.txt.
  4. ④ Repeat the Same Request with the opus Tier

    The same way, change model to opus and review again. Note how the response time and depth of insight differ from ②③.

    Change the model field in .claude/agents/reviewer.md to opus. Then use the reviewer subagent to review argument.txt.
For a simple typo check, the results from the three models barely differ, but for a task like this one — where you need to identify a flaw in the structure of an argument — the lower the tier, the more pronounced the tendency to miss the circular reasoning as circular reasoning and instead flag only surface-level phrasing. Conversely, haiku responds fastest of the three. Experiencing the speed-versus-depth trade-off firsthand makes it much clearer why the principle explained in Chapter 5 — "the heavier the judgment a role requires, the higher the tier; the more routine the task, the lower the tier" — exists in the first place.
Just ask Claude App to "change the model field" each time — there's no need to open the definition file yourself; Claude App edits it and performs the review on its own.
A Common Failure — If the model field contains a value that doesn't exist (a typo, wrong capitalization, etc.), the call may fail or silently fall back to a default model. After finishing ②③④, open .claude/agents/reviewer.md directly and confirm the model: value exactly matches the intended haiku/sonnet/opus.

Wrapping Up the Practice Session

Once you've finished all five scenarios, decide what to do with the harness-lab/ folder. Since the next chapter (Ch.5) uses a new practice folder, you don't have to keep it, but if you'd like to compare the AGENTS.md and subagent definitions you've built so far, it's worth skimming through them once before deleting.

🧹 Cleaning Up the harness-lab Folder

If you no longer need it, delete the entire folder. On macOS/Linux/Git Bash, run the command as-is.

cd ..
rm -rf harness-lab

On Windows PowerShell

cd ..
Remove-Item -Recurse -Force harness-lab
If You'd Like to Keep It — Instead of deleting the folder, you can leave a local commit as a record with git init && git add -A && git commit -m "harness-lab practice record". Keep it around for reference afterward, and start the next chapter's exercises in a separate new folder.
The CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS flag turned on in .claude/settings.json in P-1 applies only to the harness-lab/ folder, so it disappears along with the folder if you delete it. If you want to keep using Agent Teams in other projects, you'll need to add the same setting separately to that project's .claude/settings.json.
