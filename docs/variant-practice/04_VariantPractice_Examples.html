작성일2026년 7월 14일
고급 시나리오 이후의 실제 승격(Phase B) 작업은 GitHub 접근 권한과 실제 개발 환경이 필요하다. 이 문서는 그 전 단계까지, 즉 로컬에서 안전하게 따라 할 수 있는 범위까지만 다룬다.
● 초급 Getting Started 저장소를 안전한 임시 위치에 받아, 기존 variant로 프로젝트를 하나 만들어보는 첫 단계
G-1 초급 기존 variant로 프로젝트 스캐폴딩하기
처음 ai-workspace-standards를 접해보는 개발자가, 실제 워크스페이스를 건드리지 않고 별도의 연습용 폴더(예: ~/harness-lab/ai-workspace-standards-demo)에 저장소를 받아 어떤 variant들이 있는지 확인하고, 성격이 다른 두 variant(정식 variant인 co-consult, beta variant인 co-deck)로 각각 프로젝트를 스캐폴딩해서 결과물의 구조가 어떻게 다른지 직접 비교해보려 한다.
사용 요소
git clone new-project.ts list-template-versions.ts
단계별 실행
  1. 연습용 폴더에 저장소 클론

    실제 작업 중인 워크스페이스와 절대 겹치지 않는 새 폴더를 만들어 그 안에 클론한다.

    mkdir -p ~/harness-lab
git clone https://github.com/5throck/ai-workspace-standards.git ~/harness-lab/ai-workspace-standards-demo
cd ~/harness-lab/ai-workspace-standards-demo
  2. bun 설치 확인 후 사용 가능한 variant 목록 조회

    bun이 없다면 먼저 설치한 뒤, 저장소가 제공하는 variant와 버전 목록을 확인한다.

    bun --version
bun scripts/list-template-versions.ts
  3. 기존 variant(co-consult)로 데모 프로젝트 스캐폴딩

    전략 컨설팅용 variant인 co-consult로 데모 프로젝트를 만들어본다.

    bun scripts/new-project.ts "demo-project" --variant co-consult
  4. beta variant(co-deck)로 두 번째 데모 프로젝트 스캐폴딩

    이번에는 강의·발표 자료 제작용 co-deck으로 별도의 데모 프로젝트를 하나 더 만든다. co-deck은 아직 beta 배지가 붙은 variant이지만 스캐폴딩 명령은 다른 variant와 동일하다 — --variant 값만 바꾸면 된다.

    bun scripts/new-project.ts "demo-project-deck" --variant co-deck
new-project.ts는 templates/co-consult/ 또는 templates/co-deck/의 전체 구성(에이전트 정의, 스킬, 거버넌스 파이프라인)을 그대로 복제해 각각 demo-project/, demo-project-deck/이라는 새 독립 git 저장소를 만든다. 두 저장소 모두 워크스페이스 데모 폴더와는 별개이지만 CONSTITUTION.md의 공유 규칙을 상속한 상태로 시작한다.
demo-project/agents/와 demo-project-deck/agents/를 나란히 열어 파일 개수를 세어 보면 차이가 뚜렷하다 — co-consult는 7단계 워크플로우, co-deck은 11단계 워크플로우라 에이전트 수와 docs/context.md의 phase 구성이 서로 다르다. variant 이름과 beta 배지만으로는 짐작하기 어려운 "실제 구조 차이"를 스캐폴딩 결과물로 직접 비교해볼 수 있다.
co-deck은 아직 beta variant다 — 5장 §3(templates와 variant 7종)에서 다뤘듯, 정식 variant보다 구조가 자주 바뀔 수 있으므로 실무 프로젝트보다는 이런 학습·실험 용도로 먼저 접해보는 것을 권장한다.
● 중급 Daily Workflow 마케팅 캠페인 콘텐츠 제작을 돕는 새 variant, co-marketing의 Phase A 프로토타입 만들기
D-1 중급 co-marketing Phase A 프로토타입 만들기
기본 사용법은 익힌 실무자가, 기존 variant 어디에도 딱 맞지 않는 "마케팅 캠페인 콘텐츠 제작 전용 에이전트팀"을 새로 만들어보려 한다. 처음부터 정식 templates/ variant로 만들지 않고, Projects/ 아래에서 독립적으로 발전시킬 수 있는 Phase A 프로토타입으로 먼저 시작한다. G-1에서 클론한 연습용 워크스페이스 안에서 이어서 진행한다는 것을 전제로 한다.
사용 요소
create-l2-scaffold.ts agents/*.md frontmatter
단계별 실행
  1. co-marketing Phase A 스캐폴드 생성

    variant 이름은 ^co-[a-z][a-z0-9-]{1,30}$ 규칙을 만족해야 한다. co-marketing은 이 규칙에 맞는 이름이다.

    bun scripts/create-l2-scaffold.ts co-marketing --domain marketing
  2. 도메인 에이전트 정의 작성 — ① researcher.md

    Projects/co-marketing/agents/researcher.md 파일을 아래 내용 그대로 만든다. name, role, status, tier(플랫폼별), model: inherit, color, description, phases, handoff_to, handoff_from, required_skills를 포함한 YAML frontmatter가 필수 형식이다. 2.1 실습에서 AGENTS.md로 역할을 도구 중립적으로 먼저 적었던 것과 같은 원칙이 여기서는 프로젝트 규모로 나타난다 — 이 frontmatter가 곧 "역할 명세"이고, 다음 단계에서 AGENTS.md 로스터에 등록해야 실제로 팀의 정본으로 인정된다.

    ---
name: researcher
role: 캠페인 타겟 오디언스 리서치 및 근거 수집
status: active
tier:
  claude: medium
  gemini: medium
  antigravity: medium
  gemini-cli: medium
model: inherit
color: blue
description: 캠페인 주제에 대한 시장·오디언스 자료를 조사하고 근거를 정리한다. 새 캠페인 카피 초안 작성 전에 호출한다.
phases: [1, 2]
handoff_to: [writer]
handoff_from: [pm]
required_skills: [campaign-research]
---
  3. 도메인 에이전트 정의 작성 — ② writer.md

    같은 방식으로 Projects/co-marketing/agents/writer.md를 만든다. handoff_from이 방금 만든 researcher를 가리켜, researcher → writer로 이어지는 핸드오프 계약이 완성된다.

    ---
name: writer
role: 캠페인 카피 초안 작성
status: active
tier:
  claude: medium
  gemini: medium
  antigravity: medium
  gemini-cli: medium
model: inherit
color: green
description: researcher가 정리한 근거를 바탕으로 캠페인 카피 초안을 작성한다.
phases: [2, 3]
handoff_to: [pm]
handoff_from: [researcher]
required_skills: [copywriting]
---
  4. AGENTS.md 로스터 갱신

    create-l2-scaffold.ts가 만든 Projects/co-marketing/AGENTS.md는 워크스페이스 루트에서 그대로 복사되어 온 상태라 auditor·lifecycle-manager 같은 워크스페이스 전용 에이전트 항목이 남아 있다. 이 프로젝트에는 존재하지 않는 에이전트이므로 표에서 지우고, 방금 만든 researcher·writer 항목으로 채운 "co-marketing Agents" 섹션을 새로 추가한다 — 5장에서 설명한 대로 AGENTS.md는 "이 프로젝트에서 누가 그 일을 하는가"를 정의하는 정본 목록이므로, 실제 존재하는 에이전트와 어긋나 있으면 안 된다.

    cd Projects/co-marketing && bun run agent:verify
    이 시점에는 Documented agents: 2 (researcher, writer)로 나와야 한다. 다른 숫자가 나오면 AGENTS.md 표와 실제 agents/*.md 파일 목록이 어긋난 것이므로, 표를 다시 확인한다.
Phase A 프로토타입은 templates/가 아니라 Projects/co-marketing/에 독립적으로 존재한다. 이 상태에서는 아직 다른 사람이 new-project.ts --variant co-marketing로 스캐폴딩할 수 없다 — 그러려면 나중에 Phase B 승격을 거쳐야 한다(7장에서 다룸).
variant 이름 규칙(^co-[a-z][a-z0-9-]{1,30}$)을 어기면 검증 스크립트가 실패한다. 대문자, 밑줄, co- 접두사 누락 등은 모두 무효다.
● 고급 Power User 프로토타입을 검증하고, 기존 variant와 비교해 무엇이 새로 만들어졌는지 점검
P-1 고급 프로토타입 검증과 승격 준비 점검
co-marketing 프로토타입이 구조적으로 문제가 없는지 확인하고, 다음 단계인 정식 승격(promote-variant)에 앞서 무엇이 co-work 같은 기존 variant와 다른지, 무엇이 정말 새로 만든 부분인지 정리해보려는 숙련자의 상황이다.
사용 요소
validate-templates.ts agent:verify
단계별 실행
  1. 구조 검증 스크립트 실행

    에이전트 frontmatter 완결성, 필수 섹션, 명명 규칙 준수 여부를 확인한다.

    bun scripts/validate-templates.ts
bun run agent:verify
  2. co-work과의 차이 정리

    기존 co-work variant의 에이전트 로스터와 co-marketing의 로스터를 나란히 비교해, 무엇을 그대로 가져왔고 무엇을 새로 정의했는지 메모로 남긴다. Git Bash/macOS/Linux에서는 아래 명령을 그대로 쓴다.

    diff <(ls templates/co-work/agents/) <(ls Projects/co-marketing/agents/)

    Windows PowerShell에서는 프로세스 치환(<(...)) 문법이 없으므로 Compare-Object로 대체한다.

    Compare-Object (ls templates/co-work/agents/).Name (ls Projects/co-marketing/agents/).Name
validate-templates.ts는 프로토타입 단계에서도 구조적 규정 준수 여부를 미리 잡아낼 수 있게 해준다. 여기서 나온 결과는 이후 PROMOTION_CHECKLIST.md를 채울 때 그대로 근거 자료가 된다.
Phase A에서 이미 구조를 깨끗하게 맞춰두면, 7장에서 다룰 Phase B 승격 파이프라인이 훨씬 매끄럽게 진행된다.
함께 보기: 정식 승격(Phase B) 절차와 l2-to-variant-pipeline.ts는 7.0과 그 실습 파일에서 이어서 다룬다.
