### 내용 구성 (4개 섹션)

**§1 — 워크플로우란 무엇인가** (`#s1`)
- 정의: "하나 이상의 에이전트가 협력하여 복잡한 작업을 완수하는 과정 전체"
- 워크플로우 = 오케스트레이션 패턴의 조합 + 도메인 규칙 + 인간 승인 게이트
- 2장 §4에서 배운 5가지 패턴을 레고 블록에 비유: 하나만 쓰기도 하지만 실무에서는 조합해서 쓴다
- 워크플로우 vs 오케스트레이션 패턴의 관계를 SVG로 시각화 (패턴 블록들이 모여 워크플로우가 되는 다이어그램)

**§2 — 워크플로우 구성요소** (`#s2`)
- **단계(Stage)**: 순차적으로 실행되는 하나의 작업 단위
- **분기(Branch)**: 조건에 따라 경로가 갈라지는 지점 (라우팅의 실제 적용)
- **병렬(Fan-out)**: 동시에 실행되는 독립 하위 작업
- **취합(Fan-in)**: 병렬로 나뉜 결과를 하나로 모으는 단계 ← 현재 핸드북에서 누락된 개념
- **게이트(Gate)**: 다음 단계로 넘어가기 전 검증/승인 지점 (3장 가드레일의 적용)
- **루프(Loop)**: 평가자-최적화처럼 거부 시 이전 단계로 돌아가는 반복
- 각 요소를 하나의 SVG 다이어그램에 통합해서 "워크플로우의 6가지 구성요소"로 시각화

**§3 — 실제 워크플로우 사례** (`#s3`)
4~5개의 도메인별 사례를 각각 패턴 조합 + 스테이지 흐름도로 정리:

| 사례 | 도메인 | 핵심 패턴 조합 | 스테이지 수 |
|------|--------|---------------|-----------|
| 코드 리뷰 | 소프트웨어 개발 | 라우팅 → 팬아웃 → 파이프라인 → 검증-후-적용 | 5 |
| 시장 조사 보고서 | 컨설팅/리서치 | 오케스트레이터-워커 → 팬아웃 → 파이프라인 → 평가 | 7 (co-consult 기반) |
| 프레젠테이션 제작 | 강의/발표 | 파이프라인(11단계) + 팬아웃(이미지+다이어그램) | 11 (co-deck 기반) |
| 이슈 트리아지 | 운영/인시던트 | 라우팅 → 오케스트레이터-워커 → 에스컬레이션 | 4 |
| PR 생성과 검증 | 거버넌스 | 파이프라인 → 검증-후-적용(루프) → 승인 게이트 | 4 |

각 사례:
- SVG 흐름도 (스테이지 × 패턴 시각화)
- 어떤 에이전트가 어떤 단계를 담당하는지
- 인간 게이트가 어디에 들어가는지
- 해당 variant와의 연결 (co-develop, co-consult, co-deck 등 기존 variant 참조)

**§4 — 워크플로우 선택 가이드** (`#s4`)
- 의사결정 나무 SVG: "작업에 의존 관계가 있는가?" → "하위 작업을 미리 예측할 수 있는가?" → "검증이 필요한가?" → 패턴 추천
- 간단한 표: "이런 업무에는 이 패턴부터 시작하라" 매트릭스
- 조합 가이드: "팬아웃 + 파이프라인은 병렬 리뷰에 좋다", "라우팅 + 오케스트레이터-워커는 티어링에 좋다" 등

### 네비게이션 구조
- **사이드바**: §1~§4 섹션 링크 + "다른 장" (7장, 8장, 10장)
- **장 이동**: prev → 8장 기업 운영, next → 10장 배포와 SSOT
