Chapter 5

Introducing ai-workspace-standards

ai-workspace-standards is the repository that actually implements the "dev tools + variant" model previewed in Chapter 7. This chapter walks through exactly what this repository is, what files it consists of, and what project templates it provides.

Sources: ai-workspace-standards (GitHub), ADR-0031 L1–L2 Fork Model, CONSTITUTION.md §0 SSOT Architecture

What this chapter covers
  • What ai-workspace-standards is, and why it must be cloned as the "workspace root"
  • The overall directory structure of the repository, and the actual order in which a session reads and executes files (the runtime flow)
  • The roles of the four core files: CONSTITUTION.md, CLAUDE.md, GEMINI.md, AGENTS.md
  • Which domain each of the 7 variants under templates/ is designed for
  • A conceptual picture of the session-start checklist and the new-project.ts scaffolding flow

What the repository is

ai-workspace-standards describes itself as an "operating model for multi-agent teams for Vibe Coding and Harness Engineering." It is a repository designed to apply the same behaviors, workflows, and quality standards across multiple AI coding tools (Claude Code/App, Antigravity CLI/Desktop, and so on).

The way this repository is used is a bit unusual. Rather than being pulled into a single project like an ordinary library, the repository itself is cloned as the root of your workspace — for example at C:\git on Windows or ~/git on macOS/Linux. Once set up this way, every new project created underneath it automatically inherits the workspace's shared rules.

ai-workspace-standards is a repository that ships "a way to build projects," not "a finished project." — why it's cloned as the workspace root

The overall architecture at a glance

Before dissecting the four core files, the variants, and L0→L1→L2 one by one in the following sections, it helps to first see the whole picture of where these pieces actually sit in the repository and under what names, so you don't lose your bearings. Cloning the workspace root produces the following top-level structure.

C:\git\  (workspace root = L0)
├── CONSTITUTION.md          ← §2: what must be followed (master standard, L0 original)
├── CLAUDE.md                ← §2: how it runs in Claude Code/App
├── GEMINI.md                ← §2: how it runs in Antigravity CLI/Desktop
├── AGENTS.md                ← §2: who does the work (agent roster, L0 original)
├── agents/                  ← §5: original agent definitions such as pm.md (L0, edit only here)
│   └── pm.md
├── .claude/                 ← §2: Claude-only commands/skills (/sync, create-variant, etc.)
├── .gemini/                 ← §2: Antigravity-only commands/skills (same names, different implementation)
├── .agents/                 ← §2: tool-agnostic commands/skills (for tools supporting the AGENTS.md standard)
├── scripts/                 ← §4·§5: operational scripts such as new-project.ts, publish-to-template.ts
├── templates/                (L1 + official L2)
│   ├── common/              ← §5: L1 — an L0 snapshot, the starting point when creating a variant
│   │   └── agents/pm.md     ← extends: ../../../agents/pm.md (references L0 + strips workspace-only sections)
│   ├── co-develop/          ← §3: official L2 variant (software development, 6 stages)
│   ├── co-design/           ← §3: official L2 variant (UI/UX, 5 stages)
│   ├── co-work/             ← §3: official L2 variant (collaboration/documents, 6 stages)
│   ├── co-security/         ← §3: official L2 variant (red team, 6 stages)
│   ├── co-consult/          ← §3: official L2 variant (consulting, 7 stages)
│   ├── co-deck/   [beta]    ← §3: official L2 variant (lectures/presentations, 11 stages)
│   └── co-game/   [beta]    ← §3: official L2 variant (game development)
├── Projects/                 (living L2 — actual projects scaffolded via new-project.ts)
│   └── <my-project>/
│       ├── docs/context.md  ← §4: this project's own SSOT (the variant's empty skeleton, filled in)
│       ├── agents/          ← inherited from the variant, continues to evolve for this project
│       └── (an independent git repository — separate from the workspace root)
└── memory/
    └── MEMORY.md            ← §4: context carried across sessions

What's worth noting in this tree is that the three layers appear directly as the actual directory structure. Right below the root (CONSTITUTION.md, CLAUDE.md, GEMINI.md, AGENTS.md, agents/, scripts/) is L0, templates/common/ is L1, and templates/co-*/ and Projects/*/ are L2. In other words, "L0→L1→L2" isn't an abstract diagram — it's a physical layout you can see with a single ls in the cloned folder.

The actual order in which work proceeds on top of this structure (the runtime flow) is as follows.

  1. Session start. The agent reads, in order: CONSTITUTION.md → the current project's docs/context.md → AGENTS.md → memory/MEMORY.md (covered in §4).
  2. Tool-specific execution rules apply. For Claude Code/App, CLAUDE.md and .claude/ govern the detailed execution; for the Antigravity family, GEMINI.md and .gemini/ do.
  3. When a new project is needed. scripts/new-project.ts clones the entire configuration of templates/co-<variant>/ (L2) into Projects/<name>/. At this point, templates/co-<variant>/ itself had already been created a generation earlier by taking a snapshot from templates/common/ (L1).
  4. When L0 changes during operation. scripts/publish-to-template.ts, as part of the /sync pipeline, automatically and continuously publishes L0 changes to L1 (templates/common/). This is not automatically propagated to L2 instances that have already been born (Projects/*, templates/co-*). This is the "fork" principle covered in Chapter 8.
In one sentence: L0 is "right below the root," L1 is "templates/common/," and L2 is "templates/co-*/ and Projects/*/" — remembering just these three coordinates lets you follow every explanation in the rest of this chapter against the actual folder locations.

The four core files

At the workspace root there are four core files that every project references in common.

CONSTITUTION.md is the master shared standard, the document that should be read first at the start of any session. It defines the governance rules, PR workflow, and multi-agent architecture principles for the entire workspace. CLAUDE.md holds the concrete behavior for Claude Code/App (hooks, slash commands, agent dispatch rules), while GEMINI.md defines the equivalent behavior for Antigravity CLI/Desktop. AGENTS.md serves as the canonical roster of agents available across the entire workspace. This filename is the same as the open AGENTS.md standard introduced in Chapter 2, which is read in common by 30+ tools including Claude Code — though in this repository it also doubles as a more specific "agent registry and orchestration reference." The top of the file even states explicitly that "this file is not an instruction directed at AI tools, but a registry of human-defined roles."

To summarize the relationship among these four files in one sentence: CONSTITUTION.md defines "what must be followed," CLAUDE.md/GEMINI.md define "how that is executed on each platform," and AGENTS.md defines "who does the work."

An agent entry in AGENTS.md consists of the following fields. Concrete authoring guidance and tool-specific implementation mapping are covered in Chapter 8 §3, AGENTS.md Deep Dive.

FieldMeaningRequiredExample

Let's trace this relationship using a concrete rule as an example. Suppose there's a rule that says "PR titles and bodies must be written in English." This rule itself is written exactly once, in CONSTITUTION.md. CLAUDE.md defines how to enforce this rule in Claude Code — for instance, by specifying that the /sync command's language gate (language-guard.ts) automatically checks this rule before every commit. GEMINI.md defines how the same rule is enforced in the Antigravity-side pipeline. And who actually performs this check — which agent is responsible for this gate — is registered in the roster in AGENTS.md. A single rule shows up in all four files, each from a different angle. This isn't duplication; it's three different answers to the three questions of "what, how, and who."

Below the four core files sit folders holding executable assets. Commands and skills are split across three locations: .claude/ is Claude Code/App-only, .gemini/ is Antigravity CLI/Desktop-only, and .agents/ holds tool-agnostic (engine-agnostic) commands and skills not tied to a single tool. For example, slash commands like /sync//meeting or skills like create-variant/audit-workspace exist under the same name in all three folders, but each implementation differs slightly to match its own platform's tooling conventions (Claude Code's Agent tool syntax, Antigravity's sub-agent approach, etc.). External tools that support the AGENTS.md standard (Codex, Cursor, etc.) will reference the .agents/ folder when they open this repository.

templates and the 7 variants

The templates/ directory is a collection of version-controlled project template "variants." Each variant contains a complete multi-agent team configuration for a specific domain, including a PM, specialist agents, and a staged governance pipeline. The variants the repository provides are as follows.

co-developFor software development. A 6-stage governance pipeline composed of PM, Architect, Designer, Code Writer, Test Runner, and Security Monitor.
co-designFor UI/UX. A 5-stage iterative workflow with Design Lead, UX Researcher, Visual Designer, Prototype Engineer, and others.
co-workFor general collaboration/document work. A 6-stage asynchronous workflow with Analyst, Technical Writer, Content Writer, and others.
co-securityFor red-team/penetration testing. A 6-stage pipeline with Red Team Lead, Pentester, Threat Modeler, and others.
co-consultFor strategy consulting. A 7-stage (Phase 0–6) workflow led by a PM (Engagement Leader), composed of 10 specialist agents including Strategy Analyst, Change Management Partner, Industry Expert, and SME. It covers the entire consulting project lifecycle — market analysis → organizational diagnosis → cross-validation (Phase 1.5) → proposal approval gate (Phase 2) → practical solution design → delivery → PR/handoff.
co-deck betaFor producing lecture and presentation materials. An 11-stage (Stage 0–11) pipeline led by a PM. It automates the entire process from research to a print-ready PDF: Research → Source Verifier (credibility verification) → Storyline → Design → Image Curation + Diagram Generation (parallel) → HTML Build → Layout Measure → PDF Export. Supports 5 themes (outline, pitch, pitch-enhanced, vertical, zen) and 5 styles (premium-dark, classic, minimal, visual-heavy, academic).
co-game betaFor HTML5 Canvas game development.

Each variant is versioned with the git tag template-vX.Y.Z, and bun scripts/validate-templates.ts checks structural compliance — the completeness of agent frontmatter, required sections, consistency with the AGENTS.md roster, and platform parity between .claude/ and .gemini/.

As the name templates/ suggests, what this repository ultimately ships is boilerplate — a starting point that already has the PM role definitions, governance pipeline, and platform-specific command/skill wiring you'd otherwise have to write from scratch for every project. But unlike the usual "copy it and everyone tweaks it themselves" boilerplate, this repository explicitly lays out, as a layered structure, where that starting point came from and how it gets updated. That's the L0 → L1 → L2 model covered in the next section.

One of the things this boilerplate has already set up in advance is model assignment per agent role. Each agent definition file (agents/*.md) inside a variant has a tier field in its frontmatter, and this value assigns high/medium/low per platform (claude/gemini/antigravity/gemini-cli). Roles requiring heavy judgment (PM, Architect) get high, roles handling actual implementation (code-writer, docs-writer) get medium, and standardized repetitive tasks (automation-engineer, etc.) get low. The actual model field is usually left as inherit, because each platform translates this tier value into its own model alias (opus/sonnet/haiku) at deployment time. In other words, the moment you clone the boilerplate, a new project already starts with "who works using which cost tier of model" already decided.

What actually goes wrong without this tiering? If every agent were uniformly assigned to the highest-performance model (high), even a role like automation-engineer that only repeats standardized script execution would incur unnecessarily expensive inference costs every time, making the whole team's operating cost pointlessly large. Conversely, if everything were unified onto the lowest-cost model (low), a role like PM — which needs to synthesize the output of multiple agents and make judgment calls — would frequently make poor coordination decisions from only shallow reasoning. Tiering is a mechanism that pre-balances cost and quality per role, matched to "how much depth of reasoning this role actually needs."

This model cost tier (high/medium/low) shares only the "three-tier" feel of its name with the L0 → L1 → L2 3-tier (the layering of where files come from and how they flow) covered in Chapter 8 — they are otherwise different concepts. One is "which model to assign to an agent," and the other is "where the original of a file lives."

Session-start checklist and scaffolding

CONSTITUTION.md defines the order every session must follow at startup. First, set the git hooks path with git config core.hooksPath .githooks, then read CONSTITUTION.md itself, then read the docs/context.md of the project you're working on. Next, check AGENTS.md for the available agents, check whether context from a previous session remains in memory/MEMORY.md, and finally load whatever skills docs/context.md specifies.

The flow for creating a new project is conceptually simple. Run the bun scripts/new-project.ts script with a project name and the desired variant, and the entire configuration of that variant is cloned into a fully independent git repository. This repository is separate from the workspace root, but it starts out having already inherited CONSTITUTION.md's shared standards from birth. The team then evolves the team by filling in the project's own docs/context.md and any project-level CLAUDE.md/GEMINI.md overrides. The exact syntax and options for this command are practiced hands-on in Chapter 6 §1.

It's worth clarifying the relationship between CONSTITUTION.md and context.md here. Both are documents read during the "session-start checklist," but they sit at different layers of the SSOT hierarchy. CONSTITUTION.md exists exactly once, at L0 (the workspace root), and defines rules shared by every project — governance, PR workflow, and multi-agent principles are all here. context.md, by contrast, exists separately for each L2 (individual project), and is the project-specific SSOT that holds that project's own goals, domain knowledge, and current progress. The difference is that CONSTITUTION.md isn't a file that's filled in by mandate from above — it's a file each project team writes for itself. That's why the checklist has an order: first confirm "what must be followed (shared)" via CONSTITUTION.md, then confirm "what this project is doing right now (individual)" via context.md.

Note, however, that a file with the same name also exists inside each variant, at templates/<variant>/docs/context.md. This shares only its name with an L2 project's context.md — its role is different. The context.md inside a variant is still an unfilled empty skeleton (a placeholder), and the moment new-project.ts runs, it is cloned exactly once, just like the other L1→L2 files. Once cloned, its connection to the original (the variant's context.md) is completely severed, and it becomes an independent document that the project team fills in with real content going forward. So even if the variant's context.md skeleton changes later, it is not automatically reflected into the context.md of a project that has already been scaffolded.

Files at the workspace root (CONSTITUTION.md, CLAUDE.md, etc.) and files in individual scaffolded projects belong to separate git repositories. It is a core principle of this repository that the two are never modified together in the same piece of work.
