강의 진행 가이드

이 핸드북 전체(1~7장)를 사용해 강의를 진행할 강사를 위한 자료입니다.

이 문서는 핸드북 본문이 아니라 강사 전용 운영 자료입니다. 학습자에게는 공유하지 않아도 됩니다.

사전 준비물 통합 체크리스트

각 장의 setup 문서(사전 설치 체크리스트)에 흩어져 있던 항목을 강의 시작 전 한 번에 확인할 수 있도록 모았습니다. 강의 시작 최소 하루 전에 참가자에게 이 목록을 공지해 현장에서 설치 문제로 시간을 낭비하지 않도록 합니다.

전체 시간 배분표

모든 절을 꼼꼼히 다루고, 실습은 참가자가 직접 타이핑(또는 복붙)할 시간을 포함해 산정했습니다. 휴식 시간은 별도입니다.

장별 강사 노트

1장: 개념

이 장은 이후 모든 장에서 쓰는 용어(오케스트레이터, 스페셜리스트, 핸드오프, 서브에이전트)의 기준점이므로 속도를 늦춰도 좋습니다. 비유(요리, 신입사원, 택배 송장)를 화이트보드에 옮겨 그리면서 설명하면 이해도가 확 올라갑니다.

참가자 활동 §4의 세 패턴(팬아웃/파이프라인/검증-후-적용) 중 하나를 골라 "우리 팀 업무에 적용한다면?"을 1분씩 옆 사람과 이야기하게 하는 것을 권장합니다.

1.5장: 가드레일과 권한 모델

2.1 실습 직전에 배치돼 있어야 하는 이유가 있다. 서브에이전트를 직접 만들기 전에 "무엇을 허용하고 무엇을 막을지"를 먼저 정하는 습관을 들이기 위함이다. 위험 작업 3단계 분류표를 보여주면서 "여러분이 방금 만들 예정인 reviewer는 어느 단계에 해당할까요?"라고 미리 물어보면 2.1 실습으로 자연스럽게 이어진다.

2.0 / 2.1: 실습

참가자를 4개 그룹(Claude App / Claude Code / Antigravity Desktop / Antigravity CLI)으로 나눠 각자 다른 도구로 같은 시나리오를 진행하게 하면, 마지막에 결과를 비교하며 3장의 "도구 비교" 내용이 훨씬 잘 와닿습니다.

참가자 활동 G-1, D-1, D-2(동적 팀 구성 관찰)는 전원 실습. P-1(병렬 실행)·P-2(모델 티어 비교)는 시간이 부족하면 강사 화면 시연으로 대체 가능. Agent Teams 설정(`.claude/settings.json`)이나 model 필드 교체는 그룹당 한 명만 해봐도 개념은 전달된다.

5장(가볍게): ai-workspace-standards 소개

1일차에서는 6장 실습에 필요한 최소한의 개념만 다룬다. 저장소가 무엇인지, 핵심 파일 4종의 역할, variant란 무엇인지, 그리고 스캐폴딩으로 프로젝트를 만드는 기본 흐름만 전달하면 충분하다. L0→L1→L2 계층 구조와 모델 티어링 등은 2일차 심화에서 다룬다.

3장: 도구 비교 (2일차)

2일차에 배치된다. 1일차 2.0/2.1 실습에서 참가자가 도구를 직접 경험했으므로, "방금 여러분이 쓴 도구에서 이게 왜 이렇게 동작했는지" 형태로 복기하면서 진행하면 순수 강의보다 훨씬 몰입도가 높습니다. 개발자 대상이므로 teammateMode(in-process vs tmux), Agent Manager 내부 구조 등 기술적 세부 사항을 깊이 있게 다룰 수 있다.

4장: 두 모델 비교 + 칸반 (2일차)

2일차에 배치된다. 칸반 절은 실제 사내 헬프데스크/지원팀 운영 경험이 있는 참가자가 있다면 그 경험을 먼저 물어보고 시작하는 것이 효과적입니다. "왜 필요한가"를 이론으로 설명하기 전에, "티켓이 밀렸던 경험 있나요?"로 시작하면 공감대가 빨리 형성됩니다. 개발자 대상이므로 GitHub Projects/Jira 연동 등 실무 구현 방법까지 언급하면 좋다.

토론 "우리 조직이라면 어느 모델이 맞을까?"를 5분 소그룹 토론 후 발표.

5장(심화): ai-workspace-standards 아키텍처 (2일차)

2일차에서 1일차의 가벼운 소개를 이어받아, L0→L1→L2 계층 구조, templates·variant 7종의 내부 구조, 모델 티어링, 세션 시작 체크리스트, 고도화 방향을 상세히 다룬다. 전부 암기시킬 필요는 없고, "왜 이런 계층 구조가 필요한가"라는 동기만 확실히 전달되면 충분하다.

6장: 기존 variant 활용 (1일차)

이 실습은 6장 전체가 기존 variant 활용에 할당된다. G-1(co-consult Physical AI 시장조사)은 40분, G-2(co-deck 한식 세계화 발표자료)은 35분으로, 두 실습 합계 75분의 충분한 시간을 확보했다. 이것이 이틀 워크숍의 핵심 실습 시간이다.

new-project.ts 실행 시 실제로 GitHub 클론이 오래 걸릴 수 있으므로, 이 실습을 시작하기 전에 미리 워크스페이스를 클론해 두라고 사전 공지하는 것이 중요합니다(위 사전 준비물 체크리스트 참고).

참가자 활동 G-1에서는 4단계 설계 프레임워크(목표·입력·출력·검증 기준)를 직접 적용하고, 에이전트팀 가동 후 핸드오프를 추적해보게 한다. G-2에서는 발표 주제 프레임(청중·목적·시간)을 직접 설정하고, 완성된 슬라이드를 리뷰·개선하며, TTS와 자동 슬라이드 설정까지 체험하게 한다. 두 variant의 구조 차이를 G-1·G-2 산출물로 직접 비교하도록 안내한다. 참가자 활동 1일차 마무리 시간(10분)에 G-1의 보고서 결과물과 G-2의 발표자료 결과물을 전원 공유하도록 한다. "co-consult 보고서를 co-deck 입력으로 넘기면 어떻게 되나요?"를 질문하면 2일차 신규 variant 생성의 동기가 자연스럽게 형성된다.

7장: 신규 variant 생성

이 장은 2일차에 배치된다. 1일차에서 기존 variant를 직접 다뤄본 경험을 바탕으로, 이번에는 자신만의 variant를 처음부터 만들어본다. 실습(D-1) 40분 + 강의 20분으로 구성한다.

참가자 활동 D-1(co-marketing Phase A 프로토타입 직접 만들기)은 전원 실습. 1일차 G-1에서 에이전트 정의 파일의 frontmatter 구조를 이미 살펴봤으므로, 이번에는 직접 researcher·writer를 만들어본다. 시간이 빠듯하면 짝 활동(한 명이 타이핑, 한 명이 검증 명령 실행)으로 진행해도 좋습니다.

8장: 승격 실습

2일차 후반에 배치된다. 7장에서 만든 co-marketing 프로토타입을 승격 준비하는 단계다.

강사 시연 전용 실제 l2-to-variant-pipeline.ts 실행은 git 이력에 영향을 주는 비가역적 작업입니다. 참가자 전원이 따라 하게 하지 말고, 강사가 화면 공유로 한 번 실행하는 것을 보여주는 것으로 충분합니다. 참가자가 직접 해보고 싶다면 강의 종료 후 개인 환경(워크숍 전용 클론 폴더)에서 해보도록 안내하세요. 참가자 활동 PROMOTION_CHECKLIST.md 항목 확인과 variant.json manifest 작성까지는 전원 실습 가능합니다.

9장: 캡스톤 실습

2일차 핵심 실습이다. 1일차(기존 variant 활용)와 7장(신규 variant 생성)의 경험을 모두 합쳐, 참가자 본인의 실제 업무에 멀티 에이전트팀을 적용하는 설계를 직접 해보게 한다.

이 장은 정답이 없다는 점을 먼저 분명히 하세요. 참가자가 "무엇을 만들어야 하나" 막막해하면, 강사가 자기 예시(예: "매주 회의록 액션 아이템 정리")를 먼저 시연해 질문 흐름을 보여주는 것이 효과적입니다. 시간이 촉박하면 4단계 전체를 다 실행하기보다, 최소한 1~2단계(문제 정의·가드레일 결정)까지는 전원이 끝내고 3~4단계는 숙제로 남겨도 좋습니다.

참가자 활동 완료 체크리스트를 활용해 각자 어디까지 마쳤는지 스스로 표시하게 하고, 캡스톤 발표 시간에 1~2명이 자기 워크플로우를 짧게 공유하도록 합니다.

마무리

이틀 전체 회고. 1일차에서 기존 variant로 에이전트팀을 가동해본 경험과, 2일차에서 신규 variant를 직접 만들고 승격해본 경험을 연결해 정리한다.
