Chapter 10

L2 Project Upgrade

Once you've created an L2 project, what happens when the upstream standards (L0/L1) change? The answer is: "nothing happens at all." Because of the fork model discussed in Chapter 8 §1, actually bringing new upstream changes into an already-created project requires a human to explicitly run upgrade-project.ts. This chapter covers that procedure.

Source: ai-workspace-standards/scripts/upgrade-project.ts (as of v1.6.0) - Related: Chapter 8 §1 Deployment and SSOT

What this chapter covers
  • Why "upgrading" in the fork model is a command you run, not something that happens automatically
  • Basic usage of upgrade-project.ts and the principle of running --dry-run first
  • The 5 file-handling modes: LOCKED, MERGE, PRESERVE, SYNC_IF_NEWER, and PRUNE
  • The automatic git stash snapshot and the --rollback safety net
  • How this is the exact opposite direction from the "promotion" covered in Chapter 11

Why it's needed — the flip side of the fork model

As established in Chapter 8 §1, L1 delivers its content to a project exactly once — at the moment new-project.ts scaffolds it. From that point on, L2 evolves completely independently, and no matter how much L1 changes afterward, nothing is automatically reflected in an L2 project that already exists.

The problem is that, over time, hook scripts get fixed on the L1 side, new skills get added, and security settings get hardened. When you want to bring those changes into an individual L2 project, manually comparing files one by one isn't realistic. upgrade-project.ts automates this reflection process while still honoring the core principle of the fork model — "never blindly overwrite parts that L2 has deliberately changed."

How to use upgrade-project.ts

Start by running --dry-run to preview what will change. It doesn't touch any files — it just prints the plan to the console.

bun scripts/upgrade-project.ts <project-path> --dry-run

If the output looks fine, run it again without --dry-run.

bun scripts/upgrade-project.ts <project-path>
# Optional flags: --variant <variant>  --platform claude|antigravity|both
#            --prune-removed  --rollback

--variant is usually unnecessary since it's read automatically from the value recorded in the project's .claude/template-version.txt. For older projects that lack this file, the script prompts for confirmation at runtime and you'll need to specify --variant directly.

The 5 file-handling categories

The script treats files differently depending on their nature — it isn't a blanket "overwrite everything with the new version."

CategoryBehaviorExample targets
LOCKEDAlways overwritten with the template's content.githooks/*, .gitattributes, .gitleaks.toml
MERGEOnly the sections wrapped in WORKSPACE-MANAGED/COMMON-CLAUDE/COMMON-GEMINI markers are merged; the rest of the local content is preserved as-isCLAUDE.md/GEMINI.md (depending on --platform), .gitignore, agents/pm.md
PRESERVEListed only, never touchedREADME.md, README_ko.md, docs/context.md, src/
SYNC_IF_NEWERCopied only when the template version is newer than the project's versionscripts/*.ts (@version comment), agents/*.md (frontmatter version:), skills/*/SKILL.md
PRUNE (optional)When --prune-removed is specified, scripts/agents/skills files that were removed from the template are also removed from the projectSame directories as the SYNC_IF_NEWER targets above

If a SYNC_IF_NEWER target has been edited locally but the template's version is newer, the script marks it as ⚠️ CONFLICT but still overwrites it with the template's content in the end. Keep in mind that this is "warn and proceed," not "ask before overwriting" — which is exactly why the safety net described below matters.

Safety nets — snapshots and rollback

Before running (when not in --dry-run mode), the script automatically runs git stash push -m pre-upgrade-snapshot-YYYYMMDD to snapshot the current state of your working tree. If you're not happy with the upgrade result, you can revert it immediately with the following command.

bun scripts/upgrade-project.ts <project-path> --rollback

This command finds the pre-upgrade-snapshot-* stash that was just created and restores it via git stash pop. The upgrade itself does not create a commit, so after reviewing the result, you'll need to check the changes with git diff as usual and commit them yourself.

As a final step, .claude/template-version.txt is updated to the new version, and security bootstrap items — .gitleaks.toml, pre-commit hooks, .gitattributes (eol=lf), .gitignore (.env patterns), and git core.hooksPath — are automatically checked (these are the same items covered by the Chapter 3 guardrail principles). If core.hooksPath is misconfigured, the script fixes it on its own.

How this differs from promotion

The "promotion" covered in Chapter 11 and this chapter's "upgrade" sound similar, but they run in exactly opposite directions.

  • Promotion (l2-to-variant-pipeline.ts, Chapter 11) — L2 → L1. Elevates changes from a project you created into the official template, turning them into a variant others can use.
  • Upgrade (upgrade-project.ts, this chapter) — L1 → L2. Pulls the latest changes from the official template into a project you've already created.

The two scripts simply move files in opposite directions — neither replaces the other. If you want to contribute back to the template, use promotion; if you want to keep an existing project up to date, use upgrade.

The script behavior described in this chapter (flags, file categories, safety nets) was written by directly examining the source of scripts/upgrade-project.ts (v1.6.0) in the ai-workspace-standards repository. This handbook repository does not include that script, so always cross-check actual execution results against the latest source.
